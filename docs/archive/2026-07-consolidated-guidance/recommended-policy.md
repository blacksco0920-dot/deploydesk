# ABCDeploy 推荐部署策略

> 首版默认组合为 CNB、腾讯云 TCR、Linux 服务器和 Caddy；这是 Provider 选择，不改变产品领域模型。

## 1. 默认策略

```text
短期 feature/* → 合并 main
  → 构建一次不可变镜像
  → 推送 OCI Registry
  → 自动部署 staging
  → 健康检查 + 用户确认测试
  → 选择任一测试通过版本
  → 人工发布 production（同一 digest）
```

- 新项目只保留 `main` 一条长期业务分支；不为环境创建长期分支。
- `main` 是版本来源，不代表直接影响正式用户。
- staging 默认自动部署；production 默认且在 P0 中始终人工确认。
- 正式发布和恢复都不重新构建。
- 已有可靠分支与流水线时先导入现状并解释差异，不静默覆盖。

## 2. 环境默认值

| 环境 | 默认行为 |
| --- | --- |
| development | 本机稳定容器运行；全部服务可靠支持时可选开发热更新 |
| staging | 自动接收 `main` 新版本；独立配置和数据；域名可选 |
| production | 选择测试通过版本人工发布；独立配置、数据和发布记录 |

staging 与 production 推荐使用不同服务器。起步阶段可以共用物理服务器，但必须保存两个明确绑定并隔离 Compose、网络、目录、卷和端口。

## 3. 资源默认值

### 代码与构建

- CNB 负责仓库事件、构建、远程部署任务和构建日志。
- 客户端隐藏流水线语法，只呈现用户阶段和恢复动作。
- 授权只申请需要的最小权限；失效时可原地替换并继续任务。

### 版本保存位置

- 国内服务器默认推荐腾讯云 TCR；CNB 制品库及其他 OCI Registry 可作为替代 Provider。
- Registry 在第一次构建前验证，作为项目设置中的明确资源展示。
- 所有发布最终固定为 digest；可变标签不作为生产门禁。

### 服务器

- 优先推荐已验证且资源充足的连接，但 staging/production 分别绑定。
- 部署前检查 Docker、磁盘、内存、端口和代理冲突。
- 可清理未被任何容器使用的旧镜像，但不删除未知容器、卷、目录或数据。
- 配置、Compose 和回滚记录使用原子替换，避免磁盘写满留下空文件。

### Caddy

- Caddy 是统一反向代理和 HTTPS 入口，不生成 Nginx 默认链路。
- 每个项目只维护自己的路由片段；变更前备份并校验，失败恢复。
- 当前项目缺失路由可幂等修复；地址被其他项目使用时要求明确确认。

## 4. 配置与数据

- 字段说明以项目注释为准，客户端不编造业务名称。
- 第三方 API Key、已有数据库凭据和业务平台凭据需要用户提供或从配置中心绑定。
- 系统可生成内部端口、服务 URL 和随机秘密，但不跨环境自动复用用户值。
- 检测到数据库时才展示相关事项；production 迁移前必须验证备份。

## 5. 域名

- staging 可使用本机安全预览或测试域名，不因缺少域名阻塞部署。
- production 发布前确认正式地址或明确选择仅内网/IP 访问。
- 系统给出精确 `A/AAAA` 记录和服务器 IP，并持续检查 DNS、443 与 HTTPS。
- DNS 未生效时结论为“部署成功，地址待完成”，不重新构建或重启健康服务。

## 6. 风险和确认

| 操作 | 默认处理 |
| --- | --- |
| 扫描、查版本、查 DNS | 自动 |
| 保存本地设置、绑定已有资源 | 说明结果 |
| 创建仓库、初始化服务器 | 单独确认 |
| 正式发布、恢复、路由接管 | 强确认并记录 |
| 删除未知资源或数据 | P0 不执行 |

推荐理由最多一句，并使用用户语言，例如：“正式发布需要确认，避免一次代码合并直接影响真实用户。”
