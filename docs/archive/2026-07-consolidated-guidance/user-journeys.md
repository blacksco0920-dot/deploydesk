# ABCDeploy 当前用户旅程

## 1. 第一次添加项目

用户从侧边栏选择项目目录。系统只读识别后进入“上线总览”，并给出两个环境的真实初始状态：测试版尚未部署、正式版尚未发布。页面只突出“完成项目设置”一个动作。

用户进入项目设置，看到代码平台、版本保存位置、测试服务器、测试配置和自动部署各自的状态。工作区只有一个已验证连接时系统推荐复用；用户仍能看见连接名称并更换。正式服务器和正式配置不阻塞首次测试部署。

## 2. 第一次生成测试版

用户完成必要设置后，把代码合并到 `main`。客户端可以关闭；CNB 自动构建不可变版本、推送 Registry 并部署测试服务器。

用户再次打开项目，普通入口仍是上线总览。测试环境卡显示当前阶段；打开详情后可以查看测试入口。用户确认业务正常，版本获得“测试通过”资格。无需再完成一遍本机运行。

## 3. 日常更新

用户继续在开发工具中工作。每次合并 `main` 都产生一个新不可变版本并自动更新测试环境。版本页保留多个版本及各自资格；失败的构建或部署作为记录显示，不把当前健康测试版和正式版改成离线。

用户只在需要业务验收、排障或正式发布时打开 ABCDeploy。

## 4. 发布正式版

用户从上线总览打开正式环境。页面推荐当前测试版，但允许选择任一测试通过版本。第一次发布时，用户补充或确认独立的正式服务器、`production` 配置和正式地址。

最终确认说明：将把哪个版本部署到哪个地址、不会重新构建、是否有上一健康版本可恢复。发布完成后总览显示正式环境在线版本。

如果容器已运行而域名尚未解析，一级结果是“正式版已部署，还差地址”。用户按提示完成 DNS 后点击“检查并完成发布”，不需要重选版本或重新部署。

## 5. 恢复历史版本

用户在版本页选择一个较早且测试通过的版本，点击“恢复到正式环境”。系统仍按 production 当前设置部署该版本的原镜像摘要，并执行与正式发布相同的确认、健康检查和记录流程。

## 6. 返回应用

返回用户点击项目时先看到上线总览，而不是被上次失败页或后台任务带走。总览同时显示在线事实与最近尝试。

若用户点击“待处理”中的某项任务，才直接回到该任务准确的环境、版本和停点。处理完或离开上下文后，下次普通打开仍回总览。

## 7. 更换失效授权

CNB Token 权限不足时，错误卡直接提供“更新 CNB 授权”。用户粘贴并验证新 Token 后，系统继续原项目、原环境和原任务。配置中心、版本记录和服务器绑定不丢失；无需移除项目。

Registry 或服务器凭据失效时遵循同一模式：在当前错误或项目设置中更换连接，验证成功后继续断点。

## 8. 复用配置中心

用户在配置中心保存一项由“说明、Key、Value”组成的配置。项目设置分别把它绑定到 `development`、`staging` 或 `production`。

项目表单只推荐 Key 匹配的条目，不擅自跨环境复用。绑定关系持续存在；配置值更新时，界面说明哪些项目和环境会受影响，并要求用户确认同步。

## 9. 在本机运行

用户可以随时进入本机页面，逐个或一键启动项目服务与运行依赖，补齐 `development` 必要配置。只有最近一次完整启动成功的服务才显示可打开结果。

如果代码、依赖声明或初始化脚本有问题，客户端给出脱敏原因和可复制给开发工具的说明，不把本机修复扩展成部署流程。

## 10. 多项目

第二个项目可以复用已验证的 CNB、Registry 和服务器连接，也能选择不同的 staging/production 服务器和配置绑定。后台任务按“项目 + 环境”独立运行；一个项目的测试构建不会抢占另一个项目的正式待办。
