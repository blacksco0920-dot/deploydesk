# ABCDeploy 产品与实现文档

> 当前状态：`0.2.0-preview.4` 封板发布候选，四场景闭环已完成本地实现与验收；对齐日期：2026-07-16。

ABCDeploy 面向缺少 DevOps 经验的 vibe coding 用户。产品不要求用户先学习 Docker、镜像仓库、SSH、Caddy 或流水线，而是围绕用户要完成的结果组织页面：系统能判断和安全完成的自动处理，只有授权、业务配置、域名和最终确认等必须由用户决定的事项才显性展示。

## 当前产品模型

添加和切换项目统一由左侧项目栏承担，不属于部署步骤。每个项目固定提供四个可以独立进入、反复使用的场景：

```text
在本机运行｜管理版本｜部署测试版｜发布正式版
```

- **在本机运行**：快捷启停项目服务和运行依赖，并检核项目配置；它不是部署测试版的强制前置条件。
- **管理版本**：首次完成“连接代码平台、保存项目版本、准备测试环境、开启自动部署”，之后查看版本记录和恢复未完成任务。
- **部署测试版**：主分支更新后自动生成版本并更新测试环境；客户端主要用于查看进度、打开测试版和确认业务结果，手动部署只是辅助入口。
- **发布正式版**：从任意一个测试通过的不可变版本中明确选择并发布，不重新构建；发布、恢复和外部网页停点都绑定具体版本。

配置中心是应用级的可选复用工具，只维护“配置说明、配置名称、配置值”。项目中可以直接填写配置，不要求先维护配置中心。

## 不变的设计原则

1. **先给推荐答案**：系统先形成可执行方案，用户只处理无法推断的信息。
2. **场景优先**：不用覆盖全流程的五步进度条；一次性设置和任务阶段留在对应场景内部。
3. **状态可恢复**：项目、场景、版本、任务和外部操作停点在重启后继续。
4. **连接可复用**：CNB 账号、镜像仓库和服务器属于全局资源，项目只保存绑定关系。
5. **安全默认开启**：敏感值进入系统密钥库，复制的安全配置在确认保存后及时清理，不进入 Git、SQLite、普通日志或截图。
6. **渐进式披露**：普通用户先看结果和下一步；YAML、文件路径、提交、摘要和原始日志默认后置。
7. **一个状态，一个动作**：失败或等待处理时说明发生了什么、已完成什么和唯一下一步。
8. **组件优先**：正式页面复用 `components/ui`、Radix UI 和语义设计令牌，不在业务页面重复设计通用组件。
9. **生产显式确认**：正式版只使用测试通过的同一镜像，默认不自动晋级，也不因 DNS 或 HTTPS 未完成而抹掉已经成功的部署结果。
10. **首版厂商不是产品模型**：CNB、腾讯云 TCR、Linux 服务器和 Caddy 是试点组合；代码平台、Registry、Runtime、Secret 和 DNS 均保留独立适配边界。GitHub 不属于首版默认链路。

## 当前门禁

| 门禁         | 当前结论                                                           |
| ------------ | ------------------------------------------------------------------ |
| 产品定义     | 四场景、一次性上线设置、自动测试部署和多版本生产发布已形成当前基线 |
| 页面规范     | 组件复用、渐进披露、任务恢复、窄窗口和可访问性规则已统一           |
| 本地实现     | macOS Apple Silicon 上已完成首个闭环和大量恢复/失败分支验证        |
| 真实目标用户 | 仍需无提示可用性测试，不能用开发者模拟代替                         |
| 跨平台       | Windows 与 Linux 仍需完整实机主线、系统密钥库和辅助功能验收        |
| 正式发布     | 未放行；当前文档整理不升级版本、不生成 DMG、不触发远端发布         |

## 文档维护规则

- `docs/` 根目录只放当前仍生效、有人维护的文档。
- 被新结论替代的方案、原型和逐次验收记录移动到 `docs/archive/`，并在归档索引说明失效原因。
- 规范文档描述稳定产品规则；频繁变化的测试数量、时间戳和逐屏过程放入归档验收日志。
- 新功能只有在产品规则、页面状态、失败恢复和验收口径同时明确后，才写入“当前实现”。
