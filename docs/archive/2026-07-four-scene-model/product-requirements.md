# ABCDeploy 产品需求文档

## 1. 文档信息

| 项目     | 内容                                                           |
| -------- | -------------------------------------------------------------- |
| 版本     | 当前产品定义，不与应用安装包版本绑定                           |
| 产品阶段 | 本地快速迭代与真实客户端模拟验收；待目标用户和跨平台实机验证   |
| 目标平台 | macOS、Windows、Linux 桌面端；预览版不依赖托管控制服务         |
| 目标用户 | 使用 AI 编程工具完成业务产品，但缺少 DevOps 经验的个人和小团队 |
| 核心结果 | 用户无需学习部署工具链即可完成可靠上线                         |

## 2. 问题定义

V1 已验证项目扫描、配置生成、CNB 构建、镜像发布、SSH、Compose、Caddy、健康检查和回滚等工程能力，但仍要求用户理解环境、分支、仓库、Token、SSH 私钥、密钥文件和域名路由之间的关系。

这类界面优化的是“熟练操作者填表效率”，不是“小白首次成功率”。ABCDeploy 必须从配置驱动转向结果驱动。

## 3. 核心用户

### 3.1 主要用户：Vibe Builder

- 能使用 Codex、Cursor、Claude Code 等工具完成前后端项目。
- 知道服务器 IP、域名和第三方 API Key，但不了解 Docker、反向代理和 CI/CD。
- 希望尽快得到一个可访问的测试地址，再逐步正式上线。
- 容易在 Token 权限、SSH、环境变量和构建日志处中断。

### 3.2 次要用户：多项目独立开发者

- 已完成过一次部署，希望后续项目复用 CNB、服务器和域名配置。
- 关注部署速度、回滚能力和项目状态，而不是重复完成一次性准备。

### 3.3 暂不服务

- Kubernetes、多集群、多区域容灾和复杂企业审批流。
- 需要自定义网络拓扑、Service Mesh 或基础设施即代码平台的专业运维团队。
- Windows Server 作为生产容器宿主机的场景。

## 4. 用户任务

> 当我完成一个本地项目时，我希望系统识别它需要什么，只问我无法自动获取的信息，然后把它可靠地部署到服务器，并在失败时告诉我唯一明确的下一步。

返回用户的任务是：

> 当我再次打开应用或开始第二个项目时，我希望直接看到现状并复用已有资源，不再重复配置账号和服务器。

## 5. 产品目标

1. 首次 staging 部署中位耗时不超过 10 分钟，不含第三方账号注册和 DNS 生效时间。
2. 黄金路径中不超过 5 个需要用户判断的决策。
3. 不要求用户打开终端、编辑 YAML、寻找 SSH 私钥路径或理解 Caddy。
4. 应用重启后恢复到原项目、场景和内部状态。
5. 第二个项目复用已有连接，3 分钟内进入首次构建。
6. 任何失败都有可操作的修复路径，并可从失败步骤继续。

## 6. 产品原则

### 6.1 询问意图，不询问实现

- 使用“先部署测试环境”代替“配置 staging target”。
- 使用“正式发布需要确认”代替“production_mode=approval”。
- 使用“连接服务器”代替“选择 SSH_KEY_PATH”。

### 6.2 推荐优先

系统必须先给出带理由的推荐方案。自定义配置是主动展开的高级能力，不能成为继续按钮前的必填表单。

### 6.3 安全动作显性，机械动作隐形

创建仓库、安装服务器组件、写入项目文件和正式发布需要明确确认。识别框架、生成随机密钥、检查端口、选择镜像标签等机械动作自动完成。

### 6.4 可恢复优先于一次成功

每个长任务和关键状态写入检查点。应用退出、电脑重启或网络中断后，用户应恢复到原场景和安全状态，而不是空白首页。

## 7. P0 范围

### FR-01 项目记忆

- 保存规范化项目路径、Git Remote、项目指纹、最近打开时间、当前场景和内部状态。
- 应用顶层始终显示项目侧边栏；单项目启动后自动进入上次项目，多项目按最近使用排序。
- 侧边栏使用用户可以直接判断下一步的具体状态，例如“正在部署测试版”“等待确认测试结果”“正式发布需要处理”“测试部署没有完成”“测试版正在运行”和“正式版可以访问”；不得只写“需要处理”“失败”或“等待部署”。
- 切换当前项目不取消其他项目的 CNB 构建；后台轮询所有活动任务。
- 路径失效时提供“项目可能被移动”恢复流程，已有配置不得丢失。
- 重复选择同一项目目录时复用现有项目身份和进度，不创建第二条项目记录。

### FR-02 只读识别

- 识别框架、服务、端口、Dockerfile、数据库、环境变量名称、Git 状态和远程仓库。
- 输出一段面向用户的结论，不先展示扫描细节。
- 扫描期间明确说明不会运行代码、不会读取环境变量的值；扫描结束后把结论保留在项目页，不能只依赖会消失的通知。
- 识别到服务时展示真实数量，说明本机运行不是上线前置条件，并用“准备上线”直接承接一次性上线设置；服务名称和本机可运行性留在当前页面供需要调试的用户查看。没有识别到服务时提示文件夹层级可能不对，并允许立即重新选择。
- 特定框架只用于提高自动配置精度，不能成为服务识别白名单。根目录或工作区业务子包的 `package.json` 已提供标准 `start` 或 `start:prod` 命令时，即使没有 NestJS、Next.js、Vite 等已知依赖，也应识别为通用 Node.js 服务，并按实际包管理器和所在目录生成安装、启动和容器文件；工作区统一调度根包不重复算作服务，也不能误报“文件夹层级不对”。
- 无服务校验即使发生在工作区生成之前，也必须转成同一条用户提示；原始字段名和清单校验信息不进入一级界面。
- 低置信度识别只询问一个可理解的问题，例如“哪个目录是后台管理端”。

### FR-03 全局资源

- CNB 账号、服务器、镜像库和 SSH 身份属于工作区资源。
- 新项目默认复用最近一次成功使用的资源。
- 项目只保存资源引用，不复制凭证。

### FR-04 CNB 授权

- 预览版主路径为引导用户在 CNB Web 创建最小权限访问令牌，并粘贴一次。
- 界面给出准确入口、建议名称和最小权限：`repo-code:rw`、`repo-cnb-history:r`、`repo-manage:rw`、`repo-cnb-trigger:rw`、`group-resource:rw`。
- 验证成功后令牌进入操作系统密钥库，后续只显示连接状态，不回显令牌。
- 未来在 CNB 支持公共客户端 PKCE/Device Flow，或项目引入可自托管授权中转后，再升级为浏览器自动返回授权。

### FR-05 服务器接入

- 用户只需先填写服务器公网 IP；用户名根据操作系统和历史记录推荐，并说明可在云服务器实例详情中查看。
- 自动检查 ssh-agent、SSH config 和系统常见密钥位置。
- 没有可用密钥时生成 ABCDeploy 专用 Ed25519 密钥。
- 支持使用服务器密码一次性安装公钥，密码不落盘。
- 手动选择私钥置于高级选项，文件选择器默认打开系统 SSH 目录。
- 验证成功后再单独确认安装 Docker、Caddy 和 ABCDeploy 运行目录。

### FR-06 一次性上线设置

- 一次性设置统一放在“管理版本”，依次为“连接代码平台、保存项目版本、准备测试环境、开启自动部署”。
- 每个当前步骤先给一个推荐答案和一个结果性主操作；替代方式只有在推荐方式不适用或用户主动展开时出现，不设计独立“推荐方案”页面。
- 用户可以提前展开后续步骤了解用途，但前置条件未完成时只显示依赖说明，不提前暴露服务器、密钥、配置表单或外部网页动作。
- 已验证且只有一个候选的全局资源自动复用；多个候选时才要求选择，任何占位值和仅存在于本机密钥库但未验证的凭据都不能冒充已准备。

### FR-07 只补缺失信息

- 项目 `.env.example` 是配置名称、注释和顺序的权威来源，未知字段原样保留。
- 默认优先展示没有值的必填项，其次展示没有值的可选项；“查看全部”展开已有值。
- 项目注释作为主标题，配置名称（例如 `DATABASE_URL`）作为副标题；没有注释时直接用配置名称作为主标题。
- 用户可以直接填写，也可以选择配置中心中的已有值；客户端不猜测字段业务含义。
- 本机、测试和正式配置相互独立，本机值不得自动进入远程环境。
- `DEPLOYDESK_*` 等由客户端维护的运行标记必须保留在实际配置文件中，但不进入用户配置列表、必填计数或配置中心候选。项目没有任何用户配置项时，页面直接说明“无需填写，使用当前配置即可”，并用“使用当前配置并继续”完成服务器准备，不能让小白查看或修改内部字段。
- 配置完成态只表达一次结果：说明保存位置（本机或运行服务器）、显示“已保存/已准备”，并把空列表提示改为“需要核对时可以查看全部配置”。不能连续重复“配置已齐全、服务器已准备、服务器已准备、没有缺少值”；没有用户字段时改为“项目默认设置已经准备 / 不需要额外操作”。
- 自动部署的一次性设置完成后，页面必须说明“代码合并到主分支后自动生成版本并更新测试版，正式版仍由用户确认发布”，不能只显示“安全配置位置已记录”等内部持久化事实。正式配置完成态只说明发布时会复用已保存配置。
- 敏感值进入安全存储，不写入 Git、普通日志或错误信息。
- 普通流程不得出现 CNB 密钥文件 URL、原始 YAML 或私钥正文。

### FR-08 部署前确认

- 只有真正发生远程写入或生产切换前才展示自然语言摘要：将创建或切换什么、写入哪里、是否影响现有服务。
- 风险动作分级为“无远程写入”“服务器初始化”“首次部署”“正式发布”。
- 原始文件 Diff 和部署方案保留在技术详情中，不恢复一页独立的旧式“查看部署计划”流程。

### FR-09 部署状态机

- 统一展示识别、验证、构建、推送、部署、健康检查和域名验证。
- 每一步有等待、成功、失败、可重试和需要用户处理状态。
- 失败恢复后从断点继续，不重复已经成功且幂等的动作。
- 需要用户在浏览器或云平台处理时保留当前任务；用户返回客户端后自动复核，主操作使用“检查并继续”或“检查并完成发布”，不能只留下含义模糊的“刷新状态”。

### FR-10 日常项目页与版本管理

- 顶部使用“在本机运行、管理版本、部署测试版、发布正式版”四个稳定场景页签。
- 每个页签第二行显示当前状态，内部根据准备、进行、验证、成功或失败状态切换页面内容。
- 返回用户恢复上次场景，不重复完成一次性准备流程。
- 正式发布必须基于测试通过的版本，本机运行不是测试部署的强制前置条件。
- 合并 `main` 自动生成新版本并部署测试；客户端关闭不影响远程任务。
- 用户手动刷新版本记录时，按钮显示正在刷新；成功后持续显示本次更新时间，即使列表没有变化也能确认操作已经完成。刷新失败继续保留原版本记录，并提供明确原因和恢复动作。
- 版本目录保留多个测试通过版本，正式环境可以选择其中任意版本发布或恢复。
- 有多个可发布版本时，选择列表必须同时显示“当前测试版”与“已选择”两个事实：默认候选写“当前测试版 · 已选择”；改选旧版本后，当前测试版写“当前测试版 · 推荐”，旧版本写“已选择”。
- 测试版确认通过后，页面只分层表达一次确认结果、一次运行事实和一个下一步：标题“测试版已通过验证”，运行卡说明“测试版正在服务器运行”，没有未决正式任务时才提供“下一步：准备正式版”。只有正式任务已经完整成功时才写“当前版本已经在线”并隐藏发布入口；如果同一版本只是在正式服务器运行但仍缺网页保存、地址或其他条件，则必须显示具体待办并继续原任务。测试地址是否能在当前电脑重新打开，不得影响已保存的测试结论和发布入口，同一屏不得用四组近义文案反复说明已经测试通过。
- 版本优先显示生成时间、用户明确填写的版本名和环境资格；普通提交说明作为“修改说明”，完整提交号只在技术信息中展示，用户不需要根据哈希猜测版本差异。
- 当前正式版本不提供重复发布动作；历史版本使用“恢复”语义，发布和恢复都必须在展示当前版本与目标版本后再次确认。
- 同一项目只允许一个未结束的正式发布意图。网页保存、授权、排队、运行、地址或失败恢复都必须绑定原始不可变版本；完成或明确取消前，版本目录、测试结果和任务中心只能继续原任务，不得创建第二条生产任务或悄悄替换候选。
- 正式配置尚未确认时，父页面只给出“先完成正式配置”一个结论，展开区使用中性的“配置内容”，不预判一定存在缺失项；具体缺少数量或无需填写由配置模块根据真实状态说明。同一屏不得连续重复“准备、确认、等待保存”等近义状态，也不得同时出现“没有需要填写”和要求空保存的矛盾提示。
- 版本的身份由完整提交、服务集合和不可变镜像摘要共同确定；生产发布不得重新构建。
- CNB Web 或手机端发起的生产发布会同步回桌面端历史，并核对目标服务器实际镜像摘要。

### FR-11 多项目复用

- 新项目自动建议已有 CNB 账号和服务器。
- 检测端口、域名、容器名称、数据库名称和网络冲突。
- 复用前展示隔离结果，不要求用户理解命名空间实现。
- 多项目可同时构建；绑定到同一物理服务器时，运行态切换和 Caddy 更新自动排队。

### FR-12 面向行动的错误

错误页固定包含：

1. 当前卡在哪一步。
2. 系统已经验证了什么。
3. 最可能原因。
4. 一个主修复动作。
5. “查看技术详情”和“换一种方法”次级入口。

## 8. P1 范围

- CNB 公共客户端 OAuth/Device Flow 或可自托管授权中转。
- GitHub、Gitee 等代码托管授权和仓库选择。
- 腾讯云 DNS 自动配置及其他 DNS Provider。
- 腾讯云 CVM/Lighthouse API 接入，自动添加 SSH 公钥和读取主机指纹。
- PR 临时预览环境。
- 桌面通知、部署审计和团队协作。
- 自动更新、崩溃恢复和匿名漏斗分析。

## 9. 非目标

- P0 不提供任意流水线可视化编排器。
- 不要求用户在 ABCDeploy 中编辑所有服务器文件。
- 不承诺自动修复业务代码错误；只给出诊断和可复现命令。
- 不以嵌入第三方网页并读取页面内容的方式获取凭证。
- 不在用户未确认时清理容器、目录、数据库或 DNS 记录。

## 10. 非功能要求

### 安全

- 长期凭证进入系统密钥库；结构化状态不包含凭证值。
- 当前 PAT 使用最小权限并进入系统密钥库；未来 OAuth 使用 `state`、PKCE、短期授权码和可撤销令牌。
- SSH 主机指纹必须保存并在变化时阻止自动连接。
- 日志、遥测、错误报告和 Diff 统一脱敏。
- 外部网页需要粘贴的完整安全配置只在内存中生成；确认保存后，如果系统剪贴板仍是该内容则主动清空，不能覆盖用户后来复制的新内容。
- 所有正式发布和破坏性操作生成可审计记录。

### 跨平台

- Windows 不依赖 WSL、Bash 或用户自行安装 OpenSSH 才能完成主流程。
- macOS、Windows、Linux 的行为、文案和恢复语义保持一致。
- 系统差异只在高级诊断中出现。

### 可访问性

- 全部主流程支持键盘操作。
- 状态不只依赖颜色表达。
- 默认字号、对比度和焦点样式达到 WCAG 2.2 AA 的相关要求。
- 通用交互优先使用现有 `components/ui` 和 Radix UI Primitive，不自行复制焦点管理和键盘行为。

### 性能

- 已知项目启动恢复目标小于 2 秒。
- 只读识别 90 分位小于 10 秒；大型 monorepo 提供渐进结果。
- 长任务不阻塞界面，用户可关闭应用后继续。

## 11. 成功指标

| 指标                      | P0 放行标准    |
| ------------------------- | -------------- |
| 无协助首次 staging 成功率 | 至少 80%       |
| 首次部署有效决策数        | 不超过 5 个    |
| 中位完成时间              | 不超过 10 分钟 |
| 返回用户恢复成功率        | 100% 测试样本  |
| 第二项目复用成功率        | 至少 90%       |
| 高级设置打开率            | 低于 10%       |
| 无恢复路径错误            | 0 个           |
| 凭证泄漏                  | 0 个           |

## 12. 预览版约束

1. CNB 当前公开 OAuth 流程需要 `ClientSecret`，不适合把长期密钥内置在开源桌面端；预览版不伪装成一键 OAuth。
2. CNB 密钥仓库由用户在 Web 端创建，并在流程真正需要时分别保存测试、生产配置；客户端负责复制当前环境模板、打开准确页面和验证后续构建结果，不要求首次设置时提前完成生产配置。
3. 预览版不引入托管控制面或服务器 Agent，源码、业务密钥和服务器 Shell 不经过 ABCDeploy 云服务。
4. CNB 是首版 Pipeline Provider，TCR 是推荐 OCI Registry Provider；产品模型保留其他代码平台、云厂商和 Registry 的适配边界。
