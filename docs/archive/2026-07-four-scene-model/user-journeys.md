# ABCDeploy 当前用户旅程

> 状态：当前产品基线。与代码和文档资产对齐于 2026-07-16。
> 详细状态、文案和交互遵循 [整体产品工作流](product-workflow.md) 与 [前端页面设计规范](frontend-design-guidelines.md)。

## 1. 第一次打开应用

1. 用户先看到“选择整个项目文件夹”的说明，再打开系统文件夹选择器。
2. 系统只读识别项目，不运行代码，也不读取真实配置值。
3. 识别完成后，页面持续展示发现的服务数量，并明确“本机运行不是上线前置条件”。
4. 用户可以进入本机页面，也可以点击“准备上线”直接开始一次性设置。
5. 添加和切换项目始终在侧边栏完成，不进入项目内部流程。

如果没有识别到服务，页面保留原因和“重新选择文件夹”，不创建一条无法使用的项目记录。

## 2. 项目中的四个场景

```text
在本机运行｜管理版本｜部署测试版｜发布正式版
```

四个入口是稳定导航，不是必须依次完成的步骤。每个入口第二行显示当前事实或下一步；用户可以进入受阻场景查看原因，页签本身不因前置条件而禁用。

## 3. 按需在本机运行

1. 页面依次展示项目服务、运行依赖和必要配置。
2. 用户可以逐个或批量启动、停止和重启服务。
3. 必要配置优先展示缺少值的项目，可直接填写或使用配置中心已有值。
4. 只有要求运行的服务最近一次完整启动成功、运行依赖可用且阻塞配置已有值时，才开放项目访问入口。
5. 项目代码、依赖或构建错误只整理为脱敏问题信息，交回用户的开发工具处理。

用户可以在这里结束，不需要连接 CNB、镜像仓库或服务器。

## 4. 首次准备自动部署

用户从“准备上线”进入“管理版本”，按真实缺口完成四项一次性设置：

1. **连接代码平台**：选择或创建首版 CNB 代码仓库；非 Git 项目由系统安全建立 `main` 和首次提交。
2. **保存项目版本**：复用或验证镜像仓库。由代码仓库派生的保存位置自动完成，不制造第二次确认；已有 TCR 等独立位置先真实验证再使用。
3. **准备测试环境**：连接 Linux 服务器，保存独立测试配置，并准备测试地址或本机安全预览。
4. **开启自动部署**：在代码平台网页完成一次安全保存。客户端逐段提供文件名和配置内容，返回后由用户明确确认。

页面允许提前展开后续项了解用途，但前置条件未完成时不提前暴露服务器、密钥、配置表单或网页操作。完成最后一项后自动进入测试场景，不再增加一次没有新信息的“继续”。

## 5. 日常生成并验证测试版

1. 用户把准备上线的代码合并到项目主分支。
2. CNB 自动构建不可变镜像并更新测试环境；客户端关闭不影响远程任务。
3. 客户端的“部署任务”统一展示后台进度，用户可以离开当前项目。
4. 部署完成后，用户打开明确标识版本的测试地址或本机安全预览。
5. 用户选择“测试通过”后，该不可变版本进入可发布候选；选择“还需要修改”则回到开发工具处理。

客户端仍提供“手动部署测试版”，但只用于首次上线和异常恢复，不取代主分支自动部署主线。被后续测试版替换且未确认的版本显示“未完成测试 · 不可发布”。

## 6. 管理多个版本

“管理版本”只把真正部署到测试环境的成功结果计入版本记录。每条版本优先展示生成时间和用户可理解的修改说明，完整提交、构建编号和镜像摘要放入技术信息。

- 多个测试通过版本可以同时保留。
- 当前测试版是首次发布的默认推荐，但用户可以明确选择任何测试通过版本。
- 当前正式版不显示重复发布；较早版本使用“用此版本恢复正式版”。
- 刷新版本后保留本次检查时间，即使列表没有变化也给出反馈。
- 失败、取消和被替代任务属于部署记录，不计入版本数量。

## 7. 发布或恢复正式版

1. 没有测试通过版本时，页面解释原因并只提供“前往测试版”。
2. 用户选择一个测试通过版本；选择只属于本次发布意图，不成为长期偏好。
3. 页面依次准备正式配置和正式地址，每次只展开当前尚未完成的一项。
4. 最终确认明确目标版本、正式地址、是否属于恢复，以及生产不会重新构建镜像。
5. 系统用测试时记录的同一不可变镜像启动正式服务；启动失败时，有上一健康版本才自动恢复。
6. 正式服务已经运行但 DNS 或 HTTPS 尚未完成时，统一显示“正式版已部署，还差地址”，并保留“检查并完成发布”。
7. 全部地址可访问后，页面展示正式版本和按网页/H5、API、其他服务排序的全部访问入口。

同一项目只允许一个未完成的正式发布意图。它可能停在授权、网页保存、部署、地址或失败恢复；任务结束前，版本目录、测试结果页和任务中心都只提供继续原任务的入口，不能悄悄换成另一个候选或创建第二条生产任务。

## 8. 返回应用和跨项目工作

- 应用先恢复项目列表、上次项目和准确场景，再在后台读取本机与远端状态。
- 已保存的可信状态先显示，后台复核只局部更新，不用临时默认值闪现“0 个版本”“未启动”或“尚未发布”。
- 侧边栏、所有项目和部署任务使用同一套跨环境优先级：等待用户处理、失败、自动处理中、最近成功。
- 测试和正式任务按“项目 + 环境”独立保留；较新的测试成功不能覆盖仍待处理的正式发布。
- 点击任务携带准确任务、环境和版本，直接回到原停点。
- 项目目录移动后允许重新定位并迁移原项目身份，不丢失版本、服务器、配置和任务。

## 9. 配置中心

配置中心是可选工具，只保存配置说明、配置名称和值。项目字段可以把名称匹配的已有值一次性填入当前环境；之后项目保存自己的环境配置，不形成会被全局修改自动联动的持续引用。

本机、测试和正式配置相互独立。系统不根据英文名称自行翻译或猜测业务含义，项目注释始终是页面说明的首选来源。

## 10. 失败和外部操作

任何失败或等待处理页面都回答：发生了什么、影响哪个结果、系统已经完成什么、用户现在做什么、是否会重复部署。

- 授权不足：在原任务中更新授权并继续；确认远程任务未创建时才重新部署当前代码。
- 构建或代码失败：复制脱敏信息交给开发工具，不在客户端伪装自动修复。
- 服务器或配置失败：回到对应的一次性设置项，保留其他已完成结果。
- DNS/HTTPS 未就绪：保留已经启动的正式服务，只处理精确的 A/AAAA 记录和地址复核。
- 需要网页保存：状态绑定项目、环境和不可变版本；重启后重新生成可复制内容，但不落盘完整秘密。
- 外部状态仍未就绪：正常显示最近检查时间和具体对象，不把“仍需处理”误报成请求失败。
