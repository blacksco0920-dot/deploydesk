# ABCDeploy 整体产品工作流

> 状态：当前规范性工作流，已与桌面客户端实现对齐；最近对齐：2026-07-16。

页面布局、组件、视觉、文案、状态和可访问性统一遵循 [前端页面设计规范](frontend-design-guidelines.md)。

## 1. 应用结构

左侧项目栏负责添加、切换和显示项目摘要，不把“选择项目”作为部署步骤。项目主区域顶部使用四个稳定场景页签：

```text
在本机运行｜管理版本｜部署测试版｜发布正式版
```

页签名称表达用户要完成的动作，第二行显示当前状态，例如“3 个服务运行中”“2/4 项已准备”“尚未发布”。

## 2. 为什么不使用五步进度条

本机运行、测试部署和正式发布不是一次性、必须从头走到底的向导：

- 用户可以只使用本机服务启停。
- 用户可以直接准备和部署测试版，不必重复本机验证。
- 测试版会反复部署和验证。
- 正式发布只在需要时发生。

因此不使用“本机运行 → 准备上线 → 部署测试版 → 确认测试 → 正式发布”的全局 Stepper。一次性准备属于“管理版本”，部署和业务确认属于“部署测试版”；它们都是场景内部状态，不是全局导航项。

## 3. 四个场景

### 在本机运行

提供项目服务和运行依赖的快捷启停，以及后续部署需要的配置检核。该场景可以独立使用，详细定义见 [本机运行产品定义](local-running.md)。

### 管理版本

首次接入时集中完成四项一次性设置：连接代码平台、保存项目版本、准备测试环境、开启自动部署。一级标题只表达用户动作，CNB、TCR、Token 和 SSH 等实现信息只在确实需要用户处理时进入展开内容。完成后，合并 `main` 会自动生成版本并部署到测试环境。

新项目没有明确指定资源、且工作区只有一套已验证候选时，自动复用已有镜像仓库凭据和运行服务器，不再要求用户重复点击确认；存在多个候选时才让用户选择。镜像仓库“已验证”必须来自目标端点真实接受登录信息，不能只依据本机密钥库中存在用户名和密码。

版本记录先用生成时间、用户明确填写的版本名和“测试通过/当前环境”等资格帮助用户识别。普通英文提交说明或“初始化项目、完成首次上线配置”等系统动作只作为“修改说明”；多个版本说明相同时也必须用各自生成时间区分。只有测试环境实际运行成功的结果进入“版本记录”；当前正在测试但尚未确认的版本显示“等待确认测试结果”并提供验证入口，已经被后续测试版替换的未确认版本明确显示“未完成测试 · 不可发布”，不伪装成仍等待用户完成的待办。用户确认后才显示“测试通过”。失败、取消和待处理属于部署记录，默认折叠且不计入版本数量。完整提交、CNB 构建序号和服务器核对的不可变镜像摘要只放在“技术信息”中。多个测试通过版本可以同时保留为正式发布候选；正式发布不再强绑定最后一次测试，而是从这些版本中明确选择。正式环境当前版本不显示重复发布操作，较早版本使用“用此版本恢复正式版”，明确操作目标；发布和恢复均需二次确认。用户手动刷新后必须留下本次更新时间；列表没有变化也是有效结果，不能让按钮恢复原状后完全没有反馈。

存在多个测试通过版本时，首次正式发布默认选择当前正在测试环境运行的版本，并在选择列表明确标记“当前测试版 · 已选择”。用户改选历史版本后，原当前测试版改为“当前测试版 · 推荐”，历史版本只写“已选择”，让默认依据和当前选择同时可见。

### 部署测试版

首次接入设置未完成时，本页只解释“上线设置还差什么”，并提供前往“管理版本”的单一入口，不在测试页复制第二套设置流程。一次性设置统一在“管理版本”按以下顺序完成：

1. **连接代码平台**：保存项目代码；项目尚未使用 Git 时，客户端自动建立 `main` 和首次提交，同时排除 `.env`、私钥等本机敏感文件。若所选目录只是另一个 Git 项目的子目录则阻断并提示选择完整项目。
2. **保存项目版本**：准备项目版本保存位置并生成构建、镜像和环境部署所需文件；已有唯一一套已验证保存位置时直接复用。
3. **准备测试环境**：集中准备运行服务器、测试配置和测试地址。没有需要填写的配置时自动完成，不要求用户做一次空保存；默认优先使用本机安全预览，已有测试域名时也可直接使用。
4. **开启自动部署**：完成一次 CNB Web 安全保存。页面按“打开 CNB 创建保存位置 → 准备要保存的内容 → 打开网页粘贴并保存 → 返回确认”逐段只展示一个推荐主操作；“密钥仓库”等平台内部概念不作为主按钮文案。仓库完整路径默认隐藏，已经手动创建的旁路先折叠为“我已经有保存位置”，进入旁路后才显示已有位置和继续操作。最后确认后自动进入测试版，不再要求用户额外点击一次“继续”。之后合并 `main` 自动生成版本并部署到测试环境，不再重复要求授权。

完成态不再介绍安全配置保存位置，而是直接说明日常使用方式：“代码合并到主分支后自动生成版本并更新测试版；正式版仍由你确认发布。”

设置完成后，“部署测试版”只负责以下闭环：查看 `main` 自动部署的当前进度、确认正在验证的具体版本、打开测试地址、确认业务结果，以及在失败后从保留的阶段继续。用户也可以在这里主动触发一次新版本部署，但不会重新填写服务器、配置和地址。

已经开启自动部署后，客户端主动触发入口必须明确写成“手动部署测试版”，作为首次上线和异常恢复的辅助动作；不得继续用“部署新版本”暗示每次代码更新都需要回到客户端。主线始终是合并 `main` 后自动生成版本并更新测试环境。

已保存的代码仓库应立即恢复为“已准备”，账号资料和远端权限只在后台检查，不能让用户每次进入项目都等待。客户端专用授权暂时不可读时，可安全复用系统已有的 CNB Git 凭据；只有实际同步也失败时才要求更新授权。

已核对成功的测试和正式配置按“项目 + 环境 + 服务器”保存最近可信状态。再次进入同一项目时先恢复页面并在后台复核服务器摘要，不阻塞用户查看；首次连接、更换服务器或上次未通过时仍等待真实核对，实际部署前始终重新验证，缓存不能代替发布门禁。

侧边栏提供统一“部署任务”入口，后台轮询所有已知活动任务，不依赖用户停留在某个项目页。任务按“项目 + 环境”独立保留，不能只读取项目最近一条记录；较新的测试部署不得覆盖同一项目仍待处理的正式发布。任务中心必须把“等待你处理”和“自动处理中”分开：只有网页确认、业务验证、授权修复等必须由用户完成的事项进入待处理区，构建和发布等后台任务单独说明可以离开页面；两类任务同时存在时不能用一个含糊标题合并。任务条目使用“准备项目、生成版本、启动服务、确认可用”四个结果阶段；失败时同时说明已经完成的阶段和唯一下一步。最近成功的测试部署和正式发布也保留在任务中心，用户离开应用后回来仍能确认结果；全局弹窗只展示最近 4 项并提示其余结果可在项目内查看，不能让历史成功卡片把人工待办挤成部署日志。排序使用任务发生时间，后续状态复核不得改变历史顺序。点击任务应携带准确任务和版本，直接回到对应的上线设置、测试部署或正式发布停点，而不是只打开项目首页。

全局项目摘要必须保留最近可信结果：正式容器已经运行而 DNS、HTTPS 或地址确认尚未完成时，侧边栏、“所有项目”和任务中心统一写“正式版已部署，还差地址”，不能在项目页写成功、全局入口却退回泛化的“正式发布需要处理”。项目场景进一步写“还差设置地址”，不能写“等待地址生效”暗示用户不需要新增或核对解析记录。

项目内的部署结果卡与任务中心使用同一套四阶段语义。测试部署显示“准备项目、生成版本、启动服务、确认可用”，正式发布显示“准备项目、确认版本、启动服务、确认可用”，不能让“生成版本”暗示生产环境会重新构建。任务进入等待处理或失败后，阶段条不能消失；必须标出当前停点，并明确说明任务与已完成步骤已经保留。同一状态只突出一个恢复动作：需要上线设置时进入设置，需要本机安全预览时直接打开预览，需要复核外部条件时检查并继续，失败后才重新部署。普通正式发布失败应允许用户在处理原因后直接“重新发布同一版本”，不得让用户重新选择版本或误以为系统会再次构建。

如果下一步需要用户到浏览器、DNS 或云平台处理，客户端保留当前任务。用户返回时立即自动复核，停留在可见页面时也定期继续检查；页面仍保留“检查并继续”“检查并完成发布”等立即检查动作，并显示最近检查结果，不能让用户判断刚才是否已经部署成功，也不能要求用户反复手动刷新。

正式地址填写阶段必须提前展示域名需要指向的服务器公网 IP，并提供复制入口；根据服务器地址准确说明需要添加 A 或 AAAA 记录，发布后系统会自动检查。不能等部署完成后才第一次暴露 DNS 依赖，也不能因为 DNS 尚未生效而丢失已经启动的服务或要求重新构建。

正式服务已经在服务器运行、只剩 DNS 或 HTTPS 时，一级结论统一写“正式版已经部署”，不能降级成“部署失败”。页面列出当前项目全部正式地址及每一条需要添加的 `A/AAAA 记录 → 服务器 IP`，用户完成后只需“检查并完成发布”，无需重新选择版本或重新部署。客户端应自动修复由当前项目设置生成的缺失 Caddy 路由；只有地址正被其他项目使用时，才要求用户确认接管。

一次公网复核存在多个未就绪地址时，一级原因必须汇总实际检查到的全部失败地址，不能只展示第一个却在下方列出更多待办。每条地址继续保留独立的 DNS、HTTPS 或响应原因，已经完成的容器部署结论不受影响。

客户端可以通过公开 NS 记录识别域名服务商，并提供对应管理控制台入口，帮助小白找到实际操作位置；这项判断不能读取云账号或部署密钥，也不能把所有项目写死为首版试点厂商。同一域名区只查询和展示一次；项目跨多个域名服务商时分别给出入口。识别失败时保留通用步骤，不阻断解析和后续自动复核。

外部安全保存也必须跨重启恢复。客户端只记录“仓库创建页已打开、仓库已确认、保存页已打开”等非敏感阶段和仓库位置，不落盘完整环境配置；重新进入时按需重新生成同一份内容，并直接回到对应的创建或网页保存确认，不让用户重复开始。

完整安全配置只在用户明确复制时进入系统剪贴板。用户确认网页保存成功后，如果剪贴板仍是刚才那份配置，客户端主动清空；如果用户已经复制了其他内容，则不能覆盖用户的新剪贴板。该清理失败不改变已经确认的远端停点，但不得把敏感内容写入普通状态或日志重试。

CNB 授权失效属于当前部署任务的恢复动作，不要求用户离开任务去寻找全局设置。客户端在原页提供“更新 CNB 授权”，验证新令牌后继续检查同一任务；如果确认 CNB 从未创建远程任务，则切换为“重新部署当前代码”，避免用户反复点击无效的状态检查。

一次性设置完成后，同一页签内部依次进入：

```text
准备测试版
→ 正在部署测试版
→ 验证测试版
→ 测试版已通过验证
```

用户选择“还需要修改”时回到开发工具处理，再重新部署；选择“测试通过”后开放正式发布。

测试通过后的主线动作是“准备发布正式版”，“重新打开测试版”降为复查动作。页面只保留“测试版已通过验证”一个确认结论；运行卡说明服务器事实，复查卡只说明不会重新部署，下一步卡只说明继续使用当前测试版本。当前测试版本已经是线上正式版时不再提供重复发布入口，只说明“当前版本已经在线”。即使客户端暂时无法恢复测试地址，已经保存的业务确认仍然有效，用户仍可继续进入正式发布。

测试通过后的页面说明、运行结果和复查入口分别表达“已经通过验证”“当前仍可用”“需要时重新打开”，不能在三个区域重复同一句“仍在服务器运行”。应用重启后的提示只解释重新打开不会再次部署。

### 发布正式版

没有测试通过的版本时仍允许进入页签，但只解释前置条件并提供“前往测试版”。不能简单禁用页签，让用户无法理解原因。

同一项目任一正式发布尚未结束时，授权、网页保存、排队、部署、地址或失败恢复都属于同一个未决发布意图。版本目录、测试结果页、任务中心和正式页必须继续原任务及其不可变候选；在用户完成或明确取消前，不创建第二条正式发布，也不显示会替换候选的“准备发布正式版”或新的测试部署主操作。

测试通过后，用户准备正式配置和正式地址，再明确点击“发布正式版”。正式版必须复用测试通过的同一版本，不重新构建。

正式版首页默认展示线上当前版本和访问状态。历史版本选择只在用户从版本目录明确发起发布或恢复时生效，不作为长期项目偏好保存；用户重新进入项目或直接点击“发布正式版”时，不得自动恢复上一次未执行的历史版本选择。

部署任务中心是例外的显式入口：用户点击某次失败任务时，客户端应以一次性导航参数打开该任务对应的测试候选和停点。离开任务上下文后参数随页面销毁，不污染项目下次普通打开的默认状态。

正式发布页使用渐进披露：顶部只用三行概括“准备发布的版本、正式配置、正式地址”，下方每次只展开当前尚未完成的一项。正式配置未完成时不同时展示地址表单；配置完成后自动进入地址；全部准备完成后才突出发布操作。已完成项默认收起，通过“修改”按需重新打开，避免让小白在同一屏同时判断多组按钮。父页面只给出“先完成正式配置”一个结论，展开区使用“配置内容”，由配置卡准确区分缺少值、无需填写或服务器已准备；不能在同一屏连续重复“准备、确认、等待保存”。

配置表单只展示项目真正需要用户理解和提供的字段。`DEPLOYDESK_*` 等客户端内部运行标记仍随原配置保存和同步，但从表单、缺失统计和复用选择中隐藏；如果隐藏后没有任何用户字段，正式环境明确写“无需填写，使用当前配置即可”，主操作为“使用当前配置并继续”；测试环境继续自动准备，不制造一个只能展开内部变量的“查看全部配置”。

配置完成后，配置卡只保留“配置已保存到本机/运行服务器”“已保存/已准备”和“需要核对时可以查看全部配置”三层不同信息。空配置自动准备后写“项目默认设置已经准备 / 不需要额外操作”，不能把同一个成功结果在标题、徽标和空列表中重复三次。

运行配置尚未完成时，后续网页安全保存只说明“先完成测试/正式配置”，不能预判用户一定需要填写或保存字段，也不能在尚未进入外部操作前提前要求理解 CNB 等提供者。

最终确认只复述影响决策的事实：目标正式地址、当前版本将切换或恢复到哪个测试通过版本，以及镜像不会重新构建。已有正式版本时，明确说明“目标版本启动失败会自动恢复当前版本”；不能用“新服务启动失败”等容易被理解为新增服务的表达。首次发布没有上一版本时不得虚构回滚保障。技术摘要和内部阶段不进入最终确认。

多服务项目完成部署后，访问入口优先展示网页、H5 或静态站点，再展示 API 和其他服务；正式成功页列出全部已验证地址及对应服务名称，不能把清单中第一个后端域名误称为“正式网站”。

客户端优先使用 CNB 构建触发权限直接晋级；令牌缺少触发权限时，自动把已验证的完整提交推送到 ABCDeploy 专用审批分支，由仓库内的 `cnb:apply` 调用同一条生产流水线。两种入口都只读取测试阶段已经标记的不可变镜像摘要，不要求用户更换令牌，也不把普通业务分支当成生产环境。

审批分支本身只是一条控制记录，不产生新版本，也不得覆盖“测试已通过”的状态；版本目录只收录真正执行测试部署或正式部署的记录。

如果正式发布由 CNB 网页、手机或另一台电脑完成，客户端根据生产记录关联的测试提交和服务器镜像恢复真实运行状态；这只用于展示当前正式版，不自动授予其他版本“可发布”资格。

配置中心是应用级的可选复用工具，不是添加项目、本机运行或上线的前置步骤。项目缺少配置时允许直接填写；只有同一配置会在不同项目中反复使用时，才建议保存到配置中心。项目字段只展示名称匹配的已有配置，选择后填入当前环境，避免用户先去维护一套并不需要的全局配置。

测试与正式配置均独立保存，但同一个 CNB 账号默认复用一个安全仓库，并以项目和环境独立文件名隔离内容。任一项目确认过安全位置后，后续项目直接使用该位置新增自己的文件，不再要求重复创建仓库；用户仍可按需更换。首版使用 CNB 时，系统负责生成已脱敏说明和完整安全配置内容、复制到剪贴板并打开对应页面；只有账号首次创建仓库时提醒选择“密钥仓库”而不是默认公开仓库，正式配置直接在已记录仓库中新增文件，不再次诱导用户创建。用户在 CNB 保存后，客户端只记录配置位置，并在部署时验证其是否真的可用，不能把无法读取的外部保存结果宣称为“已验证”。该操作是平台安全边界，不得伪装成已经自动完成，也不要求用户理解 CI 变量或流水线文件。

## 4. 强制关系

本机运行与测试版之间不建立强制门槛，系统可以推荐先本机验证，但不能阻止用户直接部署测试版。

唯一必须保持的核心门槛是：

```text
正式发布必须基于测试通过的版本
```
