# ABCDeploy 当前实现与验收基线

> 状态：`0.2.0-preview.4` 封板发布候选已完成本地自动化与模拟用户验收。
> 对齐日期：2026-07-16。本文只保留当前可复验结论；含真实试点现场的逐次验收日志由维护者私有留存，不进入公开仓库。

本文只记录当前仍成立的能力和门禁，不再追加按时间排序的开发流水账。产品规则分别以 [整体产品工作流](product-workflow.md)、[产品需求文档](product-requirements.md) 和 [前端页面设计规范](frontend-design-guidelines.md) 为准。

## 1. 当前已形成的闭环

### 应用与项目

- 侧边栏统一提供所有项目、配置中心、部署任务、项目切换和添加入口。
- 新项目执行只读识别；已有项目恢复时不等待 Docker、Git 或远端检查才显示首屏。
- 项目目录移动、重复选择、无服务目录和嵌套 Git 子目录均有独立恢复路径。
- 当前项目、场景、版本、任务、外部网页停点和最近可信状态可以跨重启恢复。

### 四个项目场景

- 顶部固定为“在本机运行、管理版本、部署测试版、发布正式版”。
- 本机运行不是测试部署门禁；正式发布只接受测试通过的不可变版本。
- 场景页签可进入查看原因，不依赖全局五步进度条。
- 页面状态、侧边栏、所有项目和部署任务使用同一套跨环境结论。

### 在本机运行

- 项目服务、运行依赖和必要配置按用户关注顺序纵向展示。
- 服务和依赖支持逐个与批量启停；启动中显示耗时、可取消，并对端口、代理和无输出超时给出明确反馈。
- 默认使用稳定容器运行；只有全部服务都能可靠支持时才显示开发调试。
- 支持已知框架和通用 Node.js 根项目/工作区服务；包管理器安装与裁剪发生在镜像构建阶段，运行容器不临时联网下载工具。
- 自动生成的静态网页镜像统一使用 Caddy，不生成 Nginx 配置；项目自有 Dockerfile 保持原样。
- 代码与构建问题提供脱敏摘要和“复制给开发工具”，不在客户端内连接 AI 自动修复。

### 管理版本与首次上线

- 一次性设置统一为“连接代码平台、保存项目版本、准备测试环境、开启自动部署”。
- 真实可复用的 CNB、TCR 和服务器会自动建议或复用；占位仓库、未验证凭据和模板值不会被当作已准备。
- 非 Git 项目可安全建立 `main` 和首次提交，并排除 `.env`、私钥和客户端内部状态。
- CNB 网页安全保存按单动作顺序引导，外部停点跨重启恢复并绑定具体项目、环境和版本。
- 版本记录按不可变提交与镜像集合去重，区分测试通过、等待确认、不可发布、当前测试和当前正式。

### 部署测试版

- 主分支更新自动触发 CNB 构建和测试部署；客户端可以关闭或切换项目。
- 手动部署只作为首次上线和异常恢复辅助入口。
- 测试部署、业务确认和正式发布是不同事实；只有用户确认通过才产生可发布资格。
- 本机安全预览、真实测试地址、多个服务入口和无法恢复地址的分支均有明确结果。

### 发布正式版

- 可以选择任意测试通过版本发布或恢复；生产复用同一镜像摘要，不重新构建。
- 候选版本只属于当前意图；普通进入正式页默认回到线上当前版本。
- 同一项目的未完成正式发布构成项目级门禁，所有入口都继续原任务和原候选。
- 配置、地址和最终确认按渐进披露依次出现；空配置不会制造一次无意义保存。
- 正式服务运行但 DNS/HTTPS 未完成时保留部署成果，显示全部地址、精确记录值和“检查并完成发布”。
- 版本启动失败时只有存在上一健康版本才承诺自动恢复；首次发布不虚构回滚保障。

### 全局部署任务

- 测试与正式任务按“项目 + 环境”分别保存，不互相覆盖。
- 人工待办、后台任务和最近完成结果分区展示；最近结果限量，全量历史回到项目查看。
- 任务使用“准备项目、生成/确认版本、启动服务、确认可用”四个用户阶段，并保留失败停点和唯一恢复动作。
- 点击任务携带准确任务和版本，直接恢复上线设置、测试确认或正式发布停点。
- 外部条件返回后自动复核，窗口隐藏时暂停轮询，恢复前台时补查一次；无活动任务时降低刷新频率。

## 2. 配置与安全基线

- 配置中心只维护配置说明、配置名称和值，是可选复用工具，不是上线前置。
- `.env.example` 的注释、名称、顺序和未知字段保持原样；客户端不猜测字段含义。
- 本机、测试和正式配置独立保存；内部 `DEPLOYDESK_*` 字段不进入普通表单，但保存时不丢失。
- Token、私钥、密码、敏感连接串和分环境运行配置使用系统密钥库；SQLite、Git、普通日志和错误卡不保存明文。
- 第三方原始响应、命令输出、路径和镜像摘要默认只进入脱敏技术详情。
- CNB 安全配置只在需要时于内存生成并复制；用户确认网页保存后，如果剪贴板仍是该份内容，客户端主动清空；若用户已经复制其他内容则不覆盖。
- 服务器主机指纹首次确认并固定，变化时阻断；一次性服务器密码使用后立即清空且不落盘。

## 3. 体验、性能和可访问性基线

- 项目首屏先恢复本地可信信息，远端和 Docker 状态在后台局部补齐，不用整页等待外部请求。
- 项目切换、异步保存、同步和部署结果绑定原项目，不能覆盖用户后来打开的项目。
- 本机服务状态批量读取，轮询按前台/后台和活动任务自适应；相同远程检查在同一轮合并。
- 页面不主动滚动抢走用户位置；长任务、刷新和检查即使结果未变化也留下时间反馈。
- `640 × 480`、`1280 × 720` 和 `1440 × 960` 有明确布局门禁，不允许页面级横向溢出。
- 状态不只依赖颜色；键盘、焦点、Label、对象化按钮名称、增强对比度、强制颜色和减少动画均有实现与自动化约束。
- 项目移除、任务继续、版本更换等图标或短文案操作，在可访问名称中包含具体对象和真实后果。

## 4. 当前验证范围

项目约定的本地门禁如下，具体执行范围按改动风险决定；功能变更完成后生成签名 `.app`，纯文档整理只运行文档链接、格式和一致性检查：

```bash
pnpm --filter @abcdeploy/desktop test
pnpm --filter @abcdeploy/desktop build
cargo test --workspace
cargo clippy --workspace --all-targets -- -D warnings
pnpm tauri:build:app
```

最近一轮长期迭代已经覆盖：首次空工作区、真实项目恢复、通用 Node.js 根项目与工作区服务、本机容器真实构建和访问、CNB/TCR/Linux/Caddy 测试与正式路径、多版本选择、网页安全保存、正式地址待办、跨项目任务恢复、最小窗口与签名 macOS Apple Silicon `.app`。

测试项数量和每次签名包时间不写入当前规范；公开验收以本页门禁和 GitHub CI 记录为准。

## 5. 性能目标与证据状态

性能目标不能因为“界面感觉更快”就写成已经达成。当前证据如下：

| 指标               | 产品目标             | 当前证据                                                                      | 结论                       |
| ------------------ | -------------------- | ----------------------------------------------------------------------------- | -------------------------- |
| 已知项目冷启动恢复 | 小于 2 秒            | 历史实机记录约 3.1–3.4 秒；后续虽继续做并行恢复和降频，但尚无固定脚本的新基准 | 未放行                     |
| 已加载项目间切换   | 立即展示本地可信页面 | 历史界面采集约 0.7–1.5 秒，且有异步串项目回归                                 | 基本可用，仍需稳定基准     |
| 项目只读识别 P90   | 小于 10 秒           | 有超时与渐进反馈实现，但没有覆盖代表性项目集合的 P90 数据                     | 未验证                     |
| 空闲后台开销       | 不干扰项目恢复       | 无活动任务降频、窗口隐藏暂停、前台单次补查已有自动化                          | 逻辑已验证，资源占用待量化 |

正式发布前应增加可重复的冷启动、项目切换和扫描基准，固定机器、项目样本、起止事件和多轮统计；不能继续把人工截图采集时间当成性能门禁。

## 6. 正式发布前仍未完成

- 使用真实目标用户完成无提示首次上线与返回用户测试，并记录是否知道“在哪、做什么、为什么”。
- 在 Windows 和 Linux 实机完成完整主线、系统密钥库、SSH、文件选择器、强制颜色和屏幕阅读器验收。
- 完成生产数据库备份、迁移和恢复策略后，才开放自动生产迁移。
- 在 CNB 提供适合桌面公共客户端的 OAuth/Device Flow 前，继续使用最小权限 PAT；不能伪装一键授权。
- CNB 密钥仓库网页保存仍是明确的安全边界，不能宣称完全自动化。
- 正式发布前重新执行全量测试、安装包签名、校验值和多平台分发检查；当前迭代不升级版本、不生成 DMG、不触发远端发布。
