# ABCDeploy 当前实现与验收基线

> 对齐日期：2026-07-18。当前处于 macOS Apple Silicon 本地快速迭代，新模型尚未构成正式发布承诺。

本文是当前迭代的验收契约。Round 12 收口后的静态原型反例集为 **61/61 通过**；数量只记录既有反例回归，不替代实际桌面测试和签名 `.app` 用户验收。

## 1. 产品模型门禁

| 范围 | 必须满足 |
| --- | --- |
| 项目入口 | 普通点击始终到发布中心；旧 `test/production` 场景迁移到发布中心 |
| 稳定导航 | 只显示“发布中心 / 在本机运行 / 版本 / 项目设置” |
| 当前任务 | 发布中心全页只有一条当前任务和一个主动作；打开后先给出完整但简短的剩余计划，再只展开当前一个问题，环境卡只展示事实 |
| 环境详情 | 测试版和正式版从发布中心进入，不复制设置流程 |
| 版本 | `versionsById` 是完整 commit、来源 Task/Attempt 和每个服务不可变 ArtifactRef 的唯一权威，服务集合与 Project 精确相等；Validation 只保存并逐项匹配来源 Attempt 的环境证据，部署失败记录不冒充版本 |
| 测试 | `main` 合并自动生成版本并部署 staging；手动部署只是辅助 |
| 正式 | 可选任一测试通过版本；按完全相同的服务产物集合发布或恢复，不重新构建、不漏服务 |
| 状态 | 在线版本与最近尝试同时保留；地址失败不抹掉部署成功 |
| 设置 | 按云端自动更新、测试/正式运行位置、分环境配置和访问地址组织；供应商与技术参数只在连接详情中可见和更换 |
| 配置中心 | 项目按环境持续绑定并复用，不只是一次性填入 |
| 待处理 | 任务入口恢复具体停点，但不劫持普通项目入口 |
| 自动化 | 主分支规则保存期望/实际状态、Provider 引用和最近错误 |
| 多项目 | 账号级 CodeConnection 可以共享；项目 SourceBinding 只能由明确的本机 remote + Provider 仓库读取证据建立，恢复/切换不能伪造或升级；任务、尝试、版本、环境与自动化按项目隔离 |
| 配置应用 | ConfigProfile 不保存反向 bindings；引用来自 EnvironmentBinding。仅保存不影响环境，显式应用成功且冻结输入仍一致后才推进 applied revision |

## 2. 数据与迁移门禁

- 历史项目、版本和部署记录无损保留。
- 历史页面场景安全映射为 `overview`，应用启动不进入不存在的旧页签。
- 原单 Profile 绑定迁移为项目 + 环境 + Profile 多绑定，重复执行幂等。
- staging 与 production 服务器分别读取、保存和验证；不能因一个已绑定就假定另一个已准备。
- 全局 Connection 更新不改变项目身份、配置绑定和历史任务。
- 环境聚合测试覆盖“旧版本在线 + 新尝试失败”“地址失败但容器成功”“历史失败不抢状态”。
- 部署任务在服务器、文件、Git 或云端副作用前创建；失败后使用同一 Task ID 和新 Attempt 继续。
- 等待用户输入的 Task 保持零 Attempt；production Task 只固定 Version ID、环境、意图和期望结果。真正执行才创建 Attempt，并从 Version 复制完整产物集合，同时冻结 SourceBinding、连接 revision、服务器能力 revision、完整配置 ID/revision 集合、服务/部署定义 revision 和地址输入。成功测试 Attempt 另存实际环境输出快照，production 每次开始或恢复前重新验证资格。
- 测试 Task 创建时固定完整 40 位 commit；后续 HEAD 变化不得改写 Task，Attempt 必须继承该 commit。
- 版本身份以 `versionsById` 为唯一权威；`version_validations` 只保存环境验证证据，不复制产物、提交或来源 Task/Attempt。旧重复字段只允许一次性迁移后清除。

## 3. 关键交互验收

### 发布中心

- 首屏不等待 Docker、CNB、Registry 或 SSH 固定超时才出现。
- 新项目第一个测试版产生前只展示“生成一个可以打开的测试版”的结果目标，不显示空正式环境卡；已有环境事实时，任务卡和环境卡应让用户在 10 秒内说清“现在怎样、下一步是什么”。
- 缺少资源时只突出一个上下文动作，并定位相应连接或配置；不得要求用户主动遍历项目设置。
- 从环境详情返回发布中心后，页面上下文和选择不混入其他项目。
- 新增或切换到第二项目后，首屏识别服务数必须读取 `currentProject.services`，不得沿用首个项目的硬编码数量或缓存文案。

### 版本与部署

- 多个测试通过版本可独立选择；当前测试版只是推荐。
- 用户把业务结果标记为“测试有问题”后，该版本不得继续出现发布动作；技术部署失败与业务“测试有问题”必须使用不同状态。
- production 当前版本不显示重复发布，历史版本使用“恢复”。
- 未完成的正式任务继续原候选；完成或取消前不能创建第二条。
- 测试和正式准备失败后，重试复用原 Task ID；完整提交和外部构建编号在触发前后可靠持久化。
- 新生产启动失败且存在旧健康版本时恢复；首次发布不显示虚假保障。
- 首次正式发布依次补齐正式运行位置、正式配置、正式地址和生产影响确认；不得因为测试版已通过就假定正式环境已经准备。
- 修改正式地址时旧 `active_address` 保持在线，只有待切换地址的路由、DNS 和 HTTPS 全部通过才提升；地址维护不得生成部署 Task/Attempt。
- production 启动、配置缺失、服务器断连与路由冲突都复用原 Task，只新增完整输入 Attempt；成功前旧正式版本与地址持续在线。
- 授权故障只影响测试环境。四类 production 故障的环境卡必须同时显示当前测试版已通过、旧正式版在线和当前任务 #42 的准确停点，不得残留测试授权任务 #24。

### 设置和授权

- CNB 权限不足时可在原任务更新 Token，验证后继续。
- 第一次远程构建前，当前任务必须解释“版本保存”用途并显示是否已验证；具体 Registry 名称、供应商和参数进入授权或连接详情，仍提供更换入口。
- 测试与正式服务器可绑定不同目标并分别替换。
- 服务器 Connection 不带环境语义；任何服务器首次绑定到任一环境前都完成“凭据 → 指纹 → 运行前提 → 共享访问服务分类 → 影响确认 → 验证 → 绑定”，测试与正式新增服务器使用同一门禁。
- 当前“干净服务器”只指 Docker Engine/Compose 已可用且 80/443 空闲；裸机安装尚未闭环。受支持的边缘能力是服务器级共享 Docker Caddy 容器与项目 + 环境独立 `sites` 片段，每次变更必须 backup → `caddy validate` → reload，失败 rollback。
- 已有兼容 Docker Caddy 可以复用；systemd Caddy、其他代理、端口冲突、旧服务或归属不明必须阻断。只有另一已知 ABCDeploy 项目占用时提供明确“转移地址”。production 选择全新服务器执行同一检查仍是待实现门禁，不得写成完成能力。
- 历史 Connection 只能恢复为“已配置/待验证”；只有真实 CNB、Registry 或 SSH 校验成功才显示可用。
- 删除再添加项目不是任何凭据恢复的必要步骤。

### 配置

- `development/staging/production` 分别列出绑定、缺失数和保存目标。
- 一个环境可绑定多个 Profile；敏感值不进入 SQLite 或普通日志。
- 配置完整性按精确必填 Key 判断；任意名称或条目数量不能冒充已满足。已绑定 Profile 的 Key 只读，Profile 与环境应用都有 revision。
- “仅保存”不能改变运行环境；显式应用才创建目标项目 + 环境的配置 Task，production 先确认。Attempt 冻结完整 Profile ID/revision 集合，成功且完成时绑定仍一致后才用整份快照推进 applied revision。无 revision 变化不创建任务；解除绑定只影响指定项目 + 环境，不得删除其他引用。
- 无候选配置时可在当前字段新增；手填值可保存为常用配置并自动绑定当前环境，失败不覆盖当前内容或旧绑定。
- `.env.example` 注释、顺序、未知字段和内部字段在往返保存后不丢失。
- 跨环境不会静默复制值；无用户字段时不出现空保存流程。

### 本机运行

- 服务、依赖和必要配置按规范顺序显示并支持批量/逐项动作。
- 状态轮询不自动滚动；项目可打开状态只在本次完整启动成功后点亮。
- 热加载只在所有相关服务可靠支持时承诺。

## 4. 性能与稳定性门禁

| 指标 | 目标 |
| --- | --- |
| 已知项目首屏 | 先显示本地可信信息，目标小于 2 秒 |
| 已加载项目切换 | 即时出现缓存/骨架，不出现约 5 秒整页卡死 |
| 后台检查 | 局部更新、可取消或超时，不阻塞导航 |
| 轮询 | 前后台与活动任务自适应，无任务时降频 |
| 滚动与焦点 | 异步更新不抢滚动位置和焦点 |
| 异步隔离 | 请求结果绑定原项目，不能覆盖后来打开的项目 |

性能结论必须来自固定机器、固定项目样本和多轮可重复测量，不能只凭视觉感受写成“已完成”。

## 5. 自动化与本地包

按改动范围执行：

```bash
pnpm --filter @abcdeploy/desktop test
pnpm --filter @abcdeploy/desktop build
cargo test --workspace
cargo clippy --workspace --all-targets -- -D warnings
pnpm tauri:build:app
```

桌面功能通过后生成使用稳定 Apple Development 身份签名的 macOS Apple Silicon `.app`，供立即验收。不升级版本、不生成 DMG、不打标签、不触发远端多平台发布。

## 6. 通用夹具行为测试

不得把真实项目名、真实凭据或某个业务的字段写进稳定验收契约。使用不包含敏感信息的通用夹具和故障注入覆盖以下状态：

| 夹具 | 初始事实 | 重点验证 |
| --- | --- | --- |
| `new-project` | 只有本机识别结果，没有远程版本和环境 | 首次测试目标、短计划、逐题披露 |
| `capability-reconciliation` | 分别模拟全部复用、只缺一项和多个有效候选 | 零问直达、只问真实缺项、只做一次选择 |
| `existing-deployment` | 项目目录含既有部署定义，尚未作接入选择 | 选择前零远程同步，继续管理与重新设置分流 |
| `test-ready` | 测试版可打开，尚未给业务结论 | 打开后确认、标记“测试有问题” |
| `production-ready` | 正式版在线，另有多个测试通过候选 | 版本选择、发布、历史恢复和失败保留旧版 |
| `access-pending` | 正式工作负载健康，DNS 或 HTTPS 未完成 | 部署事实与访问状态分离，只继续地址任务 |
| `server-capability` | 分别模拟干净服务器、兼容 Docker Caddy、systemd/其他代理和端口/归属冲突 | 一次性准备、共享 Caddy、validate/reload/rollback 和阻断 |
| `task-attempt-history` | 同一任务先失败、替换连接后再次执行 | Task ID 不变、Attempt 递增、完成与重载恢复 |
| `artifact-set` | 多服务版本中缺失、替换或重复一个 digest | Version 不完整或资格失效；正式端必须精确复用完整集合 |
| `config-key-revision` | 同数量不同 Key、仅保存、精确解绑和 revision 变化 | 缺失 Key 不被冒充，运行环境不被静默更新，其他引用不丢失 |
| `active-desired-address` | 旧地址在线，新地址分别处于路由、DNS、HTTPS 检查阶段 | 旧地址持续生效，完成后原子提升，新地址维护不制造部署任务 |
| `multi-project-workspace` | 两个项目分别有任务、版本、环境和页面历史 | 打开/切换不串线，普通重载回发布中心，合法 deep-link 才恢复指定项目页 |
| `source-binding` | 账号连接可用，但当前项目仓库绑定为空、错误、缺证据或 revision 过期 | 恢复/切换只降级；显式只读核对成功才建立证据。失败不创建 Task/Attempt、不改规则、不立即重输 Token |
| `version-authority` | Version、来源 Task/Attempt、Validation 或 production 页面选择分别被修改 | 任一不一致失去资格；正式完成只提升 Task 固定的 Version |
| `automation-rule-sync` | desired/observed/Provider/sync 分别模拟一致、暂停和失败 | 暂停、恢复和修复复用规则，不创建部署 Task 或版本 |
| `config-application` | Profile revision 高于 applied revision、production 待确认、执行中改绑和无变化 | 显式配置 Task、成功推进、改绑阻断、精确解绑、无变化零任务 |

每个夹具至少覆盖：普通进入立即显示发布中心；连接和配置能在当前任务中处理并在设置中长期更换；退出、切换项目和重启后恢复可信任务卡但不被旧浏览页劫持。真实生产写入、地址接管和脏工作区修改只在获得明确授权后执行。

### 必测状态

1. 新项目首屏只承诺“生成一个可以打开的测试版”。全新账号最多显示代码更新授权、版本保存位置、测试运行位置和必要配置四项，但按能力事实对账：全部复用零问、部分复用只问缺项、多个候选只问一次选择；同一时刻只展开当前一个问题，最后一项保存后直接生成。
2. 检测到已有部署时先显示接入选择；停留 30 秒、切换前后台或重启都不得查询远程历史、恢复旧绑定或出现旧任务。
3. 选择“继续管理已有部署”后只同步一次，重启不重复询问；历史部署事实与本机尚未恢复的运行位置连接分开呈现。
   接管依次核对代码 Connection、Registry Connection、服务器完整身份/能力并生成不可变 DiscoverySnapshot；最终确认只能导入快照已证明的版本、ArtifactRef、配置引用、地址与健康事实。确认前不得创建任何任务；确认后可以创建 completed `kind=import` 审计 Task/Attempt 证明 Version 来源，但不得创建 deployment-kind 记录、执行部署或用本机凭据存在状态伪造线上健康。
4. 选择“重新设置部署”时不得调用代码平台、版本存储、SSH 或部署接口，不得写项目文件；重启后仍从首次上线开始，截断时间之前的构建不得重新导入。
5. 测试版可打开但未确认时，唯一主动作是打开并确认；用户选择“还有问题”后状态统一为“测试有问题”，该版本不再显示正式发布动作。
6. 首次正式发布必须在同一任务中补齐正式运行位置、正式配置、正式访问地址和生产影响确认；测试环境的资源不能静默成为正式绑定。
7. 多个测试通过版本可独立选择；当前正式版不重复发布，历史版本显示“恢复”，正式环境始终使用测试通过的同一完整服务产物集合。
   多服务项目必须逐项核对不可变 Registry 引用/digest，不能只校验一个代表性镜像。
8. 正式版已在线时，新任务失败必须同时展示“当前在线版本”和“最近尝试”，不能把环境误报离线。
9. 授权、版本保存、运行位置或配置失效时，在原任务更新对应资源并继续同一 Task；不得要求删除项目或重填无关资源。
10. DNS / HTTPS 待处理时，明确说明服务已部署到哪一步；只读刷新不得修改 Caddy、重新构建或重新部署工作负载。
11. Docker/Compose 已可用且 80/443 无占用的干净服务器，只在首次确认后初始化共享运行与访问能力；缺少运行前提时不得误报成功；第二个项目复用时不得重复安装 Caddy。
    首次“绑定”发生在凭据、指纹、前提、访问能力分类和影响确认之后；生产新增服务器不得跳过初始化而直接保存绑定。
12. 兼容 Docker Caddy 只追加当前项目 + 环境路由片段，并执行 backup、`caddy validate`、reload 与失败 rollback；systemd Caddy、其他代理、多个代理、端口冲突或未知归属必须阻断且不改主配置。
13. 当前项目路由可以幂等修复；其他 ABCDeploy 项目路由只能走“转移地址”；旧服务、未知来源、非 Caddy 或配置入口不明时不提供自动接管。任何校验或热加载失败都恢复原路由。
14. 在等待版本保存、远程执行、授权恢复和地址处理等停点关闭重开，发布中心仍恢复同一任务、环境、版本和唯一下一步；普通重启不自动弹出 `Sheet`。
15. 在多个项目间快速切换时，异步结果不能写入后来打开的项目，也不能抢滚动位置、焦点或导航。
16. 必填配置使用精确 Key 校验；随意增加同数量的其他 Key 仍保持阻塞。已绑定 Key 不可直接改名，“仅保存”不推进环境 revision，解除当前绑定不影响其他项目和环境。
17. 修改正式地址后旧地址仍是 `active_address`；断网、部分解析或 HTTPS 未完成时只保留 `desired_address`，全部通过后才切换，Task/Attempt 数量不因此增加。
18. 取消地址编辑后，复核和执行快照仍使用原 EnvironmentBinding；未保存 draft 不得进入任何发布门禁。
19. Code Connection 授权轮换必须递增 revision，新旧 Attempt 各自保留当时的 `connection_id@revision`；每个项目 SourceBinding 还必须精确匹配 Project、仓库范围与连接修订。
20. 更换默认 Registry 后，历史 Version 的 ArtifactRef 仍指向原 repository、digest 与可读取的 Connection revision；不得使用当前连接冒充历史来源。
21. 单独地址维护完成后回到正常发布中心，并明确它不是发布任务；DNS 目标根据 IPv4、IPv6 或主机名分别生成 A、AAAA 或 CNAME。
22. 添加第二个真实项目后，账号连接可以共享，但仓库 SourceBinding、Task、Attempt、Version、Environment 和 AutomationRule 独立，首屏与本机服务数来自当前项目；普通重载回当前项目发布中心，只有项目与页面都合法的 deep-link 才恢复指定工作区。
23. 测试 Task 创建时冻结完整 40 位 commit；当前 HEAD 后续变化后继续任务，Attempt 仍使用冻结 commit；无法取得完整 commit 时拒绝创建 Task。
24. AutomationRule 的 desired、observed、Provider 和 sync 状态可独立核对；暂停、恢复、失败与修复都不创建 DeploymentTask、Attempt 或 Version。
25. Profile 不保存反向 bindings；使用关系只来自 EnvironmentBinding。仅保存只推进配置库 revision；显式应用到唯一项目环境才创建配置 Task，Attempt 冻结完整 ID/revision 集合，成功且绑定未变化才推进 applied revision，无变化不创建任务，精确解绑不影响其他引用。
26. production 启动失败、配置缺失、服务器断连和路由冲突都保留 Task ID，只增加冻结完整输入的 Attempt；配置和连接修复复用既有维护入口。授权故障只影响测试环境；四类 production 故障显示测试通过、旧正式版本与地址在线及任务 #42 准确停点，不得残留任务 #24。
27. Version、来源 Task、来源 Attempt 与 Validation 必须相互一致；production Task 只固定 Version ID，执行和完成不能读取后来变化的页面选择。
28. 旧 `adaptable` 服务器状态恢复为 blocked；来源不明路由不显示接管动作；旧快照中的 Validation 产物来源和 ConfigProfile bindings 必须清除。
29. Version ArtifactRef 的 service ID 集合必须与 Project 精确相等；缺少、多出或重复服务均移出权威版本并列为未解决记录。
30. Validation 的配置、服务、部署、服务器、地址与健康六项事实必须逐项匹配来源成功 Attempt 的环境输出及冻结输入；双向篡改输出和 Validation 也不能绕过冻结输入。
31. 恢复、添加、切换项目或账号 revision 变化不得自动创建、纠正或升级 SourceBinding；只有明确的只读仓库核对结果可以生成带时间、连接 revision 和证据 ID 的绑定。
32. 当前仓库读取失败时，首次生成和已有部署接入不得创建 Task/Attempt；自动化暂停、恢复、修复不得改变规则；账号仍有效时默认不显示 Token，只有用户主动更换账号时才进入授权表单，授权成功后必须继续原始生成、接入或规则动作而不是让用户重来。
33. 代码账号连接允许先保存，但 staging Task 必须在项目 SourceBinding 只读核对成功后、任何 Registry/服务器/配置副作用前创建；恢复时从全部开放任务推导占用，多个远程变更任务必须阻断。
34. Registry endpoint 不得包含用户名、密码、query 或 fragment；ArtifactRef、Attempt 和服务器验证必须按冻结 connection/server revision 解析不可变 evidence，而不是读取后来变化的默认连接。
35. 配置应用必须绑定真实在线 Version、精确服务器证据、有效地址和当前项目配置；Attempt 冻结完整输入，完成时事实漂移不得推进 applied revision，自动生成秘密不得跨项目复用。

## 7. 正式发布前仍需完成

- 真实目标用户无提示完成首次接入、凭据替换、测试确认和版本发布选择。
- Windows 与 Linux 实机完成密钥库、SSH、文件选择器、强制颜色和屏幕阅读器验收。
- 生产数据库备份、迁移和恢复形成独立闭环后，才开放自动生产迁移。
- CNB 网页安全保存仍应如实呈现外部边界，不能伪装成完全自动化。
- 裸机 Docker/Compose 自动安装、systemd/其他代理的安全接入，以及 production 全新服务器执行同一初始化检查仍未闭环；当前统一按阻断处理，不得用原型状态冒充运行时完成。
- 当前桌面运行时仍把主 Caddyfile 中的同地址路由标为可确认接管，且初始化前只识别 Docker 容器端口占用；在补齐路由归属证据和宿主机进程检测前，这两项不满足当前 blocked 门禁。
- 正式发布时重新执行全量测试、签名、校验值和多平台安装包检查。
