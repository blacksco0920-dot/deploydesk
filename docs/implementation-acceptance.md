# ABCDeploy 当前实现与验收基线

> 对齐日期：2026-07-19。当前处于 macOS Apple Silicon 本地快速迭代。本文件是工作流画布版本的验收门槛，不表示每一项已经通过；真实完成度和明确缺口以[底层技术能力](technical-capabilities.md)为准。旧“发布中心 / 测试环境 / 正式环境”原型反例数量不再作为当前产品完成证据。

截至本次对齐，桌面运行时与自动化测试已覆盖：真实任务 `status/currentStage` 到节点与连线的状态投影；配置中心展示真实 `ConnectionResource` 和 `ConfigProfile`；新线路自动打开最近的未就绪节点；节点显示 Connection 名称和文字状态；`Escape` 关闭 Inspector 并恢复焦点；运行记录图标 Tooltip；线路删除；运行记录中的真实任务、服务器、路由、commit、构建与镜像事实；以及 Inspector 开关不再触发 `fitView`。尚未闭环的是 CNB/TCR 多连接与 Connection revision 后端，以及项目级“本机运行”次级入口。

## 1. 当前产品模型

项目首页只有一张部署工作流画布：

```text
代码来源 ──> 版本构建 ──> 版本存储 ──> 部署运行
```

- 首版恰好有四种 NodeType；本地项目、CNB、腾讯云 TCR 和 Linux 服务器分别是对应的 Provider，不是 NodeType。
- Connection 是可复用授权或服务器身份；NodeInstance 是某条线路对 NodeType 的一次使用。验收页面必须能区分 NodeType、Provider、Connection、配置状态和运行状态。
- 授权不是第五个节点；授权失效只修复或更换 Connection，不删除 NodeInstance、线路或项目。
- 系统使用“小白上线模板”自动生成和排布完整线路，首次用户不需要从空画布手工连线。
- 测试、演示、生产等目标不形成全局环境向导；需要另一个服务器目标时新增一条独立线路。
- NodeType 可以在不同线路复用；Connection 可以跨项目引用；NodeInstance 的配置和运行状态必须线路隔离。
- 首版每条线路固定且恰好包含四个 NodeInstance；新增目标通过新增完整线路完成，不开放手工增删节点、连线或服务器分支。
- 线路状态、节点状态、最近运行和当前在线版本是不同事实，不能互相冒充。

## 2. 画布验收

1. 主体使用 FlowGram/Coze Studio 行为基线的网格/点阵工作流画布，不再使用普通四列业务卡片冒充流程。
2. 节点之间有可见连接线；就绪、处理中、阻塞状态同时使用颜色、图标和文字。
3. 工作区为“线路与节点资源栏｜画布｜右侧 Inspector”；选中节点有明确焦点，打开或关闭 Inspector 不丢画布位置。
4. 顶部稳定展示项目/线路上下文、保存状态、图标化运行记录入口和唯一线路主动作；底部工具栏使用 FlowGram 标准缩放与适应视图。
5. 默认线路自动排布；首版画布固定四种 NodeType 及其三条顺序连线，不提供添加、删除、改线、条件、循环或审批等自由编排能力。
6. 宽屏完整展示四节点；窄窗口允许画布横向浏览，不压缩成无法阅读的卡片。
7. 远程运行时当前节点和已完成连接线可见；关闭客户端后重新进入恢复真实停点。
8. 画布直接使用 FlowGram，画布通用工具使用 Semi Design；资源栏、Inspector 和运行状态沿用 Coze Studio 的结构与交互基线。不得把“参考 Coze”写成已导入 Coze Studio 完整业务前端，也不得同屏混用 ABCDeploy 旧式大卡片、旧按钮或旧 Sheet。
9. 进入工作流编辑器后隐藏应用级侧栏，只保留一块线路与节点资源栏；顶部提供返回应用首页的入口，不允许两块完整左栏并存。

## 3. 节点 Inspector 验收

每次打开节点，Inspector 只能处于以下一种模式：

| 模式      | 页面内容                                              | 禁止内容                     |
| --------- | ----------------------------------------------------- | ---------------------------- |
| 查看      | 当前生效摘要、最近验证、`更换…`、必要的节点内维护入口 | 输入框、新连接表单、候选列表 |
| 选择      | 配置中心候选、当前选中标识、`添加新的`、`返回`        | 当前绿色摘要、新连接表单     |
| 新增/编辑 | 一套草稿表单、保存到配置中心说明、验证并保存、返回    | 候选列表、当前生效摘要       |
| 修复      | 问题结论、仍然有效的事实、唯一恢复动作、折叠技术详情  | 无关设置、多个同级主动作     |

通用要求：

- 取消、关闭或 `Escape` 不修改当前生效绑定。
- 新草稿验证成功后才替换当前线路引用。
- 创建代码平台、版本仓库或服务器连接时明确说明“保存到配置中心，其他项目可复用”。
- 查看态按“NodeType → Provider → Connection → 状态”的固定层级展示，Token、密码、私钥不回显。
- 选择态只列出当前 NodeType 可使用的 Connection；新增态先选择 Provider，再展示该 Provider 所需的最小表单。
- 工作流资源栏和节点 Inspector 不提供“管理配置中心”入口；这里只选择、替换或新建当前引用，全局管理必须返回应用首页进入配置中心。
- 版本历史和部署运行记录不出现在节点配置中。

## 4. 四类节点验收

### 代码来源

- 首版 Provider 为本地项目；查看模式展示项目名、识别服务数和服务摘要。
- 本地快照失效时只要求“重新读取项目并上线”，不清空其他三个节点。

### 版本构建

- 首版 Provider 为 CNB；查看模式展示当前 Connection 和项目代码仓库摘要。
- 选择模式可复用配置中心连接；新增模式只要求仓库与授权信息。
- 权限不足时允许在原任务更换授权并继续，不要求移除项目。

### 版本存储

- 首版 Provider 为腾讯云 TCR；查看模式展示 Provider、Connection、仓库地址、命名空间和验证状态。
- 选择已有连接与新增连接互斥展示。
- 不展示版本列表；版本列表进入运行记录。

### 部署运行

- 首版 Provider 为 Linux 服务器；查看模式展示 Connection、访问地址数和运行配置状态。
- 更换服务器、维护访问地址、维护运行配置是三个明确入口，彼此不同时展开。
- 新服务器流程为地址/用户/端口 → 身份指纹 → 一次性密码 → 后台生成并安装专用密钥。
- 密码只用于首次连接，不保存、不要求用户自行生成私钥。

## 5. 上线与恢复验收

1. 四类基础节点全部就绪后只显示“开始上线”；首次成功后显示“更新上线”。
2. 首次上线只确认一次本地项目、目标服务器和访问地址。
3. 每次点击使用当时的本地项目内容，生成完整服务集合的不可变版本。
4. 构建成功、版本入库、服务器服务健康、公网地址可用分别保存证据。
5. 地址或 HTTPS 失败显示“应用已部署，访问地址待处理”，不重新构建。
6. 重试保留同一任务并增加执行记录；配置修复不自动触发外部变更。
7. 服务器初始化和 Caddy 变更遵循备份、校验、热加载、失败恢复；未知代理或路由归属阻断自动接管。

### 线路隔离、运行配置与跨服务门禁

1. 同一项目两条线路以及两个不同项目共用服务器时，服务别名、Docker 网络、Caddy 目标和运行配置不得串线；线路内服务使用 `<project>-<path_key>-<service>` 唯一别名。
2. H5 等 Static/Web 服务访问 API 时使用当前线路的 `API_HOST` 与 API 实际 `container_port`，不能写死裸 `api` 或 3000。
3. 线路级聚合配置部署时按服务声明过滤为 `.runtime.<service>.env`；API 专属变量不能出现在 H5 或 OCR 容器，`runtime_env` 为空的服务不挂载 env_file。
4. 旧线路新增必要变量时补 Key 但不覆盖已有值；内部应用密钥只生成一次并跨更新复用，CNB、TCR、SSH 和 AI Key 等外部凭据绝不生成。
5. 项目存在精确 `ocr` 服务时，`PP_OCRV6_TINY_URL` 不要求用户填写，且只注入声明该变量的服务；目标必须使用当前线路唯一 OCR 别名和 OCR 实际端口。
6. H5 首页和 API 单独健康不足以判定上线成功。H5 同源 `/api/health` 失败时整条线路必须阻断，并在技术详情返回不含秘密的 `AD-CTR-102`；公网验收还要覆盖真实认证或其他代表性业务链路。

## 6. 性能、可访问性和测试

- 已知项目先显示本地可信画布，目标小于 2 秒；后台 Provider 检查局部更新。
- 异步刷新不重置滚动位置、不抢焦点、不串项目。
- 键盘可选择线路、节点和抽屉动作；所有图标按钮有对象化可访问名称。
- 状态不能只依赖颜色；`640×480` 下仍可读、可横向浏览、按钮高度不低于 40px。
- 使用无真实凭据的通用夹具覆盖：首次设置、全部就绪、节点失效、配置替换失败、多线路、运行中恢复、历史版本重上线和地址待处理。

按改动范围执行：

```bash
pnpm --filter @abcdeploy/desktop test
pnpm --filter @abcdeploy/desktop build
cargo test --workspace
cargo clippy --workspace --all-targets -- -D warnings
pnpm tauri:build:app
```

桌面功能通过后生成稳定 Apple Development 身份签名的 macOS Apple Silicon `.app`。不升级版本、不生成 DMG、不打标签、不触发远端多平台发布。

## 7. FinAgentCRM 试点完成定义

FinAgentCRM 只作为真实通用项目试点，不能把它的业务字段硬编码进产品。完成必须同时有：

1. 从清理后的本地项目重新生成部署定义；
2. 本地真实安装、构建和容器构建通过；
3. CNB 构建任务与完整提交可追溯；
4. 所有服务镜像以不可变摘要进入 TCR；
5. 目标服务器完整服务集合健康，Caddy 只追加 ABCDeploy 自有路由片段；
6. 公网访问和至少一条跨服务业务路径通过；
7. 画布能恢复该线路的节点绑定、当前版本、运行记录和真实停点；
8. 试点结果与已知限制更新到当前文档，不泄露任何 Token、密码、SSH 私钥或业务密钥。

2026-07-19 的完整实机证据见[FinAgentCRM 工作流试点验收](pilot-validation-finagentcrm-2026-07-19.md)。该次验收复用了客户端可见的“上线”线路，三个服务以不可变摘要运行于目标服务器，公网首页、API 健康检查和真实登录均通过。

## 8. FinAgent 跨服务试点证据

FinAgent 用于验证本轮新增的线路唯一寻址、实际端口、服务级运行配置、托管内部密钥、托管 OCR 地址与 H5 同源 API 门禁，不能把其业务字段硬编码进 ABCDeploy。

2026-07-19 的签名 `.app` 由同一签名二进制内置的 QA 验证入口复用客户端可见线路、已验证 Connection、DeploymentTask 和正式部署引擎完成真实上线；这证明正式运行时与数据模型的端到端能力，不等同于用户从工作流界面逐项点击的完整 UI E2E。该次验证满足以下结果：

1. API、H5、OCR 三个镜像均以不可变摘要部署；
2. API 健康接口返回 200，数据库为 `connected`，Redis 为 `ok`；
3. H5 首页返回 200，H5 同源 `/api/health` 返回 200 且依赖健康；
4. OCR `/health` 返回 200，`/ready` 的 `status` 为 `ok`；
5. 随机无真实个人信息账号完成注册 201、登录 201、当前用户 200；
6. H5 同源 OCR 请求返回 201、状态为 `recognized`、识别文本非空且包含 3 个 blocks；
7. 上线未切换或推进用户分支，也未改动用户暂存区；系统只创建不挂到用户分支的独立部署快照提交，`apply_plan` 按设计更新部署生成物，因此不要求工作树状态哈希保持不变；
8. 证据不包含账号、密码、Token、SSH 私钥或完整运行配置。
9. 最终公网只读复核对 H5 `/`、H5 同源 `/api/health`、独立 API `/api/health`、OCR `/health` 和 OCR `/ready` 各请求 20 次，共 100 次，结果为 100/100 HTTP 200。

完整标识、不可变摘要和无敏感信息的逐项证据见[FinAgent 工作流试点验收](pilot-validation-finagent-2026-07-19.md)。
