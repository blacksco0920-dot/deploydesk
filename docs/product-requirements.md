# ABCDeploy 产品需求

## 1. 产品定位

ABCDeploy 帮助使用 AI 编程工具完成项目、但缺少部署经验的用户，把项目可靠地运行在本机、测试环境和正式环境。用户不需要先学习 CI/CD；客户端负责解释现状、收集必要授权与配置，并把自动化结果变成可恢复的任务。

首版试点：CNB、腾讯云 TCR、Linux 服务器和 Caddy。架构必须允许以后替换代码平台、OCI Registry、服务器厂商和 DNS Provider。

暂不服务 Kubernetes、多集群、多区域容灾、复杂企业审批、Service Mesh 和自动生产数据库迁移。

## 2. 用户任务

> 当我完成一个项目时，我希望先得到一个别人可以打开的测试地址。系统只问无法替我决定的信息；测试通过后，我可以随时选择这个或以前验证过的版本发布给正式用户。失败时，我知道已经完成了什么、什么仍然在线，以及唯一下一步。

返回用户还需要：

> 再次打开应用或开始第二个项目时，我能看到真实在线状态，复用已有连接和配置，也能直接更换失效凭据，不从头重来。

## 3. 产品目标

- 普通项目入口在 1 个首屏内回答“测试版、正式版、下一步”。
- 首次测试版以“得到可打开地址”为目标，不以“完成部署资源设置”为目标。
- 首次成功最多包含 5 类需要用户判断或提供的信息；仓库名、Registry 路径、临时地址、容器和自动化规则不让用户决定。
- 首次测试部署不要求用户打开终端或手写 YAML。
- 第二个项目可以复用已验证的代码平台、Registry 和服务器连接。
- 应用重启、网络中断或外部网页操作后能够继续原任务。
- 每个失败都有一个面向结果的主修复动作。

## 4. P0 功能要求

### FR-01 项目与入口

- 添加和切换项目只在项目栏完成，不进入项目内工作流。
- 应用维护真实项目列表；每个项目拥有独立工作区，分别保存任务、尝试、环境、版本、页面上下文与自动化规则。添加或打开第二个项目不得继承当前项目的部署事实；跨项目复用只发生在明确的全局 Connection、ServerCapability 和 ConfigProfile。代码平台账号授权可以复用，但每个项目必须拥有经验证的 `SourceBinding`，精确匹配该 Project、仓库范围、账号连接及其 revision，并保存真实只读核对的时间和证据 ID；错误、缺证据或过期绑定不得进入执行。
- 添加、切换、重载或恢复项目只校验已有 SourceBinding 证据，不能因为账号连接可用就自动生成、纠正或升级绑定。用户发起首次生成、已有部署接入或自动化写操作时才用现有账号执行一次只读仓库核对；成功后继续原动作，失败时不创建 Task/Attempt、不改变规则，也不立即要求重输 Token。只有账号本身失效或用户主动选择“更换账号授权”时才显示 Token 表单。
- 添加项目后先只读识别服务、入口、运行方式、`.env.example` 和可复用资源；高置信结果进入“查看识别结果”，只有会改变用户结果的低置信项才要求确认。
- 添加项目只做本机只读识别；检测到已有部署定义时，必须先显示“继续管理已有部署 / 重新设置部署”，用户选择前不得同步远程历史、恢复旧连接或生成待处理任务。
- “继续管理已有部署”按“代码连接 → 版本保存连接 → 服务器身份与能力 → 汇总确认”补齐本机证据，用户最终确认后才只读导入已有版本和环境事实；该过程不创建部署 Task、不执行部署，也不能用本机是否有凭据伪造线上状态。“重新设置部署”只清除本机保存的该项目部署关联，保留项目文件、全局连接、凭据和所有线上资源，并从首次上线首步开始。
- 接入决策和重新设置时间必须持久化；重启不得重复询问或把重新设置前的远程历史重新导入。
- 普通点击项目与普通应用重载始终打开当前项目“发布中心”，不得由上次浏览页、旧场景、最近失败或后台任务劫持。只有项目 ID 与项目页面都合法的显式 deep-link 才进入指定项目页面；非法或不完整链接回退到当前项目发布中心。
- 项目内稳定入口为“发布中心 / 在本机运行 / 版本 / 项目设置”。
- 只有从一条明确待办进入时，才在发布中心原地打开对应任务面板并恢复准确停点；关闭后不污染项目导航和下次普通入口。
- 重复添加同一目录复用项目身份；目录移动提供恢复，不要求删除重建。

### FR-02 发布中心

- 全页只展示一条“当前要做的事”和一个主动作；不得让测试、正式两个环境同时争抢用户注意力。
- 新项目首屏标题必须是“先生成一个可以打开的测试版”等用户结果，不得先出现代码平台、Registry、服务器或自动部署设置。
- 用户开始首次任务后，用一句简短计划说明所有尚需配合的事项和数量，但同一时间只展开当前一个问题，既避免信息墙，也避免移动终点感。
- 第一个测试版产生前不显示空的正式环境卡；正式环境在用户拥有可发布版本或主动开始正式设置后出现。
- 测试环境与正式环境卡展示在线版本、访问状态和最近一次动作，只提供“查看详情”等次级入口。
- “当前在线版本”与“最近部署尝试”独立展示；新的失败、排队或取消不能把旧的健康版本改成离线。
- 环境卡片可以在右侧任务面板中打开部署、验证、恢复和故障详情；发布中心保持挂载，这些详情不成为顶层导航。
- 首屏优先显示用户结果；构建号、提交、镜像摘要和原始日志进入技术详情。

### FR-03 不可变版本

- `main` 合并后自动产生由完整提交和完整服务产物集合唯一标识的版本；Version 同时保存项目、来源 Task/Attempt 和集合中的每个不可变 ArtifactRef（repository、digest、Registry Connection revision），是这些身份的唯一权威对象。ArtifactRef 的 service ID 集合必须与 Project 当前识别出的 service ID 集合精确相等；缺少、多出或重复任一服务时都不得产生完整版本。更换默认 Registry 不得改写历史 ArtifactRef。
- 版本记录区分：生成中、测试部署中、等待业务确认、测试通过、测试有问题、已在正式环境、已废弃；技术部署失败属于任务尝试，不冒充业务结论或新版本状态。
- 只有测试环境真实部署成功且用户确认业务结果后，版本才获得“可发布”资格。
- 成功测试 Attempt 必须保存本次实际输出的不可变环境快照。测试资格必须把 Validation 的配置修订、服务与部署定义修订、服务器、地址、健康和业务结论与来源 Attempt 的输出快照及冻结输入逐项核对；`VersionValidation` 只保存验证证据，不复制 Version 的提交、产物集合或来源任务。同一产物换用新配置后需要新的环境验证记录；正式 Task 每次开始或恢复 Attempt 前重新验证资格，不能只依赖历史“可发布”状态。
- 失败任务属于部署记录，不伪装成可发布版本，也不覆盖历史健康版本。
- 版本页允许查看生成时间、说明、环境占用和发布资格；技术标识默认折叠。

### FR-04 测试环境自动部署

- 默认主线是 `main` 合并后由远程流水线自动构建并部署测试环境，客户端可以关闭。
- 第一次测试默认使用系统生成的临时地址；正式服务器、正式配置和正式域名不得阻塞第一次测试成功。
- 用户补齐首次任务最后一项信息后直接开始生成，不再出现“准备完成”后第二次点击“生成测试版”的重复确认。
- 执行进度使用“读取项目 / 生成版本 / 启动服务 / 检查地址”等结果语言；技术阶段默认折叠。
- 客户端负责展示进度、打开测试地址、确认业务结果和恢复失败任务。
- 手动部署只用于首次接入、重新执行或异常恢复，不作为日常必经步骤。
- 新版本生成或测试部署失败不影响正式环境，也不取消历史版本的可发布资格。

### FR-05 正式发布与恢复

- 用户可从任一“测试通过”的不可变版本中选择一个发布到正式环境。
- 当前测试版只作为默认推荐，不限制用户选择历史候选。
- 正式发布使用候选记录中的同一完整服务产物集合，不重新构建、不漏掉或替换其中某个服务。
- 生产变更必须显式确认目标环境、目标版本、地址和回滚条件。
- 已有健康正式版时，新版本启动失败自动恢复旧版本；首次发布不得虚构回滚保障。
- 服务已启动但 DNS/HTTPS 未完成时，状态是“正式版已部署，还差地址”，只继续地址检查，不重新部署。地址编辑先保存为独立草稿，取消不得污染发布复核；确认保存后环境才写待切换地址。新路由、DNS 和 HTTPS 全部验证前，当前地址继续在线，验证完成后才原子提升待切换地址。解析记录按实际运行位置生成 A、AAAA 或 CNAME。

### FR-06 项目设置、连接与服务器能力

- 应用级提供“连接与资源”，统一维护账号级代码平台连接、Registry 和服务器能力，展示最近验证结果、Connection revision、影响项目数和更换入口；项目设置只展示当前项目的仓库 SourceBinding 和环境绑定。代码授权轮换推进同一账号连接 revision，并把受影响项目的旧仓库证据标为过期，在各项目下一次相关动作时只读重验；轮换本身不得批量伪造新证据。Registry 端点变化建立新活动连接，并保留历史 ArtifactRef 所需的只读来源。
- 项目设置按用户对象集中管理：云端自动更新、测试运行位置、正式运行位置、分环境配置和访问地址；Provider 与技术参数进入详情。
- Registry 在第一次构建并推送远程版本前必须已验证；国内默认推荐 TCR，不能因为后台曾有凭据就隐藏其来源和状态。
- 账号、Token、Registry 凭据和服务器连接是可复用资源；项目保存引用，不复制明文。服务器 Connection 本身没有 `staging/production` 语义，只有项目的环境绑定才决定它用于哪个环境。
- 每项连接都提供“更换/重新授权/解除绑定”，失败授权不能通过删除项目来恢复。
- 测试与正式服务器独立绑定；允许相同，但不能自动把其中一个当成另一个。
- 多个已验证候选时要求选择；只有一个时可以推荐复用，但仍能在设置中看见和更换。
- 任何服务器第一次绑定到任一环境都必须按“验证凭据 → 确认并固定主机指纹 → 检查运行前提 → 分类共享访问服务 → 展示并确认影响 → 标记连接已验证 → 建立环境绑定”执行；不得先绑定再补能力。服务器的 Docker、Compose、容量、主机身份和统一访问服务属于连接的一次性准备；其他项目复用时只做轻量复核，不重复成为部署向导步骤。当前“干净且受支持”指 Docker Engine/Compose 已可用且 80/443 无占用；裸机自动安装 Docker/Compose 尚未闭环，必须单独阻断并说明缺口。
- 首版统一访问服务是服务器级共享 Docker Caddy 容器；项目 + 环境各自拥有独立 `sites` 片段。变更必须先备份，执行 `caddy validate`，再 reload，任一步失败都 rollback 到原片段和配置。
- 已有且能确认配置入口、挂载目录与 reload 方式的兼容 Docker Caddy 可以复用；systemd Caddy、其他反向代理、多个端口占用者或无法确认归属时阻断，只保留诊断并要求更换服务器或由维护人员先整理，不自动覆盖。正式版选择全新服务器时也必须执行同一初始化检查；该正式运行时闭环当前属于待实现门禁，不得标为已经完成。
- 当前项目拥有的路由可以幂等修复；另一已知 ABCDeploy 项目拥有的路由只能走明确“转移地址”。旧服务、第三方或未知归属路由不提供一次确认式接管，只能更换地址或先由维护人员整理。

### FR-07 配置中心与三环境

- 配置中心只维护“配置说明、配置名称、配置值”，作为应用级复用资源。
- 项目按 `development`、`staging`、`production` 分别绑定一个或多个配置项；绑定是持续关系，不只是一次性复制。
- `.env.example` 的字段名、注释和顺序是用户配置表单的权威来源；客户端不猜测业务含义。
- 有注释时注释为主标题、Key 为副标题；没有注释时 Key 为主标题。
- 默认优先展示缺值必填项，再展示缺值可选项；“查看全部”才展示已有值。
- 本机值不得自动进入测试或正式环境；跨环境复用需要显式绑定或确认。
- 任务内没有匹配值时允许原地新增配置中心条目并立即绑定，不要求跳转页面。
- 配置中心必须显示每个值被哪些项目和环境引用；引用关系只从各项目 `EnvironmentBinding.configProfileIds` 派生，ConfigProfile 不保存第二份反向 bindings。修改时区分“仅保存”和“保存并应用到当前环境”，正式环境应用属于生产变更。
- 配置完整性严格按项目要求的精确 Key 判断，任意其他名称或仅凭条目数量不能替代缺失 Key。条目绑定后 Key 身份只读；要更名必须新建正确 Key 并只解除当前项目 + 环境的旧绑定，不能连带解除其他引用。
- ConfigProfile 和环境应用分别保存修订标识。编辑配置中心条目默认只是“仅保存”；只有用户显式选择应用到某环境，才创建配置 Task。Attempt 开始时冻结该环境完整的 Profile ID 与 revision 集合，不保存敏感明文。
- 显式应用配置创建独立配置 Task，而不是部署 Task；production 必须先确认影响。只有 Attempt 成功且环境绑定仍与冻结输入一致时，才用整份冻结修订推进该项目 + 环境的 applied revision；执行中改绑必须重新确认。目标环境只能有一个推荐候选，Profile revision 与 applied revision 无变化时不创建任务。
- `DEPLOYDESK_*` 等内部字段随配置保存但不进入用户表单和缺失计数。

### FR-08 在本机运行

- 本机运行可独立使用，不是测试部署的强制门槛。
- 页面按“项目服务 → 运行依赖 → 必要配置”展示，支持逐个和批量启停。
- 只有项目服务最近一次完整启动成功，才显示可打开或可验证的本机结果。
- 客户端执行已有或高置信生成的运行方式，不承担代码、依赖声明和数据库业务初始化的 AI 修复。

### FR-09 状态、待办与恢复

- 生成测试版的部署任务在创建时固定完整 40 位源提交；发布正式版的部署任务只固定 Version ID、目标环境、意图和期望结果。任务至少包含项目、不可变目标、阶段、状态、时间和恢复动作；后续当前 HEAD 变化不得改写测试 Task，页面后来选择的版本也不得改写 production Task。
- 首次任务同时保存简短上线计划、已满足项、仍需用户处理项和当前唯一问题，关闭或重启后不得重新计算成另一套顺序。
- 任务在产生任何服务器、文件、Git 或云端副作用前必须持久化；等待用户补信息时允许 Task 存在但 Attempt 数为零。只有真正开始一次外部执行时才创建 Attempt：production Attempt 按 Task 的 Version ID 从 Version 权威对象复制完整 ArtifactRef 集合，同时冻结连接及修订、SourceBinding、服务器绑定及能力修订、完整配置 Profile ID/revision 集合、服务/部署定义修订和待切换地址。重试只增加尝试记录，不创建新的顶级任务。
- 已有部署接管、连接验证和已发布环境的单独地址维护都不是“把版本放入环境”，不得为了展示进度伪造 DeploymentTask 或 DeploymentAttempt。
- 待处理、后台进行中和最近完成结果分开；自动任务不制造人工待办数量。
- 点击待办恢复准确环境、版本和停点；普通点击项目仍进入发布中心。
- 外部网页、DNS 或云平台操作后自动复核，也保留“检查并继续”按钮和最近检查时间。
- 任何状态变化都绑定原项目，异步结果不能覆盖用户后来打开的项目。
- 授权失败、production 启动失败、配置缺失、服务器断连和路由冲突都必须复用原 Task，并只在真实继续执行时追加完整输入 Attempt；修复入口复用配置中心或连接维护，旧正式版本与当前地址在成功切换前持续在线。

### FR-10 面向行动的错误

错误一级信息必须包含：发生了什么、已经完成什么、唯一下一步。授权不足直接提供“更新授权”，服务器连接失败提供“更换连接/重新检查”，DNS 未完成提供精确记录值；脱敏日志和错误码折叠展示。

### FR-11 安全与风险确认

- Token、密码、私钥和敏感配置进入系统密钥库，不进入 Git、SQLite、普通日志和通知。
- 只读识别可自动执行；本地文件写入、服务器初始化、正式发布和破坏性动作按风险逐级确认。
- 服务器主机指纹首次确认并固定，发生变化时阻断。
- P0 不自动删除未知容器、目录、数据卷、数据库或 DNS 记录。

### FR-12 自动化规则

- “主分支更新后自动生成版本并部署测试环境”是一条持久自动化规则，不是页面临时开关。
- 自动化是首个测试版成功后的默认结果，不作为小白首次流程中的额外选择题；用户只在项目设置中暂停、恢复或更换其连接。
- 规则分别保存 desired state、observed state、触发分支、目标环境、Provider 引用、sync state、最近同步时间与错误。暂停、恢复和修复是代码平台写操作，执行前必须验证当前项目 SourceBinding；仓库只读核对失败时规则的 desired、observed、Provider 与 sync 均保持原样。
- 暂停、恢复、同步失败和修复都更新同一 AutomationRule；规则维护本身绝不创建部署 Task，也不产生 Version、Attempt 或环境切换事实。
- 自动化失败产生自己的待处理事项，不伪装成某个环境已经离线；关闭自动化也不删除历史版本和部署记录。

## 5. 非功能要求

- 已知项目重新进入或普通重载时首先恢复“发布中心”的本地可信事实与当前任务；不恢复上次随手浏览的“本机运行 / 版本 / 项目设置”页面。合法显式 deep-link 是唯一可直接指定项目页面的例外。远程检查在后台局部更新，不允许整页等待固定超时。
- 已加载项目切换应即时出现骨架或缓存内容，用户不感知约 5 秒空白卡顿。
- 页面不因轮询或状态刷新主动改变滚动位置。
- 支持 `640×480` 至常见桌面尺寸，无页面级横向溢出；状态不能只依赖颜色。
- macOS、Windows 和 Linux 的系统密钥库、文件选择器、SSH 与辅助功能都需实机验证后才能正式发布。

## 6. 成功指标

- 目标用户首次测试部署成功率与中位耗时。
- 添加项目后到第一次看到可打开测试地址的中位耗时，以及期间实际用户决策组数量。
- 用户能否在 10 秒内说清当前环境状态和下一步。
- 用户在开始首次任务前，能否说清还需要自己完成几类事项且不会误以为还会无限追加步骤。
- 返回用户更换失效凭据并继续原任务的完成率。
- 第二个项目复用已有资源、无需重复粘贴凭据的比例。
- 测试部署失败、业务标记“测试有问题”或地址未就绪时，误触重新构建/重新发布的比例。
