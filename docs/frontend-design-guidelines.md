# ABCDeploy 前端页面设计规范

> 状态：当前唯一生效的前端设计规范；最近对齐：2026-07-16。
> 适用范围：桌面客户端，以及后续保持同一产品体验的 Web 页面。归档原型不属于实现依据。

本文统一此前散落在产品定义、用户旅程、历史原型和设计审计中的页面原则。若其他当前文档与本文冲突，以本文及 [整体产品工作流](product-workflow.md) 为准；历史原型只用于追溯，不作为实现依据。

## 1. 设计目标

ABCDeploy 面向缺少部署经验的 vibe coding 用户。页面设计的首要目标不是展示系统能力，而是让用户始终理解：

1. 我现在处于哪个使用场景。
2. 当前状态是什么。
3. 是否需要我处理。
4. 下一步会得到什么结果。

页面使用用户结果组织信息，Docker、Compose、镜像、SSH、Caddy、流水线和文件路径等实现细节默认后置。

## 2. 组件优先原则

线上版本现有 UI 组件库方向是正式基线。实现页面时必须优先复用现有组件和设计令牌，不为单个页面重复设计按钮、输入框、弹窗、开关、提示、进度条或其他通用控件。

### 2.1 当前技术基线

桌面端组件体系位于 `apps/desktop/src/components/ui`，当前基于：

- Radix UI：提供可访问的 Dialog、Tooltip、Switch、Label、Progress、Select、Dropdown Menu、Collapsible 等基础行为。
- Tailwind CSS：负责布局和样式组合。
- `class-variance-authority`：管理组件变体。
- `clsx` 与 `tailwind-merge`：合并条件样式。
- Lucide：统一图标来源。
- Sonner：统一轻量消息反馈。
- `apps/desktop/src/styles.css` 中的 CSS 变量：统一颜色、表面、边框、状态和深浅色主题。

当前已有 `Button`、`Dialog`、`Input`、`Label`、`Progress`、`Separator`、`Switch` 和 `Tooltip` 等本地组件。新页面必须先从这些组件和已安装的 Radix Primitive 中组合。

### 2.2 组件选择顺序

实现一个交互时，按以下顺序决策：

1. 直接使用 `components/ui` 中已有组件。
2. 使用已有组件组合成页面区块。
3. 若缺少通用组件，基于已经安装的 Radix Primitive 在 `components/ui` 中补充一次，并保持现有组件 API、令牌和可访问性风格。
4. 只有产品特有的复合交互才创建业务组件，例如项目服务列表、场景状态页签或部署时间线。
5. 不在页面文件中临时复制一套按钮、弹窗、下拉框或状态标签。
6. 不因单个页面引入新的 UI 组件库；确需引入时必须单独评审一致性、包体积、可访问性和维护成本。

业务组件内部仍应组合基础 UI 组件，不能绕过组件层自行实现焦点管理、弹窗遮罩、键盘交互或浮层定位。

### 2.3 设计令牌

- 页面使用 `styles.css` 中的语义变量，例如 `--background`、`--surface`、`--foreground`、`--border`、`--accent`、`--success`、`--warning` 和 `--destructive`。
- 业务页面不硬编码一套新的品牌色或状态色。
- 新增颜色必须先成为语义令牌，并同时考虑浅色和深色主题。
- 间距、圆角、字号和控件高度优先沿用现有组件，不为单页制造近似但不同的尺寸。

## 3. 信息架构

### 3.1 应用级结构

- 左侧项目栏始终存在，只负责添加项目、切换项目和展示项目状态摘要。
- 项目状态摘要必须说明具体场景和结果，例如“正式发布需要处理”或“测试部署没有完成”；不得使用脱离场景的“需要处理”“失败”“等待部署”。详细待办恢复后，优先展示“还差网页保存”“等待确认测试结果”等可直接行动的结论。
- 配置中心和应用设置属于应用级入口，不混入单个项目的部署步骤。
- 配置中心必须明确为可选复用工具，不得用空状态或主文案暗示用户需要提前维护；项目配置仍允许直接填写。选择已有值只是把内容一次性填入当前项目环境，不形成持续绑定；之后修改或删除配置中心条目不能反向改写项目已保存配置。
- 选择项目不是部署流程的一步。
- 首次空工作区必须在系统文件夹选择器之前说明“选择整个项目的最外层文件夹，不要只选前端或后端子文件夹”；系统选择器标题继续携带同一提示。这个说明属于添加入口的即时帮助，不能扩展成新的部署步骤或多页教学流程。
- 选择新项目后必须先显示只读识别状态，再给出一段持久的识别结论和一个具体下一步。部署型产品的主操作写“准备上线”，并明确本机运行不是上线前置条件；短暂 Toast 只能作为补充，不能承担“识别到了什么、接下来做什么”的全部信息。
- 持久识别结论已经完整包含服务数量、是否影响上线和唯一下一步时，不再同时弹出内容相同的成功 Toast。Toast 只补充页面上没有的新信息，不能让用户在两个位置读同一结果。
- 新项目识别和已有项目恢复是两种不同场景：前者写“正在识别”，后者写“正在恢复”。不能用“找到上次位置”描述刚刚选择的陌生项目。
- “没有识别到项目服务”只能用于扫描器确实没有找到可运行入口的目录，不能把“没有命中已知框架”当成同义词。根目录存在标准 Node.js `start` 或 `start:prod` 脚本时直接进入项目服务页面；框架名、包管理器和自动生成 Dockerfile 属于实现细节，不在识别成功结论中要求小白确认。
- 外部状态检查返回“仍需处理”也是一次真实完成的检查，不等于请求失败。页面原位显示检查时间和仍待处理的对象，例如“地址还未生效”；只有请求本身失败才显示失败提示，并保留上一次可信检查。尚未发生本机检查时才使用“任务记录”时间，不能在用户刚看到转圈结束后仍用模糊时间冒充反馈。
- 错选空目录或子目录时，把识别问题保留在“所有项目”页并给出“重新选择文件夹”，不能依赖会消失的技术错误 Toast。未通过识别的目录不进入项目列表。
- 重复选择已有项目属于导航，不属于再次添加。界面应说明不会创建重复项目，并打开原部署进度，不重复展示首次识别结论。

### 3.2 项目级场景

项目主区域顶部使用四个稳定、可反复进入的场景页签：

```text
在本机运行｜管理版本｜部署测试版｜发布正式版
```

- 页签标题表达用户动作，第二行显示当前状态。
- 不使用覆盖全部流程的五步 Stepper。
- “管理版本”承接一次性上线设置、自动部署状态和可发布版本选择，不与测试环境强制耦合。
- “准备测试版、正在部署、验证测试版”是“部署测试版”页签内部的状态，不是全局导航。
- 页签允许进入并查看受阻原因，不通过禁用页签隐藏信息。
- 正式发布必须基于测试通过的版本；本机运行不是测试部署的强制前置条件。
- 版本的第一识别对象是生成时间、用户命名和“测试通过/当前环境”等业务资格。任意英文提交说明和“初始化项目、完成首次上线配置、更新上线配置”等产品动作只作为“修改说明”展示，不能直接充当版本主标题；用户真正填写的中文说明和明确版本号可继续作为主标题。多个版本拥有同一产品动作说明时，主标题必须回退到各自生成时间，不能让历史版本看起来同名。长说明默认截断，完整内容在悬停或技术详情中保留。
- 多个测试通过版本可发布时，默认候选必须标明“当前测试版 · 已选择”。用户改选历史版本后，当前测试版仍显示“当前测试版 · 推荐”，目标历史版本显示“已选择”；不能只用背景色或两个没有原因的“测试通过”让用户猜测系统为什么默认选择。
- 全新项目的版本空状态先解释“版本可以测试或发布”，并明确首次设置只需完成一次、自动生成测试版不会直接发布正式版。`main`、监听、流水线等实现词不进入空状态；需要说明触发方式时统一写“把准备上线的代码合并到项目主分支”。
- 正式版已经成功时，页面标题、当前版本和访问地址已经足以证明结果，不再追加一张重复“当前状态：已经完成”的部署卡。运行复核保留为标题区次要动作，失败和需要处理时才展开完整部署状态。
- 测试版成功状态同样不追加泛化的“当前状态：已经完成”。“测试版正在服务器运行”、测试地址和用户确认结果已经分别说明系统结果与业务结果；只有部署仍在进行或停住时才显示“当前进度 / 任务停在”。

### 3.3 页面信息顺序

- 用户最关注的结果放在最前面，底层实现放在后面。
- 正常状态压缩显示，异常和需要用户处理的内容自动提升。
- 数量不固定的项目使用通栏纵向列表，不为了视觉对称强行左右分栏。
- 页面只保留完成当前场景所需的信息，不能把所有能力同时展示成控制台。
- 一个结果需要连续准备多类信息时，先显示紧凑状态摘要，再只展开当前未完成的一项；已完成项默认收起并允许按需修改。

## 4. 页面布局

- 桌面端以稳定侧边栏加单一主内容区为基础，不在主内容中再嵌套第二套侧边导航。
- 侧边栏项目位置在一次应用会话内保持稳定。点击项目、后台刷新状态或更新时间变化只更新该行内容，不按“最近打开”把当前项目反复挪到第一位；新增项目可以出现在顶部，移除或重新关联项目后再更新结构。“所有项目”页仍按人工待办、自动任务和最近使用排序，两种区域分别承担稳定导航与任务发现。
- “所有项目”页不能把人工待办、后台任务和正常项目混在一张名为“最近项目”的列表中，只依赖排序暗示优先级。列表保持同一套紧凑项目行，但明确分为“等待你处理、自动处理中、其他项目”；没有任何待办或后台任务时，正常分组直接称为“项目”。分组标题同时展示数量，让用户重新打开应用后无需逐行判断先做什么。
- 主内容保持明确的阅读宽度和自然纵向滚动，不用多栏压缩核心任务。
- 桌面窗口必须支持至少 `640 × 480` 的逻辑视口，覆盖 `1440 × 960` 显示器在 200% 缩放下的可用范围；窄窗口可以折叠侧边栏文字，但图标按钮必须保留 `aria-label` 和 Tooltip，不能在视觉收缩时变成屏幕阅读器中的空按钮。
- 页面标题使用正常产品字号，不使用营销落地页式巨型标题。
- 独立对象、独立选择或独立状态组可以使用卡片；不使用卡片套卡片。
- 同类数据优先使用列表行，共享表头或分组标题，避免每一项都做成大卡片。
- 主操作在页面结尾或当前问题附近保持稳定位置，不让用户在多个区域寻找下一步。

## 5. 导航与状态

- 导航表示稳定位置，动作不伪装成导航。
- 浏览页面必须保持只读：仅切换页签、查看空状态或展开说明，不得自动创建待办、改写项目状态、保存上线进度或复用远程资源。只有用户点击“准备自动生成版本”等明确动作后，才记录任务并执行准备；从阻塞页点击“继续完成设置”本身可视为这次明确动作。
- 多步骤设置必须区分“用户正在查看的步骤”和“系统真正等待完成的步骤”。展开已完成项或后续项只改变页面内容，不能改写侧边栏状态、任务中心、持久化停点或自动推进顺序；全局待办始终由最早的真实未完成项决定。重新进入设置时自动展开这个真实缺口，不恢复上一次临时查看的其他折叠项。当前未完成项写“现在完成”或“需要你完成”，用户展开的非当前项写“正在查看”，尚未轮到的项写“随后处理”。
- 页签名称固定，页面标题可以随内部状态变化，例如“准备测试版 → 正在部署测试版 → 验证测试版”。
- 等待、自动处理中、成功、失败、需要用户处理和不可用状态必须具有一致语义；第一层状态应直接说明是否需要用户操作。
- 一个复合步骤必须拆开表达各层结果，不能同时用“已同步”和“还差保存”描述同一个“配置”。例如先写“配置内容已齐全、运行服务器已准备”，再单独写“还差网页保存”，让用户知道已经完成的部分不会丢失。
- 父容器不知道子模块具体缺口时使用“确认配置”等中性动作，不自行猜测“补齐缺少项”。配置卡的标题负责状态，空内容区负责解释为什么没有字段，底部按钮负责下一步；三处不重复同一句“保存后即可使用”。
- 由客户端维护的保留字段不属于“高级配置”，不得因为用户点击“查看全部”而暴露。隐藏仅作用于界面和缺失判断，保存时仍保留原内容；没有用户字段时展示空状态和对象明确的准备动作。
- 上一步可能是填写、自动准备或仅确认时，后续提示统一写“先完成配置”，不得擅自写成“先保存上面的配置”。外部提供者名称只在用户进入对应操作后出现，尚未轮到时使用“代码平台网页”等对象语言。
- 外部平台要求勾选多项授权时，先用中文动作列出完整清单，再把平台权限代码作为每项的辅助信息，并提供一次复制全部代码的现成动作。不能只给一串 scope 让小白自己猜含义；令牌输入仍保持安全遮盖，并明确只保存在系统密钥库。
- 同一种账号授权在首次连接、测试失败恢复和正式发布恢复中必须复用同一个组件与完整权限清单，不能在失败时退回一行原始 scope。页面明确账号供所有项目复用，并诚实区分“已验证账号或当前只读访问”和“将在真实写入、创建或触发前继续核对”；不能为了测试权限静默创建仓库、修改设置或触发构建，也不能把只读检查写成全部权限已验证。
- 需要后台识别后才能确定的可复制字段，在识别期间显示“正在识别”，失败后明确要求以外部平台为准，不能先渲染一个看似完整但可能错误的值再静默替换。域名主机记录尤其不能临时使用完整域名，避免用户在 DNS 平台生成重复后缀。
- 页面不能把瞬时外部状态写成可跨重启恢复的事实。例如只能记录“还差网页保存”，不能在任务中心长期写“网页已经打开”；重新进入时主操作明确写出会重新复制什么并打开网页，由用户决定是否再次打开。
- 外部网页停点的项目标题、准备摘要和当前操作不能连续重复“还差网页确认”。项目标题只说明一次总体缺口，摘要只保留已经完成的事实，当前操作区只说明网页动作；已经完成的配置表单不再占据当前操作区。
- 状态不能只依赖颜色，同时提供文字，必要时提供图标。
- 绿色只表示已经验证成功；红色只表示阻断或危险；警告色表示需要注意但未必失败；主操作色不承担状态含义。
- 长任务显示“准备项目、生成版本、启动服务、确认可用”等用户可理解阶段，不直接暴露内部状态键；应用关闭或刷新后应恢复状态，不重新开始。
- 同一项目的测试部署和正式发布是两条可独立进行的任务；全局任务入口必须按环境保留上下文，不能用单一“项目最新状态”覆盖另一环境尚未完成的操作。
- 项目摘要必须汇总全部环境的未完成任务，按“需要用户处理、失败、自动处理中、最近成功”显示唯一结论；侧边栏、所有项目和任务中心不得同时给出互相矛盾的状态。
- 项目摘要先表达最近可信事实，再补充可选结果：测试环境实际运行但尚未人工确认时写“等待确认测试结果”，确认后写“测试已通过，可发布”。“可以发布”不属于必须处理的任务，不计入“等待你处理”；不得用较低层的“容器正在运行”覆盖已经确认的业务结果。
- 任务中心的完成记录也要区分技术部署结果与业务确认结果：只有当前不可变测试版本已人工确认，或已经作为正式版来源使用时，才写“测试已通过”；仅完成服务器部署时写“测试版部署完成”，尚待确认的当前版本只进入“等待你确认”，不能同时出现在最近完成。
- “所有项目”必须先回答有没有事情需要用户处理：保持单一项目列表，用一行概况汇总人工待办与自动任务，并将人工待办置顶、自动任务其次；正常项目内部继续按最近使用排序。搜索结果不能冒充全局概况。
- 恢复已保存业务状态时，在数据确认完成前使用中性的“正在恢复”，禁止用空数组或默认值渲染“0 个版本”“未启动”“尚未发布”等业务结论；占位应说明不会触发启动或部署。
- 应用刚启动、尚未读完本机工具状态时，侧边栏使用“正在检查本机环境”和中性进行态，不能把未知状态先画成警告色的“本机环境待准备”。检查明确失败后改为“本机环境检查未完成”并原位提供重试；后台首次失败不弹全局错误，也不能阻塞项目列表和上次项目恢复。
- 已有成功测试版本时，“测试环境上次已准备”是可信历史状态。查看一次性设置期间继续显示完成；同一轮后台复核只要还有任一服务器、配置或地址子检查未结束，就必须保留完成态并局部显示“后台确认中”。只有整轮检查结束且得到明确的当前缺口，才更新为待处理并展示原因。
- 多项目长任务从侧边栏统一进入，离开当前项目页仍继续后台刷新；失败卡片同时展示“已经完成”和“下一步”。全局任务中心只保留少量最近完成结果，更多历史回到对应项目查看，不能用成功记录淹没当前待办。
- 任务卡的标题、动作、正文和页脚必须各自回答不同问题：标题只写当前停点，动作只写下一步，正文只说明如何完成，页脚只保留进度已保存或已经完成的事实。同一缺口不能用“完成、继续、还差、当前”等近义句在一张卡里重复四遍。
- 等待用户在浏览器、DNS 或云平台完成外部操作的任务，在恢复页面和应用重启后主动复核一次，窗口重新获得焦点时立即复核，停留在可见页面时定期继续检查；并发检查必须合并，应用不可见时暂停。复核失败必须明确写“这次检查没有完成”，保留原任务停点并继续提供同一个立即检查动作，不能把请求结束误写成状态已经检查成功。
- 部署任务的通用 `updatedAt` 只能写“任务记录更新”，不能写“上次检查、上次确认、上次状态”；代码平台历史同步也会刷新它。只有真正调用环境复核并成功返回后生成的本轮时间，才可以显示为“最近检查”。
- 同一恢复动作不能串行重复验证同一个外部对象，也不能重复执行已经由可信停点证明成功的幂等准备。可以复用本轮内存中的非敏感验证结果，但服务器身份、必填配置和最终外部结果仍要在各自真正影响安全或推进流程的边界核对；敏感配置不能为了提速写入普通缓存。
- 能用公开信息安全判断外部操作位置时，提供具体入口，例如根据域名 NS 记录打开对应解析控制台；不能要求用户自己搜索，也不能为了便捷读取、上传或持久化无关账号密钥。无法可靠判断时回退到通用说明，不伪造确定性。
- 同一次检查发现多个地址或服务未就绪时，一级摘要要覆盖全部失败对象或明确数量，不能只报第一个却在明细中列出更多。聚合不能抹掉“服务已经部署”等已完成事实，也不能把地址复核变成重新部署。
- 同一行的主状态和副说明必须回答不同问题。当前状态已经写“未启动”时，副说明优先展示“上次随全部服务启动成功”等可信历史结果；没有历史结果时才写“等待启动”，不能让两处重复表达未运行。“随全部服务”用于说明这条历史结论来自最近一次完整启动，不能把单个服务行写成容易误解为该对象包含全部服务的“上次全部启动成功”。

## 6. 按钮与操作

- 一个页面或一个当前问题只突出一个主操作。
- 主按钮使用结果性文案，例如“创建代码仓库”“部署测试版”“发布正式版”，不使用含糊的“下一步”“确定”“执行”。
- 页面达到已确认结果后，主操作要承接用户的下一件事：测试通过且尚未正式上线时突出“准备发布正式版”，把“重新打开测试版”降为复查；同一版本已经在线时移除重复发布动作并说明事实。确认结果、当前运行事实、复查作用和下一步必须各自只说一件事，不能在页面标题、状态卡、复查卡和下一步卡连续重复“已经测试通过”。
- 同一结果页的页面说明、状态卡和辅助卡分别承担结论、当前事实和可选动作，不能连续重复同一句状态。用户已经确认测试结果后，页面说明写“已经通过验证”，复查卡写“需要时重新打开”，服务器运行事实只保留在部署状态中。
- 通用异常复核写“重新检查部署结果”，不写“刷新状态”或“检查并继续”。已知问题必须进一步收敛成“更新授权、检查服务原因、重新检查地址”等具体结果，不能让通用文案覆盖已有诊断。
- 只读刷新也必须形成可感知闭环：执行中使用“正在刷新版本记录”等对象明确的状态，完成后保留更新时间；不能因为数据没有变化就让操作静默结束，迫使用户反复点击。
- 健康正式版的“检查运行状态”也遵循同一闭环：只把真正完成公开地址检查的时间保存为“上次成功检查”，不能用部署记录或代码平台同步的 `updatedAt` 冒充健康确认。没有检查记录时诚实说明页面显示最近一次部署结果；主动检查成功后写“最近检查 + 正式版仍可访问”，失败后保留上一次成功检查并持久写“仍显示上次可信结果”，不能只让按钮转一圈或只弹临时 Toast。
- 测试版和正式版成功页的“检查运行状态”必须先通过该环境绑定的服务器连接，按照本次部署时已经提交的项目清单逐一核对服务进程与健康状态，再核对需要公开访问的地址；不能只更新时间、读取历史部署记录或只检查网页入口。检查只读且不重新构建、不重新部署；服务缺失或未就绪时就地转为具体待办，检查请求本身失败时保留上一次可信部署结果。测试版使用本机临时打开方式时，服务复核成功后仍保留“重新打开测试版”，不能误改成公开地址流程。
- 打开说明与执行恢复必须分开命名。只打开失败详情的按钮写“查看处理方法”，不能写“查看并继续”；真正会重试、发布或修改外部状态的按钮在详情中使用独立的结果性文案。
- 运行状态必须说明对象：“本机环境可用”“2/3 个项目服务运行中”“正式版可以访问”分别表达环境、服务和发布结果，禁止用一个笼统的“运行正常”覆盖不同层级。
- 批量操作名称必须符合实际作用范围。部分项目服务因开发问题不可运行时，写“启动可运行的 2 个服务”，并在同一区域说明剩余数量；不能保留“一键启动全部”让用户把部分成功误解成全部成功。
- 没有任何可运行对象时不展示零数量动作，也不使用“即可启动”一类承诺。页面直接说明哪些对象需要先由开发工具补齐，添加成功反馈只陈述已经真实完成的识别结果。
- 会改变正式环境的候选版本属于一次性操作意图，不作为页面默认状态长期保存；重新进入页面先展示线上事实，只有用户显式选择版本后才展示发布或恢复动作。唯一例外是用户已经进入需要离开客户端的外部停点：此时必须把非敏感停点与候选版本的不可变身份一起保存，直到该停点完成，避免重启后恢复到另一个版本；停点结束后立即回到线上事实优先的规则。
- 全局任务卡的标题、右侧动作和底部停点使用同一个真实阻塞对象：地址、授权、配置、服务或版本核对。禁止标题写“等待地址”、动作写“继续处理”、停点再写“检查访问地址”；未知失败使用“查看处理方法”，不提前承诺“继续”或成功。
- 分支、提交和流水线等开发术语不作为第一层动作说明；面向用户优先写“代码合并到主分支后自动更新测试版”，`main` 等实际标识只在设置详情或技术信息中出现。
- 一次性设置的完成态必须说明以后会发生的用户结果，不能只陈述系统保存了什么。自动部署完成后写“代码合并到主分支后自动生成版本并更新测试版，正式版仍由你确认发布”；正式配置完成后写“发布时会使用这里保存的配置”。安全位置、密钥文件和检查机制只在重新设置或技术详情中出现。
- 恢复已知项目时先显示该项目的结构化占位，不得闪回项目列表或整页白屏；上次场景属于首屏核心数据，读取完成前不得先渲染一个默认场景再跳转。本机工作区和场景可用后立即交互，最近记录保存、列表刷新和远端同步均在后台继续。
- 新项目扫描占位必须说明系统只读取项目结构，不运行代码、不读取配置值，并承诺识别完成后给出结论。识别结果只陈述真实发现的服务数量，不提前承诺项目可以运行；主操作直接承接一次性上线设置，本机服务和配置作为可选检查留在当前页面。
- 桌面窗口出现到前端脚本接管之间也必须有极简、可访问的启动状态，不能留下空白根节点；恢复当前项目是首屏最高优先级，全项目待办扫描等非关键工作必须等首屏出现后再启动，避免争用本机存储和文件扫描资源。
- 异步忙碌态必须绑定到用户正在执行的具体动作，不能用一个全局 `loading` 顺带锁住无关入口。已有项目列表一旦可用，“添加项目”不能因后台恢复另一个项目而转圈或禁用；只有系统文件夹选择器尚未返回时才显示该动作正在进行，并合并重复点击。
- 系统密钥库已经保留账号、但本机摘要丢失或外部平台响应较慢时，账号恢复不能无限显示加载动画。短暂等待后原位说明“读取用时较长”，提供“重试读取 / 重新连接”两个明确恢复动作；后台请求必须设置总超时并并行读取互不依赖的账号资料。一级文案使用“代码平台账号”，具体厂商只在重新连接弹窗中出现。
- 最近项目的原文件夹不存在时，把它提升为“需要重新找到项目”的人工待办，并在所有项目、侧边栏和任务中心提供同一个恢复入口；用户选择新位置后必须先核对代码仓库或项目结构，再整体迁移部署记录、服务器与配置绑定、页面进度和当前项目。恢复后的本机项目身份不得随绝对路径改变，不能让目录移动表现成配置、版本或密钥全部丢失。
- 次要操作使用 `secondary`、`ghost` 或 `link` 变体，不与主操作竞争。
- 全局固定入口只保留一个稳定位置。项目添加统一位于侧边栏项目区；已有项目时，所有项目页不再重复同名入口。只有首次空状态可以在正文提供带说明的“选择项目目录”，帮助新用户完成第一个动作。
- 危险或不可逆操作使用 `destructive`，并在执行前说明具体影响。
- 图标按钮必须使用 Lucide 图标，并提供中文可访问名称和 Tooltip。
- 图标按钮只有一个实际动作时，名称和 Tooltip 直接写该动作，不使用“更多、选项、操作”等会暗示还有菜单的泛化名称。确认弹窗必须重复具体对象名，避免用户离开原列表上下文后不知道正在影响哪个项目。
- 禁用操作必须同时解释开放条件；能进入页面解释原因时，不直接禁用导航入口。
- 已经完成且当前没有改动的保存操作直接隐藏，不保留一个不可点击的“保存配置”与真正下一步竞争；用户展开并修改内容后，再让保存按钮出现。
- 后续场景同时缺少多项前置条件时，只展示最早且可处理的真实阻塞项，并让主操作直达该项；不能先提示后续结果，再让用户经过中间页面才发现更早的设置没有完成。
- 已经存在进行中或等待处理的部署任务时，当前页面不能同时提供创建另一条部署意图的入口。授权、网页保存、服务、地址等停点先原地完成；“发布或更换版本”“准备发布正式版”“手动部署测试版”等新任务入口只在当前任务成功结束后恢复。从历史结果进入也必须读取当前任务，不能因为入口来自“最近完成”就退回旧结果页的默认下一步。
- 顺序设置中只有当前展开项可以写“正在处理”或“需要你完成”；尚未轮到的折叠项统一写“随后处理”，不能把多个未来步骤同时表现成等待用户处理的待办。
- 由前一步自动派生的设置只有在前一步的真实业务对象已经通过校验后才能自动完成；模板占位值、示例名称和待替换地址都不算准备完成。展开尚未轮到的派生项时，只说明它会跟随哪个用户动作自动准备；如果存在已经保存且可验证的替代资源，仍保留原位检查和切换入口。
- 用户提前展开尚未轮到的步骤时，可以解释它依赖什么以及之后会发生什么，但不提前渲染服务器、配置、凭据、网页授权或部署按钮。轮到该步骤后再在原位置呈现唯一真实操作；不能用一组禁用表单代替依赖说明。
- 一个设置项内部仍可能包含依赖顺序时，继续逐段披露。例如测试环境必须先确认服务器，再展示配置，配置验证通过后才展示地址；不能因为它们属于同一张卡片就同时放出多个可推进动作。
- 流程底部的预期说明必须覆盖全部剩余步骤。只有所有前置项完成后才能写“正在进入测试版”；中途统一说明会依次完成剩余设置，不能把当前小步骤完成误写成整个流程完成。

## 7. 表单与配置

- 表单只收集系统无法安全推断或自动完成的信息。
- 输入控件必须有可见 Label；占位文字不能代替字段名称。
- 项目配置项优先显示缺少值的必填项，再显示缺少值的可选项；“查看全部”才展开已有值。
- 配置注释作为主标题，配置名称（例如 `DATABASE_URL`）作为副标题；没有注释时直接显示配置名称，不由客户端猜测业务含义。
- 配置值既可以直接填写，也可以选择配置中心中的已有值。
- 本机配置已经没有必填缺口、且不会覆盖项目现有 `.env` 时，由客户端自动保存并生成运行文件，不增加一次纯机械确认。发现不同的现有 `.env` 时自动动作静默停止，保留明确的手动保存入口；只有用户主动执行后才弹出备份并更新确认，不能在进入页面时突然要求覆盖文件。
- 配置中心只向当前字段提供名称匹配的已有值，不能把无关配置列表或全局维护任务推给用户。
- 敏感值默认遮盖，不在普通页面、日志、通知或错误信息中显示原文；识别不能只依赖字段名，带账号密码或令牌的数据库、缓存和普通 URL 也按敏感值处理。输入时允许通过带对象名称的眼睛按钮短暂显示，切换配置名称、关闭编辑或重新进入时恢复遮盖；编辑已保存敏感项并留空代表保留原值，不能把它改写成空的普通配置。
- 保存、验证和外部写入是不同动作时必须明确区分，不能用一个含糊按钮混在一起。
- 没有用户配置字段但正式环境仍需生成内部运行文件时，页面写“无需填写，使用当前配置即可”，动作写“使用当前配置并继续”。不能让标题、摘要、卡片和按钮分别重复“准备、确认、等待保存”，也不能把没有输入对象的动作命名为“保存配置”。
- “已经保存”不能替代“可以使用”。第三方登录信息能够低成本验证时，只有真实验证通过后才能显示“已准备”；验证失败必须留在原输入区说明原因和下一步，不能先保存、推进流程后再等远程构建暴露错误。
- 配置完成卡的标题说明保存位置，徽标只写“已保存/已准备”，空列表只解释“需要核对时可以查看全部配置”。三个区域不得重复“配置已齐全、服务器已准备、没有缺少值”；没有用户字段时改为“项目默认设置已经准备 / 不需要额外操作”。
- 已保存的第三方登录信息和服务器连接只能描述为“信息已保存”或“上次已经验证”，不能跨重启继续声称“当前连接正常”。用户主动复用并将改变项目设置时，必须先做真实检查，成功后再保存新设置；失败时保留原设置并在原位置说明处理方法。为了首屏速度，不在每次进入项目时为了装饰绿色状态重复联网，真正生成版本、连接服务器或明确复用时再以实际结果为准。
- 外部平台必须按完成顺序逐段展示操作；用户尚未打开外部页面时，不提前显示“我已完成”，返回后再突出唯一确认动作。
- 已经手动完成的旁路不能与推荐主操作并排竞争；先用次要入口说明适用条件，用户主动进入后才替换成该旁路的唯一下一步。
- 需要离开客户端的多段操作必须保存非敏感进度；重新进入时恢复到最近可信阶段，不重复要求已经确认的外部动作，也不为恢复方便而落盘密钥或完整配置。
- 外部操作进度不能脱离业务对象单独恢复。网页保存、授权确认或地址设置已经针对某个环境和版本开始时，持久化状态至少要绑定项目、环境和不可变版本身份；版本目录在该版本行直接显示具体待办和唯一继续入口，不能让用户重新点击“发布”创建第二条意图。
- 重新进入项目时先渲染最近可信的已保存业务状态，再做局部复核。后台读取不能用“正在确认”等临时业务标题整页覆盖“还差网页保存、等待确认地址”等准确停点；确需显示检查过程时使用局部、次要的“正在检查最新状态”。
- 恢复外部网页操作时，先让用户在“网页已保存，继续”和“还没保存，重新打开网页”两个互斥结果中选择，不能把重新打开、复制内容和确认保存三个并行动作同时抛给用户。第三方为了保护密钥而不提供读取能力时，客户端必须在未发生本轮复制的直接确认前说明无法读取网页内容，并要求用户明确自证；不能把仓库存在冒充成文件已保存，也不能用“检查并继续”过度承诺。
- 外部网页需要粘贴客户端生成内容时，重新进入页面必须先在本机恢复可复制内容，不把重新打开网页当成恢复数据的前置条件。用户选择重新操作后按“复制文件名并打开网页 → 复制配置内容 → 粘贴保存并继续”依次推进；当前步骤只突出一个主动作，复制完成前不显示可误点的保存确认，复制完成后再把确认设为唯一主动作。
- 用户确认外部网页保存成功后，如果系统剪贴板仍是刚才生成的完整安全配置，客户端主动清空；如果用户已经复制其他内容则不得覆盖。页面不要求用户理解这项内部清理，也不能因为清理失败而丢失已经确认的业务停点。
- 恢复外部网页停点时，远程保存位置确认与不会改变远端状态的本机配置重新生成可以并行，不能为了等待一个网络请求再开始另一个独立准备。但远程确认仍是显示文件名、复制内容和确认保存动作的门禁：位置尚在检查、已经失效或检查失败时不得提前写“已经准备好”。本机重新生成失败后必须保留原停点和明确的重新准备入口，不要求用户重走代码平台、服务器与运行配置。
- 凭据输入不能成为断头路。令牌、密钥等必须由第三方网页创建时，在同一个输入弹窗内提供明确的创建入口和“创建后回到这里粘贴”提示；权限代码只作为次要说明，不能要求小白退出客户端后自行搜索创建位置。
- “部署配置已准备”和“当前可以查看密钥内容”是两个状态。某份远端配置已经针对同一项目、环境和服务器真实验证后，本机密钥库暂时需要确认读取时不能把部署降级成“等待保存”，也不能覆盖原成功记录；发布继续使用服务器上的已验证配置，只有用户主动查看或修改敏感值时才请求系统确认。
- 失败状态跨页面或重启保留时，只保存错误编号、用户可理解的标题和下一步；不保存第三方原始响应、命令、日志、配置值或其他可能包含敏感信息的内容。
- 最后一项准备完成后，如果下一场景已经明确，自动进入结果页面，不增加一次没有新信息的“继续”。
- 预计超过 1 秒的本机生成或远程准备动作必须替换原操作区并说明正在做什么，不能只把原按钮全部禁用。
- 启动/停止、启用/停用等互斥批量动作按当前真实状态渐进显示。当前只能启动时不保留禁用的停止按钮，全部运行时不保留禁用的启动按钮；只有部分状态允许两个方向时才同时出现。

## 8. 卡片、列表与渐进披露

- 默认界面展示用户结论、影响范围和下一步。
- 技术详情、原始日志、文件路径、命令输出、Plan 和 Diff 默认折叠。
- 展开技术详情后仍先提供中文摘要，再展示脱敏原文。
- 正常的底层检查合并成简短摘要；只有异常时展开具体检查项。
- 不用大量装饰性卡片制造信息密度，卡片必须有明确边界和独立语义。
- 历史失败与当前待办必须分层：只有最新且仍有效的失败可以要求用户处理；已经被后续部署替代的记录统一放入历史区，并明确写“无需处理”。
- 部署记录的“最近检查时间”不能改变版本先后。历史记录复核后仍留在原位置；当前测试版和正式版只按真实部署开始时间及环境证据确定。
- 运行中或等待外部处理的任务要显示最近确认时间，并明确进度已保存。应用重启后必须继续使用持久化时间，不能让用户重新猜测刚才的操作是否还存在。
- 部署主体已经完成、只剩 DNS 或 HTTPS 等外部生效检查时，不重复展示完整四阶段进度和多段恢复说明。默认卡只回答“服务是否已经启动、还差什么、回来后如何检查、是否会重新部署”，记录类型和目标地址保留在当前操作区，内部路由与代理实现进入技术详情。
- DNS 引导使用域名平台表单里的字段名“主机记录、类型、记录值”，并根据公开托管域把完整域名转换成 `@` 或相对子域名。系统不能确定记录已经存在时，状态写“待添加或生效”，不能只写“等待生效”让用户误以为无需新增记录。
- 历史错误详情先说明它是否影响当前环境，再提供当时原因；第三方原始响应、错误 JSON、构建编号和完整提交号只能放在折叠的技术详情中。

## 9. 反馈、错误与空状态

每个失败页面或错误区块必须回答：

1. 发生了什么。
2. 哪个用户结果受到影响。
3. 系统已经完成了哪些检查。
4. 用户现在应该做什么。
5. 是否可以重试。

- 主信息不直接倾倒第三方响应、Shell 输出或堆栈。
- 已知错误编号必须先映射成结果语言；原始供应商响应、容器名称、镜像摘要和权限代码统一进入默认折叠的“查看技术详情”。横幅、任务卡和历史记录遵循同一规则，不能因为组件不同重新暴露原文。
- 普通异常字符串和结构化 Provider 检查必须共用同一套错误码翻译。底层已经返回 `summary` 或 `nextSteps` 不代表可以直接展示；其中出现 Caddyfile、挂载目录、SSH 私钥、镜像仓库、命名空间、分支名或文件路径时，首层先翻译成“访问服务、安全身份、项目版本保存位置、代码版本”等用户对象，原始摘要和证据进入技术详情。没有已知映射时才保留经过脱敏的 Provider 建议。
- 错误码解析必须覆盖“中文摘要 + 换行后的第三方 JSON、命令输出或原始信息”。换行不能让已知错误退化为未知错误并把整段响应显示在首层；首层继续使用错误码映射，完整脱敏响应只进入默认折叠的技术详情。
- 项目开发问题不在 ABCDeploy 内伪装成自动修复；提供可复制、已脱敏的问题上下文，交给用户的开发工具处理。
- 本机历史状态只知道“上次启动未完成”但没有保留原因时，不能把普通“启动全部”原样当作恢复动作。页面应明确本次操作会“重新启动并记录原因”；若仍失败，保留可理解的原因、下一步和交给开发工具的安全摘要。该动作只在服务列表中出现一次，不能在历史提示和批量操作区重复放置两个启动按钮。
- Toast 只用于轻量、无需后续决策的反馈；需要用户理解或处理的内容必须留在页面中。
- 后台自动复核失败时只在当前状态区保留可恢复提示，不弹全局错误，也不能按轮询周期反复打扰用户。只有用户主动点击检查且本次检查失败时，才额外使用一次全局反馈；两种失败都必须保留原任务、已完成步骤和上次可信状态。
- 检查请求正常返回只代表系统完成了检查，不代表目标环境健康。页面必须读取返回任务的业务状态；只有结果仍为 `success` 才能记录“最近检查”或“仍可访问”，返回 `needs_action`、`failed` 等结果时直接展示对应问题，不能因为 IPC 或网络请求没有抛错就留下绿色时间。
- 环境复核、任务中心刷新和项目摘要刷新是三种不同证据。环境复核返回真实任务后，当前页面必须立即结束“正在检查”并展示该结果；任务中心和侧边栏摘要随后在后台更新。非关键摘要读取变慢或失败不能延长主检查、弹出环境故障，也不能覆盖已经取得的健康或待处理结论。
- 任务已经携带配置、授权、地址、版本核对等准确待办类型时，场景页一级标题必须直接使用同一个具体结论，不能退化成“还需要你完成一项操作”。侧边栏、任务中心和项目页不能因为组件不同而丢失系统已经掌握的阻塞对象；只有底层确实没有提供类型时才使用带环境对象的通用回退。
- “版本已在正式服务器运行”和“正式发布已完成”是两个不同结果。测试结果页即使确认正式环境使用同一镜像，也必须继续读取正式任务状态：地址、授权或其他步骤仍待处理时，不能写“已经在线”或“无需再次发布”，而要说明已完成的服务状态、尚未完成的用户结果，并提供直达原停点的唯一继续动作。只有访问结果真正成功时才使用“已完成正式发布 / 正式地址可以访问”。
- 版本记录中的状态标签不是动作。当前正式版本仍有地址、授权或修复待办时，版本行必须紧邻提供与该对象一致的继续按钮，不能要求用户猜测要切换顶部场景、打开任务中心或进入技术信息。正式任务成功后再隐藏继续按钮；被后续部署替代的历史记录只允许查看证据，不能重新激活旧操作。
- 空状态说明为什么为空，并提供一个明确入口，不能只显示空白或“暂无数据”。

## 10. 文案

- 使用小白能理解的结果语言，不把技术名词作为一级标题。
- 全局任务、折叠步骤和首次说明优先使用稳定的产品对象名，例如“代码平台网页、项目版本保存位置、本机运行版本”。具体厂商或应用名只在用户下一步确实需要打开、登录或授权该对象时出现，不能让尚未进入操作的人先学习 CNB、TCR、OCI、容器构建等实现词。
- 必须出现技术名词时，先解释用途，再在次要位置显示原名。
- SSH 通道、端口、隧道、容器和镜像摘要等实现方式不能用来命名用户动作；页面写“打开测试版、检查服务原因、核对服务器版本”等用户结果，机制只在确有排障价值时进入技术详情。
- 句子简短、具体，避免营销口号、抽象平台术语和责任不清的被动语态。
- 不宣称系统已经完成无法验证的动作。
- 状态文案保持一致，例如统一使用“未启动、正在启动、运行中、启动失败、已停止”。
- 地址尚未完成时使用“正式版已部署，还差地址 / 还差设置地址”，不能写“等待地址生效”。系统无法确认解析记录已经存在时，文案必须同时覆盖“尚未添加”和“正在生效”两种真实可能。
- 安全确认必须先翻译成用户能判断的对象和后果，例如先确认“这是刚才填写的服务器”，再把 SSH 指纹放到可展开的技术详情；不能把一串用户无法核对的散列值当作主确认内容。
- 正式版切换或历史恢复的最终确认使用“目标版本启动失败，系统会自动恢复当前版本”，不能写“新服务启动失败”，避免把整个目标版本的启动检查误解为新增服务检查。

## 11. 可访问性

- 主流程支持完整键盘操作。
- 默认字号、对比度和焦点样式至少满足 WCAG 2.2 AA 相关要求。
- 使用 Radix Primitive 已有的焦点管理、键盘行为和 ARIA 语义，不自行复制低质量实现。
- 所有输入均有关联 Label，错误信息与对应输入建立程序化关联。
- 项目主区域顶部必须提供一个稳定的可访问名称，完整包含当前项目名称、本机路径和项目服务数量；视觉标题因窗口拖动区域、截断或短项目名而被浏览器合并时，屏幕阅读器仍要能一次读出项目身份。扫描结果缺少项目名称时使用路径的最后一级文件夹名回退，不能出现空标题；可访问名称已经概括完整信息时，内部视觉文本从辅助功能树隐藏，避免重复播报。
- 列表中重复出现的“打开、启动、停止、重新启动”等按钮，必须把当前对象写入可访问名称，例如“启动后端服务”“停止本地数据库”；不能让屏幕阅读器只读出一组无法区分的同名按钮。
- 图标按钮和短文案动作必须承诺真实且唯一的后果。例如移除项目只删除本机记录时，可访问名称写“移除本机项目记录”，确认框包含项目名并明确不删除代码、部署文件或服务器数据；任务卡的“继续”和版本选择的“更换”也必须在可访问名称中包含任务阶段或版本对象。
- 状态同时使用文字和视觉标识。
- 状态不能只依赖自定义颜色：系统强制颜色替换调色板后，当前项、成功、等待和处理中仍要通过文字、图标、填充、描边或轮廓至少一种额外方式区分。
- 尊重 `prefers-reduced-motion`；动画不承担唯一信息表达。
- 设计令牌必须响应 `prefers-contrast: more`；`forced-colors: active` 下使用 `Canvas`、`CanvasText`、`ButtonText`、`Highlight` 等系统颜色，不通过 `forced-color-adjust: none` 固定整页自定义主题。只有状态点、数量标记等需要保留形状差异的小元素可以关闭自动替换，并且仍只能使用系统颜色。
- 当前页和键盘焦点在强制颜色模式下必须使用系统 `Highlight` 轮廓，不能只靠浅色背景或阴影表达。
- 后续正式验收包含实际屏幕阅读器、Windows 实机强制颜色和 200% 缩放检查。

## 12. 视口与视觉验收

- 主要桌面画布为 `1440 × 960`，同时验证 `1280 × 720` 和 `640 × 480` 逻辑视口。
- 所有尺寸下都不得出现页面级横向滚动、文本挤出、按钮不可达或固定操作栏遮挡内容；最小窗口下 `document` 与主滚动区域的 `scrollWidth` 必须等于 `clientWidth`。
- 多字段加主按钮的横向表单在窄视口必须自动改为纵向排列，不能依赖输入框被强行压缩来勉强通过无横向滚动检查。
- `640 × 480` 下，四个场景入口改为两列；服务器、域名和登录信息等多字段区域改为单列；配置中心隐藏桌面表头并把每条配置折叠成信息卡；状态与操作同行时允许操作区换到内容下方。`1280 × 720` 及以上必须恢复四列场景、完整表格和横向表单，不能长期牺牲桌面信息密度。
- 内容超出首屏时允许自然纵向滚动，关键状态和主操作仍应容易找到。
- 深色模式不能只是颜色反转，必须检查文字、边框、状态色、遮罩和焦点对比度。

## 13. 原型与正式实现

- 低保真原型只用于验证产品模型、信息顺序、文案和主线，不创造新的视觉系统。2026-07 原型已经归档，不能用其静态状态覆盖真实客户端的新结论。
- 正式实现必须回到现有 `components/ui` 和语义令牌，不直接复制原型中的临时 CSS。
- 原型中验证成功的结构不等于批准其自制控件样式。
- 每次只实现一个闭环，先完成状态、错误和恢复路径，再扩展次要功能。

## 14. 页面评审清单

进入实现或合并前逐项确认：

- 是否复用了现有 UI 组件，是否存在页面自行复制的通用控件？
- 缺少的通用组件是否补进 `components/ui`，而不是只服务一个页面？
- 是否使用语义令牌，是否出现不必要的硬编码颜色和尺寸？
- 用户能否立即看懂当前场景、状态和下一步？
- 项目入口展示待办状态时，点击后是否直达这个待办，而不是恢复到无关的上次浏览页面？
- 项目入口展示“测试已通过，可发布”时，点击后是否直接进入正式发布准备，而不是回到本机运行或版本列表？
- 项目入口展示“正式版可以访问”时，点击后是否直接进入正式版结果，而不是恢复到上一次浏览的测试版或本机页面？
- 是否只有一个明确主操作，文案是否描述结果？
- 技术细节是否默认后置并完成脱敏？
- 状态是否同时有文字，不只依靠颜色？
- 键盘、焦点、Label、错误关联和减少动画是否完整？
- `1440 × 960` 与 `1280 × 720` 是否无横向溢出？
- 页面是否与 [整体产品工作流](product-workflow.md) 和对应模块产品定义一致？
