# ABCDeploy 前端页面设计规范

> 目标：让完全不懂部署的用户在每个页面都知道“我在哪、现在怎样、下一步做什么、为什么需要我做”。

## 1. 组件优先

正式页面按以下顺序选择实现：

1. 复用 `apps/desktop/src/components/ui` 已有组件；
2. 使用 Radix UI 原语组合；
3. 扩展现有组件的语义变体；
4. 只有确认是通用新模式时才新增基础组件。

业务页面不得自行复制按钮、输入框、对话框、下拉框、提示、Badge、折叠区或 Tooltip。颜色、间距、圆角、字体和状态必须使用设计令牌，不在页面散落临时值。

HTML 原型可以验证结构，但不能作为新组件库。正式实现以线上已有组件方向为准。

## 2. 信息架构

### 应用级

左侧固定提供：所有项目、配置中心、连接与资源、任务、添加/切换项目。添加项目不是部署步骤。

“连接与资源”维护可跨项目复用的代码平台、版本存储和服务器能力，展示最近验证、影响项目数和更换入口。服务器 Connection 不显示为天然属于测试或正式环境；项目设置中的环境绑定才说明用途。项目设置只展示当前项目绑定，不复制一套凭据编辑器；失效连接可以原地替换，不要求移除项目。

“所有项目”展示真实项目列表并负责添加、打开和切换。每个项目使用独立工作区，项目级 Task、Attempt、Version、Environment 和 AutomationRule 不得因切换而复用；只有应用级 Connection、ServerCapability 与 ConfigProfile 可以跨项目引用。普通重载和普通打开项目回到发布中心，只有合法显式 deep-link 才恢复指定项目页面。

“任务”使用稳定页面并分开“需要你处理 / 正在自动进行 / 最近完成”；同一任务下展示当前第几次执行和历史执行结果，重试只增加 Attempt。等待用户输入的 Task 不伪装成“第 1 次执行”；真正开始执行后才出现 Attempt。Badge 只计算确实需要用户动作的事项，后台构建和轮询不制造红点。点击任务可以进入准确停点，点击项目则始终进入发布中心。

### 项目级

固定导航为：

```text
发布中心｜在本机运行｜版本｜项目设置
```

- 不再使用“管理版本 / 部署测试版 / 发布正式版”作为并列页签。
- 测试版与正式版是发布中心中的环境卡和上下文详情。
- 流水线步骤不进入导航。
- 历史路由恢复到发布中心，不显示兼容用旧页签。

## 3. 发布中心

已有发布事实的项目首屏依次展示：

1. 项目名称和一句整体结论；
2. “当前要做的事”任务卡，全页只有一个主动作；
3. 测试与正式环境事实卡；
4. 自动化规则摘要和最近活动。

新项目是例外：首屏先只展示“先生成一个可以打开的测试版”和唯一主动作，不展示空的正式环境卡、资源清单或自动化机制。第一个测试版产生后，页面再进入测试/正式环境事实结构。

每张环境卡只回答三个事实：

- 现在在线的是哪个版本；
- 最近一次动作是什么结果；
- 地址是否可用；

环境卡最多提供“查看详情”等次级入口，不各自制造主动作。全局唯一下一步由上方任务卡决定；一切正常且没有待办时允许没有主按钮，不为了满足布局制造无意义操作。

示例：

```text
当前要做的事
7 月 16 日版本发布没有完成
当前正式版仍是 7 月 15 日版本
[继续当前发布]

测试环境                 正式环境
7 月 16 日版本运行正常    7 月 15 日版本运行正常
[查看详情]                [查看详情]
```

不能只写“需要处理”“部署失败”或让一个红点承担含义。在线版本与最新尝试必须同时存在于可访问文本中。

## 4. 上下文任务面板

从发布中心发起的首次设置、测试验收、故障恢复和正式发布，默认使用基于现有 Radix 组件的右侧 `Sheet` 原地展开，不切换项目导航，也不把后台流水线阶段伪装成新页面。关闭面板后，发布中心的任务、环境和滚动位置仍在原处。

Sheet 内的可取消编辑必须使用独立 draft。输入事件不得直接改写 EnvironmentBinding、Connection 或 Task 快照；只有用户点击保存并通过校验后，draft 才提升为权威状态，取消与 Escape 都只丢弃 draft。

重新进入应用或普通点击项目时统一回到“发布中心”。只有用户从“待处理”入口点击某条明确任务时，才允许直接打开对应任务面板；“本机运行 / 版本 / 项目设置”只是浏览位置，不作为重启后的恢复目标。

直接从待处理通知恢复历史停点时，可以打开同一面板并携带任务、环境和版本 ID；旧版 `test / production` 路由只做迁移兼容，不再作为新交互入口。

测试任务一旦创建，页面技术详情可以展示其完整 commit，但不得再读取当前 HEAD 覆盖目标。后续继续或重试生成的 Attempt 必须显示并使用 Task 创建时冻结的 commit。

任务面板按当前状态只展开一块。首次任务先用紧凑清单展示完整剩余计划与已满足项，让用户知道总量；随后只展开当前一个用户问题。全新账号最多涉及“代码更新授权 / 版本保存位置 / 测试运行位置 / 必要配置”四项，但这不是固定四步：全部复用为零问，部分复用只问缺项，多个候选只问一次选择。不得按 CNB、Registry、SSH、Caddy 的技术顺序把资源编辑器当成工作流：

- 尚未准备：按当前真实缺口展示“允许自动获取更新 / 连接版本保存位置 / 选择运行位置 / 补齐项目运行信息”，Provider 名称只作为来源说明；
- 自动处理中：使用“读取项目 / 生成版本 / 启动服务 / 检查地址”的进度、可离开说明和最近更新时间；
- 等待业务验证：版本、地址、打开与确认动作；
- 等待正式确认：候选、目标和风险摘要；
- 失败：已完成事实、原因、唯一恢复动作；
- 已部署但地址未完成：同时说明当前仍在线的旧地址与待切换的新地址，只处理路由/DNS/HTTPS。

不得复制第二套代码平台、Registry、服务器或配置设置表单。面板复用同一领域编辑器，但用当前任务的用户目的包装；保存后刷新发布快照并自动切换到下一个真实问题。首次任务的最后一项保存后直接进入“正在生成测试版”，不再增加第二次生成确认。面板仍然只有一个主动作，并通过标准关闭按钮、`Escape` 和焦点管理保证可访问性。

## 5. 版本页

版本列表只收录不可变版本，并按生成时间倒序。一级信息：用户可识别的时间/名称、测试资格、当前测试、当前正式和可执行动作。完整提交、构建号，以及每个服务的不可变 Registry 引用/digest 集合放在技术详情；不得只展示一个代表性 digest 让多服务版本看似完整。

动作规则：

- 测试通过且未在线：发布到正式环境；
- 当前正式版本：只说明“当前在线”，不提供重复发布；
- 较早测试通过版本：恢复到正式环境；
- 等待确认版本：前往测试验证；
- 失败记录：默认折叠在最近活动，不计入可发布版本数量。

选择候选时，“当前测试版”和“已选择”是两个独立标签。当前测试版默认推荐，但用户改选后不能仍表现为被选择。

## 6. 项目设置

设置按用户认识的结果分组，而非按流水线步骤：

- 云端自动更新；
- 测试版运行位置；
- 正式版运行位置；
- 测试与正式配置；
- 测试与正式访问地址。

每项显示用户用途、连接来源、验证状态、被哪个环境使用，以及“更换/重新授权/解除绑定”。CNB、TCR、SSH、Docker 和 Caddy 进入连接详情。只有一个候选时可以推荐复用但保留更换入口；多个候选时要求选择。新增服务器首次绑定使用同一条渐进流程：凭据 → 指纹 → 运行前提 → 共享访问服务分类 → 影响确认 → 验证 → 绑定；任何一步未完成都不能提前显示为环境已绑定。

代码平台卡表达账号级授权，不把仓库名混进 Connection；当前项目的精确仓库 SourceBinding 放在项目设置中核对。共享账号连接不得让第二项目继承第一项目的仓库、版本或自动化规则。添加、切换或恢复项目时界面只显示“账号可用，当前仓库待核对”，不能直接点亮“仓库已验证”；用户发起需要仓库的动作时自动做只读核对，成功后继续，不额外增加一步。

发布中心打开设置面板时定位当前阻塞项，但不改变项目导航，也不自动滚动抢走用户之后的手动位置。首次接入由当前任务带入相应资源；项目设置不是要求用户主动遍历的黄金路径。

## 7. 本机运行

按“项目服务 → 运行依赖 → 必要配置”通栏纵向排列。服务与依赖数量不同，不能强行放入对齐网格。详细规则见 [本机运行](local-running.md)。

## 8. 状态语言

状态必须是事实 + 对象：

| 避免 | 使用 |
| --- | --- |
| 需要处理 | 正式版已部署，还差设置地址 |
| 失败 | 新版本启动失败，当前正式版仍在线 |
| 等待部署 | `main` 更新后会自动部署测试版 |
| 刷新状态 | 检查并继续 / 检查并完成发布 |
| TCR 登录 | 连接版本保存位置（说明中再写腾讯云 TCR） |
| 准备服务器 | 选择测试版运行的位置 |
| 配置 Caddy | 设置访问地址 |
| 启动流水线 | 生成测试版 |
| 地址已修改 | 新地址正在检查，旧地址仍可访问 |
| 需要重新授权 | 当前账号还不能读取这个项目（先重新核对；用户选择更换账号后才显示授权表单） |

状态不只依赖绿、黄、红；同时使用文字、图标和可访问名称。刷新即使无变化也保留最近检查时间。

## 9. 操作层级

- 一个可见状态只突出一个主按钮。
- 次要操作使用次按钮或文字链接；危险操作单独确认。
- 按钮写结果和对象，如“发布 7 月 15 日版本”“更新 CNB 授权”，不写模糊的“确定”“下一步”。
- 长任务开始后显示正在做什么、可否离开和取消后果。
- 前置条件未完成时解释原因并给入口，不用不可点击按钮让用户猜。

## 10. 表单与配置

- 先展示缺值必填项，再展示缺值可选项，已有值折叠。
- 项目注释为主标题，Key 为副标题；不自行翻译。
- 选择配置中心值时显示说明、Key、绑定环境和来源，不回显敏感 Value。
- 完整性按精确必填 Key 展示；不能因为有相同数量的其他配置就显示“已齐全”。
- 没有匹配配置时在当前字段原地提供“添加常用配置”；手填值可直接“保存为常用配置”，保存成功后绑定当前环境，不跳离任务页。
- 配置保存后只表达一次成功结果，并说明保存/同步位置。
- 配置中心编辑字段仍只有“说明、名称、值”；详情另外展示正在使用它的项目与环境，不把使用关系变成第四个必填字段。
- 已绑定条目的 Key 只读；要更名时引导新建正确 Key，再精确解除当前项目 + 环境绑定，其他引用保持不变。
- 修改已被引用的值时明确区分“仅保存”和“保存并应用到当前环境”，并显示 Profile revision 与环境当前使用的 revision。“仅保存”不改变运行环境，正式环境应用需要生产确认。
- 显式应用配置时只能推荐一个明确的项目 + 环境目标，并创建配置 Task；成功前 applied revision 不变，失败从同一配置 Task 继续。无 revision 变化时不显示会创建任务的主动作；解除引用只作用于当前选中的项目 + 环境。
- 配置条目的“被哪里使用”从各项目 EnvironmentBinding 动态派生，不能读取 ConfigProfile 自带的反向字符串。执行中绑定发生变化时停止完成并要求重新确认，不用旧 Attempt 静默推进。
- 连接错误必须区分“账号授权失效”和“当前项目仓库尚未验证”。前者在原表单替换授权，后者先提供只读重新核对、打开平台权限和更换账号三个动作；核对失败不得立即展示 Token，也不得要求移除项目。
- Secret 默认遮盖，复制和显示均为显式动作。

## 11. 错误与恢复

错误卡固定顺序：

1. 用户结论；
2. 已完成或仍在线的事实；
3. 唯一恢复动作；
4. 默认折叠的脱敏技术详情。

第三方原始 JSON、Shell 输出、仓库完整路径、私钥、Token 和整段日志不得进入一级页面。授权、Registry、服务器、DNS 和 HTTPS 分别提供对应动作，不能都退化为“刷新状态”。

production 启动、缺配置、服务器断连和路由冲突都展示原 Task 编号、旧正式版本和旧地址，并复用配置中心或连接维护入口。完成修复后只追加 Attempt，不能把用户带到新的发布线；只有另一已知 ABCDeploy 项目占用地址时显示“转移地址”。旧服务、未知来源、非 Caddy 或配置入口不明时只允许更换地址或交给维护人员，不显示自动接管。

版本页以 Version 为提交、产物和来源的唯一记录；“测试通过”等 Validation 只决定资格，不在页面另存一套产物来源。production 任务建立后，页面后来改选版本不得改变任务标题、复核或完成结果。

AutomationRule 维护页同时表达期望状态、Provider 实际状态和同步错误。可见主动作按状态只显示“暂停”“恢复”或“修复并重新同步”之一；这些动作不得制造部署 Task、版本或环境离线状态。写操作前若当前项目仓库证据缺失或过期，系统先自动只读核对；失败时四类规则事实全部保持原样，页面说明“尚未修改自动更新规则”。

## 12. 渐进披露

普通用户只看结果、影响和下一步。以下内容默认折叠：Provider 名词解释、分支映射、Docker/Compose/Caddy、构建编号、commit、digest、文件路径、端口清单和原始日志。首次任务首屏不得仅靠折叠隐藏这些词；主标题和流程结构本身也必须与技术实现解耦。

折叠不是隐藏阻塞原因。用户完成动作所需的服务器 IP、DNS 记录、权限名称和目标地址必须直接可见并可复制。

## 13. 布局与可访问性

- 页面主体单一垂直滚动，不嵌套多个无必要滚动区。
- 异步刷新不重置滚动位置、不抢焦点、不整页闪烁。
- `640×480` 使用单列和紧凑间距；`1280×720` 首屏能看到环境结论与下一步；宽屏限制内容行长。
- 键盘可完成所有操作；焦点可见；Label 与输入关联；图标按钮具有包含对象的可访问名称。
- 支持增强对比度、强制颜色和减少动画；部署状态不以动画为唯一反馈。

## 14. 页面验收问题

逐屏必须能回答：

1. 用户是否知道自己在哪个项目、哪个环境？
2. 是否能区分在线版本与最近尝试？
3. 是否只出现一个明确下一步？
4. 是否知道为什么必须由自己处理？
5. 返回或重启后能否继续，同时普通项目入口仍回发布中心？
6. 是否复用了现有组件，而非新增一次性 UI？
7. 新项目是否先承诺一个可打开的测试结果，而不是要求用户搭建资源？
8. 用户是否从一开始知道还需要配合几类事项，同时当前只看到一个问题？
9. 页面是否把 CNB、TCR、SSH、Docker、Caddy 等词限制在授权来源或技术详情？
10. 最后一项信息保存后是否直接开始任务，没有重复的“准备完成 → 再确认生成”？
11. 等待用户输入时是否没有伪造 Attempt，执行详情是否能还原当次版本、连接、服务器、配置和地址快照？
12. 修改地址时是否明确区分旧的当前地址与新的待切换地址，并保证检查完成前旧地址仍在线？
13. 多项目切换、普通重载和显式 deep-link 是否分别表达了打开项目、回发布中心和指定页面的真实意图？
14. 配置仅保存、显式应用、精确解绑和无变化是否分别产生正确的 revision 与 Task 结果？
15. 自动化暂停、恢复和修复是否只更新规则状态，不伪造部署执行？
