# ABCDeploy V2 推荐部署策略

## 1. 目标

推荐策略需要同时满足三个条件：

1. 小白无需理解完整 CI/CD 模型。
2. 测试通过的版本才能进入生产。
3. 用户已有分支和基础设施时，不强行改造成固定模板。

## 2. 默认发布模型

### 新项目推荐

```text
短期 feature/* 分支合并到 main
  -> 构建一次不可变镜像
  -> 上传所选 OCI 镜像仓库（国内默认 TCR）
  -> 自动部署 staging
  -> 健康检查与候选标记
  -> 用户在 CNB 页面或 ABCDeploy 确认发布
  -> 同一镜像摘要部署 production
```

- 新项目只保留 `main` 一条长期分支；AI 开发使用合并后删除的短期 `feature/*` 分支。
- 不为 development、staging 和 production 额外创建长期环境分支。
- `main` 是稳定代码和发布候选来源，不等于跳过测试直接替换 production。
- CNB 原生部署环境负责 production 权限和审批，不使用 `production` 分支模拟环境；staging 验证成功后创建候选标签。
- production 默认必须人工确认；完成至少一次成功发布后才允许开启自动晋级。
- production 永远复用 staging 验证过的镜像摘要，不重新构建。

### 兼容现有项目

- 已存在 `test` 或 `staging` 分支且有部署历史时，建议继续作为 staging 来源。
- 已存在可靠流水线时，只生成差异和迁移建议，不直接覆盖。
- 发现 `main` 当前直接部署 production 时，先保留行为并建议启用候选验证，不静默改变上线规则。

## 3. 三种用户方案

### 推荐方案

适合绝大多数个人产品和小团队。

| 项目       | 默认值                     |
| ---------- | -------------------------- |
| 分支       | `main` + 短期 `feature/*`  |
| staging    | `main` 更新后自动部署      |
| production | 人工晋级同一镜像摘要       |
| 发布入口   | CNB 原生部署页或 ABCDeploy |
| 镜像库     | TCR 国内镜像加速           |
| 反向代理   | Caddy                      |
| HTTPS      | 域名就绪后自动签发         |
| 健康检查   | 自动识别或生成             |
| 回滚       | 健康失败自动回滚           |
| 备份       | 数据迁移前默认备份         |
| 服务器     | 起步可共用，环境严格隔离   |

### 快速上线方案

适合无数据库的静态站点、演示项目和短期活动页。

- 默认分支更新后自动部署。
- 健康检查通过后可以自动晋级 production。
- 仍保留不可变镜像和一键回滚。
- 检测到数据库、迁移脚本或持久化卷时，不推荐此方案。

### 现有架构方案

适合已有服务器、分支、镜像库和发布规则的用户。

- 系统先导入并解释现状。
- 用户选择保留、逐项迁移或仅使用监控能力。
- 所有变更先生成 Plan 和影响说明。

## 4. 环境默认值

### development

- 运行在本机。
- 使用热更新和本地共享基础设施。
- 不因代码保存触发远程部署。
- 不进入首次云端部署必填流程。

### staging

- 首次远程部署目标。
- 自动部署发布候选。
- 使用独立容器命名空间、数据库、缓存前缀和运行变量。
- 域名可选，允许使用受保护的临时访问方式完成验证。

### production

- 首次 staging 健康前保持锁定。
- 默认人工晋级。
- 使用独立服务器为正式推荐；一台服务器仅作为起步方案。
- 使用独立数据库、密钥、持久化目录和发布记录。

## 5. 资源默认值

### 构建与镜像

- CNB 负责构建和记录状态。
- 国内轻量服务器默认推荐 TCR，普通界面称为“国内镜像加速”；账号和命名空间作为全局资源只配置一次。
- CNB Docker 制品库作为无需额外镜像账号的兼容回退；其他 Registry 和自建仓库位于高级设置。
- 所有镜像以 commit 唯一标识，并在部署时转换为 digest 引用。
- 构建端 push 地址与服务器 pull 地址分别配置，支持同一 Registry 的公网、内网或跨地域入口。
- staging 与 production 必须拉取同一 digest，生产流水线禁止再次构建。

### 服务器

- 优先复用已经验证且资源充足的服务器。
- 自动生成 Compose Project、容器、网络和目录名称。
- 检测 80/443、磁盘、内存、Docker 版本和现有反向代理冲突。
- 不自动清理未知容器和目录。

### Caddy

- Caddy 是默认反向代理和 HTTPS 入口。
- 已存在 Nginx、Caddy、Traefik 或端口占用时停止初始化并解释冲突。
- 项目只维护自己的路由片段，不覆盖其他项目配置。
- 同一服务器上的项目可并行构建，Compose 切换和 Caddy 热加载由服务器级锁自动排队。

## 6. 分支推断规则

按以下顺序推断：

1. 读取现有 `deploy.yaml` 和部署记录。
2. 读取远程默认分支。
3. 检测 `staging`、`test`、`develop`、`dev` 和 `production` 分支是否存在并近期活跃。
4. 检测现有 CI/CD 文件中的环境绑定。
5. 新项目推荐 `main + 短期 feature/* + 环境晋级`；已有项目生成兼容或迁移建议，但不自动创建、删除或重新绑定分支。

用户主界面只看到：

> 开发任务合并后先进入测试环境；确认测试结果后再发布正式版本。

具体分支映射位于高级详情。

## 7. 环境变量策略

### 自动完成

- 根据服务生成内部 URL 和端口。
- 生成 JWT Secret、Session Secret、Webhook Secret 和部署 Token。
- 根据项目与环境生成数据库名称、用户名建议和 Redis 前缀。
- 为前端生成对应环境的 API Base URL。

### 需要用户提供

- 第三方 API Key。
- 已有数据库或存储服务的凭证。
- 支付、短信、邮件、地图等业务平台凭证。
- 无法从项目判断的业务配置。

### 展示规则

- 默认显示中文业务名称，不先显示变量名。
- 变量名、来源文件和注入位置放在详情中。
- 相同值跨环境复用时必须显式确认，敏感生产值默认不复用。

## 8. 域名策略

- staging 首次部署不因域名缺失而阻塞。
- production 发布前要求确认域名或明确选择仅 IP/内网访问。
- 系统生成精确 DNS 记录并持续检测生效状态。
- DNS 未生效时状态为“部署成功，公网入口待完成”，不能标记部署失败。
- Caddy 只有在 DNS 指向正确且端口可达后才进入证书验证。

## 9. 数据库策略

- 检测到 Prisma、TypeORM、Sequelize 或数据库变量时才展示数据库步骤。
- 默认推荐 staging 与 production 使用不同实例或至少不同数据库。
- 生产迁移前要求可用备份；无法验证备份时阻止自动迁移。
- 首次部署不自动迁移历史数据库数据，必须作为独立任务确认。

## 10. 风险分级

| 级别       | 示例                            | 默认行为         |
| ---------- | ------------------------------- | ---------------- |
| 只读       | 扫描项目、检查 DNS、验证连接    | 自动执行         |
| 低风险写入 | 生成本地配置、创建私有仓库      | 展示摘要后确认   |
| 服务器变更 | 安装 Docker/Caddy、写入运行配置 | 单独确认并记录   |
| 生产变更   | 发布、迁移、回滚                | 强确认和审计     |
| 破坏性操作 | 删除容器、目录、数据库、DNS     | V2 P0 不自动执行 |

## 11. 推荐理由的表达

每项推荐最多一句话：

- “正式发布默认需要确认，避免一次提交直接影响真实用户。”
- “优先使用 CNB 制品库，可以少配置一套镜像账号。”
- “当前只有一台服务器，测试和正式会被严格隔离；用户量增长后建议拆分。”
- “检测到数据库迁移，发布前将先验证备份。”

不要在主流程讲解 Docker、Compose、digest 或反向代理原理；这些内容放在“为什么这样做”中。
