# DeployDesk V2 可用性测试与验收方案

> 状态：待执行。研发启动前先用可点击原型测试，P0 完成后使用隔离环境做第二轮端到端测试。

## 1. 测试目标

验证目标用户能否在没有旁人代操作、没有终端命令和没有预先学习 DevOps 的情况下：

1. 找回以前配置过的项目。
2. 为新项目连接 CNB 和服务器。
3. 接受或调整推荐部署方案。
4. 完成首次 staging 部署。
5. 理解 staging 与 production 的关系。
6. 从常见错误中恢复，而不是放弃或破坏已有服务。

测试不以“用户最终点到了成功页面”为唯一标准，还要观察犹豫、误解、求助和不安全操作。

## 2. 参与者

首轮招募 10 人，最低有效样本为 8 人：

- 至少 6 人是使用 AI 写过项目、但没有独立配置过 CI/CD 的真实小白。
- 2 人有 Docker 或云服务器基础经验。
- 最多 2 人是开发或运维从业者，用于发现专业性缺口，不计入小白成功率。
- 不招募直接参与 DeployDesk 设计和研发的人。

每位参与者测试 45 至 60 分钟。测试前记录操作系统、项目经验、是否用过 Git、Docker、云服务器和 CNB。

## 3. 测试环境

- 准备独立 CNB 测试账号或最小权限测试组织，不使用参与者真实生产账号。
- 每人分配可重置的 Ubuntu 测试服务器和测试域名。
- 准备一个结构清晰但没有部署配置的示例项目。
- 准备一个已有 `test` 分支和现有部署规则的兼容性项目。
- 所有测试凭据在测试后自动撤销，服务器恢复快照。
- 屏幕和声音仅在参与者书面同意后录制，日志在分享前脱敏。

## 4. 主持原则

主持人使用思考出声法，但不解释界面、不提示按钮位置、不代替参与者操作。用户提出“我该点哪个”时，先反问“你现在期望发生什么”，记录其心智模型；若任务无法继续，再提供分级提示。

提示分级：

| 等级 | 定义                   | 是否算独立成功 |
| ---- | ---------------------- | -------------- |
| H0   | 无提示完成             | 是             |
| H1   | 只重述任务目标         | 是，但记录犹豫 |
| H2   | 指出界面区域或解释概念 | 否             |
| H3   | 告知具体操作或代操作   | 否             |

## 5. 测试任务

### T1：返回已有项目

场景：昨天已经完成项目识别，今天重新打开 DeployDesk。

成功条件：参与者无需重新选择目录，能进入项目并说出测试环境当前状态。

### T2：部署第一个新项目

场景：把本机示例项目部署到一台新服务器的测试环境，暂不发布生产。

成功条件：完成项目识别、CNB 授权、服务器连接、推荐方案确认和首次部署；全程不打开终端、不编辑 YAML、不寻找 SSH 私钥路径。

### T3：没有 SSH Key

场景：电脑没有可用 SSH Key，但已知服务器账号密码。

成功条件：参与者理解密码只用于首次连接，并能让 DeployDesk 创建后续安全连接；不会尝试把密码写进项目配置。

### T4：处理 CNB 权限不足

场景：授权账号缺少创建仓库或流水线配置的权限。

成功条件：参与者能判断需要切换账号还是申请权限，能保留当前进度，并且不会重复创建资源。

### T5：处理 DNS 未生效

场景：应用已运行，但域名解析错误。

成功条件：参与者能说出哪些工作已完成、自己需要修改什么、何时会自动复查，并能安全离开后继续。

### T6：理解生产晋级

场景：staging 已验证通过，现在决定是否发布 production。

成功条件：参与者理解 production 使用已测试的同一镜像制品，但域名、数据库和配置独立；不会认为生产在直接使用测试数据库。

### T7：添加第二个项目

场景：使用同一个 CNB 账号和同一台服务器部署另一个项目。

成功条件：系统自动复用已连接资源，参与者只确认资源选择和新项目缺失信息。

## 6. 量化指标

| 指标                                  | 原型放行线  | P0 发布线                       |
| ------------------------------------- | ----------- | ------------------------------- |
| 小白首次 staging 独立完成率           | 至少 8/10   | 至少 9/10                       |
| 首次 staging 中位完成时间             | 10 分钟内   | 10 分钟内，不含云端实际构建等待 |
| 用户需要做的实质决策                  | 不超过 5 个 | 不超过 5 个                     |
| 返回项目成功率                        | 100%        | 100%                            |
| 第二项目中位配置时间                  | 3 分钟内    | 3 分钟内                        |
| 打开终端、YAML 或手找 Key 路径        | 0 人        | 0 人                            |
| 误把 staging 数据当作 production 数据 | 0 人        | 0 人                            |
| 首次流程打开高级设置比例              | 低于 10%    | 低于 10%                        |
| SUS 可用性量表                        | 80 分以上   | 85 分以上                       |

时间指标同时记录“用户操作时间”和“系统等待时间”，不得用加载等待掩盖交互复杂度。

## 7. 观察记录

每个任务记录：

- 开始与结束时间。
- H0 至 H3 提示等级。
- 首次点击是否正确。
- 明显停顿超过 10 秒的位置。
- 用户原话和预期结果。
- 是否理解当前环境、外部修改和回滚范围。
- 错误恢复结果。
- 主观信心，1 至 5 分。

禁止只记录“通过/失败”。录屏中的敏感信息必须遮挡，原始录像按约定期限删除。

## 8. 问题分级

| 级别    | 定义                                                   | 处理规则                     |
| ------- | ------------------------------------------------------ | ---------------------------- |
| P0 阻断 | 导致大多数用户无法完成，或可能泄密、误删数据、误发生产 | 停止研发或发布，先修复并复测 |
| P1 严重 | 需要 H2/H3 提示，或关键概念被普遍误解                  | 本轮必须修复并复测           |
| P2 一般 | 能完成但明显犹豫、绕路或文案误导                       | 排入当前里程碑               |
| P3 轻微 | 不影响任务的视觉或措辞问题                             | 记录并择期修复               |

相同问题在 2 名参与者中出现即形成模式；涉及安全或生产发布的问题出现 1 次即按模式处理。

## 9. 每轮测试节奏

1. 先用 2 名参与者做试测，修正测试脚本本身。
2. 使用同一版本测试 5 人，不在中途偷偷修改原型。
3. 汇总问题、原因和证据，确定下一版假设。
4. 修复 P0、P1 和高频 P2。
5. 使用新的目标用户复测，不让原参与者代替新用户。
6. 达到放行线后冻结黄金路径，再进入 P0 研发。

## 10. 人工配合清单

以下工作无法由开发工具替代，必须由产品负责人或测试负责人完成：

| 需要人工提供         | 获取方式                                            | 占位符                              |
| -------------------- | --------------------------------------------------- | ----------------------------------- |
| 8 至 10 名目标参与者 | 从真实 vibe coding 用户中招募，避免全部来自开发团队 | `TEST_PARTICIPANTS=`                |
| 录屏与隐私同意       | 使用评审通过的中文同意书，测试开始前签署            | `CONSENT_TEMPLATE_URL=`             |
| CNB 隔离测试组织     | 新建不含生产仓库和秘密的测试组织                    | `CNB_TEST_ORG=`                     |
| 可重置云服务器       | 使用快照或 Terraform 每人重建，禁止连接现有生产机   | `TEST_SERVER_POOL=`                 |
| 测试域名             | 创建独立子域名并允许自动改 DNS                      | `TEST_DOMAIN_ZONE=`                 |
| 主持人与观察员       | 主持人不参与当前界面设计，观察员负责时间与问题记录  | `TEST_MODERATOR=`、`TEST_OBSERVER=` |

## 11. 测试报告模板

每轮报告至少包含：

1. 版本、日期、参与者构成和测试环境。
2. 各任务完成率、耗时、提示等级和信心分。
3. P0 至 P3 问题清单及对应录像时间点。
4. 事实、推断和设计假设分开记录。
5. 本轮决定修改和明确不修改的内容。
6. 下一轮复测范围与放行结论。

只有达到量化放行线且没有未解决的 P0、P1，才可把“首次部署黄金路径”状态改为研发已放行。
