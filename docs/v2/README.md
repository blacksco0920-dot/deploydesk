# DeployDesk V2 产品定义

> 状态：设计评审中。V2 应用代码冻结，未通过本文定义的设计门禁前不进入研发。

DeployDesk V2 的目标不是让用户“能够配置部署”，而是让没有 DevOps 经验的 vibe coding 用户，在理解少量必要概念的前提下，可靠地完成首次上线并持续管理多个项目。

## 产品承诺

用户只需要回答业务问题，例如“这是测试还是正式上线”“域名是什么”“这个 API Key 从哪里获得”。Docker、Caddy、镜像摘要、SSH 私钥路径、环境隔离和流水线文件由系统负责选择、生成和验证。

## 设计原则

1. **先给推荐答案**：系统先生成可执行方案，用户只处理无法推断的信息。
2. **任务流优先**：首次部署使用线性向导，不把后台菜单当作操作步骤。
3. **状态可恢复**：项目、进度和错误处理在重启后继续，不要求重新开始。
4. **连接可复用**：CNB 账号、服务器和镜像库属于全局资源，可供多个项目使用。
5. **安全默认开启**：最小权限、系统密钥库、不可变镜像、健康检查、备份和回滚默认启用。
6. **渐进式披露**：普通用户不看 YAML、文件路径和原始日志；高级信息随时可展开。
7. **每次只解决一个问题**：失败页面必须说明发生了什么、系统做过什么、用户下一步做什么。

## 竞品启发

- [Vercel](https://vercel.com/docs/git) 将 Git 仓库连接、框架识别和默认构建策略放在前面。
- [Railway](https://docs.railway.com/quick-start) 允许用户直接部署，或只在缺少变量时进入补充流程。
- [Zeabur](https://zeabur.com/docs/en-US/get-started/quick-start) 自动识别语言、框架和构建方案。
- [Coolify](https://coolify.io/docs/knowledge-base/server/introduction) 可以在用户没有 SSH Key 时生成专用密钥。

DeployDesk 不复制任何单一产品，而是把这些低门槛模式应用到“用户自有服务器 + CNB + Caddy”的国内部署场景。
