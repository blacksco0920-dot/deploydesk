# ABCDeploy 产品实施合同

> 状态：工作流画布版本的产品实施合同。对齐日期：2026-07-19。本文是当前面向用户的产品设计与实现验收的唯一权威；它规定应当实现的产品行为，不表示每一条已经完成。真实完成度和明确缺口以[底层技术能力](technical-capabilities.md)为准。其他涉及“发布中心、测试环境、正式环境、版本管理”的旧产品文档只保留历史背景，不得覆盖本文。

## 1. 用户目标

ABCDeploy 面向使用 AI 完成项目、但不了解部署概念的个人和小团队。用户的目标只有一个：

> 把电脑里的项目可靠地放到服务器，并得到一个可以打开的地址。

Docker、CNB、TCR、Git、镜像摘要、SSH、Compose 和 Caddy 都是实现这一结果的底层能力，不是用户必须理解的工作流。

## 2. 唯一主模型

每个项目首页直接展示一张可视化部署工作流。默认“小白上线模板”自动生成四类基础节点：

```text
代码来源 ──> 版本构建 ──> 版本存储 ──> 部署运行
```

工作流使用以下稳定概念，页面和实现不得混用：

| 概念 | 定义 | 首版示例 |
| --- | --- | --- |
| `NodeType` | 一种稳定、可重复实例化的部署能力 | 代码来源、版本构建、版本存储、部署运行 |
| `Provider` | 某类能力的具体实现者 | 本地项目、CNB、腾讯云 TCR、Linux 服务器 |
| `Connection` | 可复用、可验证、可替换的授权或服务器身份 | CNB 账号、TCR 登录、SSH 服务器连接 |
| `NodeInstance` | 某条工作流中对一个 NodeType 的一次使用，绑定 Provider、Connection 和线路配置 | “上线”线路中使用“我的 CNB 账号”的版本构建节点 |
| `Workflow` | NodeInstance 与连线组成的一条部署线路 | 上线、测试服务器、演示服务器 |
| `WorkflowTemplate` | 用于快速生成完整线路的节点模板 | 小白上线模板 |

- 四个 NodeType 的固定名称是“代码来源、版本构建、版本存储、部署运行”；状态单独显示，不能把名称和状态混为一谈。
- CNB、腾讯云 TCR 和 Linux 不是节点类型，而是 Provider；Token、Registry 密码和 SSH 身份不是节点，而是 Connection。
- 授权不是第五个节点。需要授权时，节点选择或新建一条 Connection；连接正常时只展示名称和验证事实，不展示秘密。
- Provider 和 Connection 作为节点小字辅助信息，例如“版本构建”下显示“CNB · 我的 CNB 账号”。
- 首页不再以“发布中心、测试版、正式版、版本管理”组织用户任务。
- 进入工作流编辑器后隐藏应用级侧栏，只保留线路与节点资源栏；顶部“返回 + 项目名”负责退出编辑器，两块完整左栏不得并存。
- 工作流直接使用 FlowGram 承担画布、节点、端口、连线、缩放、平移和历史能力，以 Coze Studio 的工作区结构与交互习惯为基线，并在可独立复用处使用 Semi Design。Coze Studio 没有作为完整业务前端导入；ABCDeploy 仍负责部署节点语义、Connection 绑定、任务、证据和运行状态。
- NodeType 可以被不同 Workflow 重复实例化，Connection 可以跨项目引用；NodeInstance 的配置、状态和运行证据不能跨线路复用。
- 系统根据模板生成和自动排布默认线路。首版不要求小白从空画布搭建流程；画布首先是部署事实的可视化，其次才是可编辑工作区。
- 本机运行、历史版本、运行记录和技术证据应使用当前项目内的独立次级入口，不混入节点配置，也不占用应用一级导航。其中运行记录入口已接入当前工作区；项目级“本机运行”次级入口仍未重新接入。

## 3. 部署线路

- 添加新项目后，系统自动生成一条“上线”线路并从本地项目节点开始检查，名称可修改。
- 用户主动删除最后一条线路后可以回到无线路状态，此时首页主动作是“创建上线线路”；重新进入项目不会擅自恢复已删除线路。
- 如果用户还要部署测试服务器、演示服务器或其他服务器，就新增一条独立线路。
- 默认线路包含完整四类节点，直接从当前代码来源到一个明确的部署目标。
- 线路之间可以复用同一个构建服务、版本仓库或服务器连接，但状态和部署记录独立。
- 不存在强制的“测试通过后才能进入正式环境”页面流程；用户需要测试线路时，它就是另一条完整线路。
- 同一项目的多条线路可以在左侧线路列表中切换。新增测试、演示或其他服务器目标时，从模板新增一条完整线路，名称由用户自行区分。
- 首版每条线路固定且恰好包含四个 NodeInstance，不开放手工增删节点、连线或服务器分支。条件、循环、审批、通知等通用自动化节点不进入首版。

截至 2026-07-19，线路创建、切换和删除已经进入桌面运行时；删除最后一条线路后会保持无线路状态。这里的“固定四节点”只限制线路内部编排，不表示整条线路不能删除。

## 4. 节点配置

首次设置时，系统自动检查四类节点，只展开距离代码来源最近的未就绪节点：

1. 代码来源：选择 Provider，并识别项目、服务和当前内容；首版默认是本地项目。
2. 版本构建：选择 Provider 与 Connection；首版默认是 CNB。
3. 版本存储：选择 Provider 与 Connection；首版默认是腾讯云 TCR。
4. 部署运行：选择 Provider 与 Connection，并维护当前线路的运行配置和访问地址；首版默认是 Linux 服务器。

完成首次设置后，四个节点始终可以独立点击、查看、验证和修改。某个节点失效时只修复该节点，不让用户重走整条线路。

当前桌面实现已经按上述顺序打开新线路最近的未就绪节点。画布节点同时展示 Connection 名称和文字状态；运行中的节点与连线状态来自持久化任务的 `status/currentStage`，不由节点位置或旧的 ready 值推断。

节点抽屉一次只能处于一种明确模式：

1. **查看**：只展示当前生效配置、验证时间和“更换”动作，不出现可编辑输入框；
2. **选择**：只展示配置中心已有候选和“添加新的”，不同时展示当前绿色摘要与新增表单；
3. **新增/编辑**：只展示当前草稿表单、保存去向和取消/验证动作；
4. **修复**：只展示问题、仍然有效的事实和一个恢复动作，技术详情默认折叠。

从“查看”进入“更换”，再进入“选择”；只有点击“添加新的”才进入表单。返回上一级不改变当前生效绑定。

当前 Inspector 支持用 `Escape` 关闭并把焦点还给触发节点；打开或关闭 Inspector 只改变可用画布尺寸，不自动重新适应视图。顶部运行记录图标提供 Tooltip，运行记录正文展示真实任务阶段、服务器和路由检查事实，完整 commit、构建编号与镜像摘要收在折叠技术信息中。

有意修改节点时使用草稿：先验证新配置，验证成功后再替换当前生效配置；验证失败时旧配置和已在线服务继续有效。

节点与 Connection 的授权表现固定为：

- 未绑定：节点显示“待连接”，Inspector 提供“选择已有连接 / 连接新的”；
- 已绑定：节点显示“连接正常”，Inspector 只读展示 Provider、Connection 名称、最近验证以及“更换连接 / 重新验证”；
- 已失效：节点显示“授权失效”，Inspector 只展示仍有效事实、“重新授权 / 改用其他连接”和折叠技术详情；
- 新增连接：先选择 Provider，再填写该 Provider 所需的最少信息；验证成功后默认保存到配置中心，用户可以选择仅用于当前线路；
- Secret 只进入系统密钥库，节点、线路快照和普通日志只保存 Connection ID 与 revision。

## 5. 配置中心

首次新增代码平台、版本仓库或服务器连接时，界面提示：

> 保存到配置中心，其他项目以后可以直接使用。

可复用连接默认勾选保存。创建新线路或修改节点时，可以直接从配置中心选择已有连接。

以下信息默认不跨项目共享：

- 项目的运行环境变量和业务密钥；
- 项目域名与服务路由；
- 当前线路的部署状态、版本和执行记录。

配置中心保存 Connection 本身；项目和线路只保存 Connection 引用与 NodeInstance 绑定。替换共享连接凭据会推进 Connection revision 并展示影响范围；节点内“更换”默认只替换当前 NodeInstance 的引用。

上述 revision 与影响范围是产品合同，不是当前完成声明。当前配置中心已经读取并分别展示真实 `ConnectionResource` 与 `ConfigProfile`，但 CNB/TCR 仍各自只有首版默认连接；多连接及 revision 后端尚未闭环。

配置中心只存在于应用首页一级导航，不嵌入工作流资源栏或节点 Inspector。节点 Inspector 只允许选择、替换当前线路引用或创建一条可复用连接；批量修改、删除和影响范围必须返回应用首页进入“配置中心”。工作流内的连接候选只负责当前 NodeInstance 的绑定，不是另一套全局管理器。

## 6. 上线行为

- 四个节点全部就绪后，画布工具栏只突出一个主动作“开始上线”。
- 首次上线成功后，主动作变成“更新上线”。
- 新线路第一次点击上线时，只确认一次“本地项目 → 运行服务器 → 访问地址”；同一线路后续更新直接执行，不反复询问。
- 点击上线时使用用户本地项目在点击时的当前内容；不要求用户先理解 Git 提交。
- 用户明确点击前不自动上线。
- 远程任务开始后可以关闭客户端；重新打开后恢复同一任务和真实停点。画布中的节点和连接线显示当前执行位置，但不把底层阶段伪装成用户步骤。
- 重试同一目标时保留任务，只新增一次执行记录，不覆盖此前失败证据。
- 节点配置保存只改变线路绑定，不自动触发任何远程操作；只有“开始上线”“更新上线”“继续上线”会执行外部变更。

## 7. 成功与失败

“上线成功”必须同时有真实证据证明：

1. 项目构建成功；
2. 完整服务版本已经进入版本仓库并固定到不可变摘要；
3. 服务器上的完整服务集合正在运行且健康；
4. 前端、API 等跨服务访问链路能够通过真实入口连通；
5. 访问地址能够从公网打开，并返回预期项目响应。

构建、服务器运行和公网访问是三个独立事实。域名、证书或 443 失败时应显示“应用已部署，访问地址待处理”，不得重新构建，也不得冒充整个部署失败。

失败页只突出发生问题的节点及一个修复入口；原始提交、摘要、命令、日志和 Provider 错误默认折叠。

每次上线都生成一个覆盖全部服务的不可变版本。历史版本与每次运行记录进入独立“运行记录”视图；“版本仓库”节点只负责说明当前保存位置和连接状态。失败时服务器自动保留或恢复旧在线版本；用户可以从运行记录选择一个已验证历史版本重新上线，且不得重新构建或解析可变标签。

当前独立运行记录已经补齐真实任务关键事实；历史版本重上线仍按本合同作为完整验收项，不因记录界面已存在而自动视为所有交互闭环。

## 8. 安全和恢复红线

- Token、Registry 密码、SSH 私钥和业务密钥只进入系统密钥库或专用安全文件，不进入 Git、SQLite 明文、普通日志或截图。
- 项目内部签名、会话等内部密钥可以由客户端按项目与线路安全生成并复用；代码平台、版本仓库、AI 服务等外部凭据绝不自动伪造。
- 项目运行配置按线路保存，部署时只把服务声明的变量注入对应容器；服务之间不得因为共用一条线路而获得彼此的业务密钥。
- 首次 SSH 连接必须确认并固定主机指纹，后续严格校验。
- 服务器统一使用 Caddy；只管理当前线路拥有的独立路由片段，不覆盖未知主配置。
- 路由修改必须备份、校验、热加载，失败恢复原配置。
- 同一服务器的 Compose 切换和 Caddy 修改使用服务器级锁串行执行。
- 失败时旧在线版本继续运行；回退只切换到已知不可变版本，不删除数据库或持久卷。

## 9. 必须通过的产品状态

正式实现至少覆盖以下状态：

1. 没有部署线路；
2. 首次逐节点设置；
3. 所有节点已就绪，等待上线；
4. 正在上线，并可关闭客户端；
5. 上线成功，地址可打开；
6. 本地内容发生变化，可以更新上线；
7. 某个节点失效，可以单独修复；
8. 更换节点配置，旧配置在新草稿验证前继续生效；
9. 同一项目存在多条独立线路；
10. 复用配置中心的已有连接创建新线路。
11. 已就绪节点只展示摘要，主动更换时才进入候选选择；
12. 新增连接时只展示一套表单，保存后回到摘要；
13. 运行记录与版本历史不出现在节点配置抽屉中；
14. 画布在窄窗口可横向浏览，节点名称、状态和唯一下一步仍可读。

## 10. 防跑偏门禁

以下情况一律视为偏离产品合同：

- 用“测试环境/正式环境”重新组织一级页面或强制流程；
- 用旧的 staging/production 字段给多条线路换皮；
- 要求用户先进入版本页选择版本才能完成日常上线；
- 某节点出错时清空整条线路或要求删除项目重来；
- 页面根据步骤位置推断成功，而没有底层执行证据；
- 为了页面简单而合并“在线版本、最近失败、服务器健康、地址可达”等不同事实；
- 自动把用户本地未确认的变化上线；
- 把项目专属密钥当成同名通用配置跨项目复用。
- 在同一节点抽屉里同时展示“当前已连接”摘要和可编辑的新连接表单；
- 把版本历史、部署日志或运行记录塞进版本仓库配置；
- 把 CNB、TCR、Linux 或授权本身定义成新的 NodeType；
- 让首次用户为了完成默认四节点线路而必须从空画布手动拖拽、连线或配置技术阶段；
- 在同一个工作流工作区混用 Coze 式画布与 ABCDeploy 旧式大卡片、旧按钮、旧 Sheet 两套视觉语言。
- 在工作流编辑器中同时展示应用级侧栏和线路资源栏，挤压画布并制造两套导航层级。

## 11. 实施完成定义

实施完成不以静态原型或单元测试通过为准。必须同时满足：

当前仍未闭环的界面能力至少包括：CNB/TCR 多连接与 Connection revision 后端，以及项目级“本机运行”次级入口。配置中心已有对象展示、底层已有本机启停命令，均不能替代这两项端到端验收。

- 新产品首页和全部十四类状态可操作；
- 工作流画布、节点单一模式、配置中心复用和独立运行记录通过桌面端交互验收；
- 线路、节点引用、配置草稿、任务、执行和证据形成持久化闭环；
- 桌面端自动化测试和当前 macOS Apple Silicon `.app` 验收通过；
- `Documents` 下具备 `deploy.yaml` 的试点项目（明确排除 ECAT）逐个真实上线；
- 每个试点项目首页包含可目视识别且互不混淆的上线标识；
- 逐个核对构建、不可变镜像、服务器健康和公网首页标识四类证据。

当前工作流模型与跨服务链路的端到端证据见[2026-07-19 FinAgent 工作流试点验收](pilot-validation-finagent-2026-07-19.md)；此前的工作流证据见[2026-07-19 FinAgentCRM 工作流试点验收](pilot-validation-finagentcrm-2026-07-19.md)，历史八项目底层能力证据见[2026-07-18 真实项目上线验收](pilot-validation-2026-07-18.md)。
