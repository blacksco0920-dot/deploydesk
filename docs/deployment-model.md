# 分支、环境与镜像晋级

## `main` 的作用

`main` 表示稳定代码线和发布候选来源。它不等于某台服务器，也不等于生产环境本身。

ABCDeploy V2 默认只使用一个发布分支：

| 事件 | 构建 | 目标环境 |
| --- | --- | --- |
| `main` 更新 | 验证并构建一次不可变镜像 | 自动部署 `staging` |
| 测试健康 | 不重新构建 | 等待生产确认 |
| 用户确认生产 | 复用测试记录的完整提交 SHA 与镜像摘要 | 部署 `production` |

项目已有 `test` 或 `staging` 分支时可以继续保留用于代码协作，但不需要通过“一个分支对应一台服务器”来维持测试环境。环境是独立资源，版本通过部署记录晋级。

## “使用测试通过的同一镜像摘要”

它不是把测试服务器上的容器复制到生产，也不是让生产共用测试数据库。

1. `main` 提交生成带完整提交 SHA 的镜像标签。
2. staging 使用该镜像启动并等待容器健康。
3. ABCDeploy 保存测试部署的完整提交 SHA。
4. 用户点击“发布生产”时，CNB 精确检出这个 SHA。
5. production 使用相同镜像摘要启动，不执行第二次业务构建。

测试与生产仍使用各自的域名、变量、数据库、卷、网络、服务器和密钥文件。

## 为什么不重新构建生产镜像

即使源码相同，第二次构建也可能因为基础镜像、系统包、锁文件之外的依赖或构建时间差异产生不同内容。复用同一摘要可以证明生产运行的就是测试通过的制品。

`latest` 会随推送改变，无法提供这种证明，也无法可靠回滚。Manifest 校验会拒绝不含 `{commit}` 的标签模板。

## 失败与恢复

- 构建失败：不触碰测试和生产。
- staging 不健康：恢复上一份发布文件，生产按钮不出现。
- production 不健康：恢复发布前版本并保留失败记录。
- DNS 未解析：容器部署结果保留，界面只要求修复 DNS 后重新检查。
- HTTPS 未就绪：提示检查 443 和 Caddy 证书状态，不重复构建。
- 手动回滚：仅切换上一健康 `.release.env`，保留数据库、运行变量和 Caddy。

## 发布审批

生产默认人工确认。按钮只在以下条件同时满足时出现：

- 最新测试部署状态为成功。
- 测试记录包含完整提交 SHA。
- 当前没有已成功的同版本生产发布。

生产部署记录保存 `source_run_id`，因此可以追溯它来自哪一次测试验证。
