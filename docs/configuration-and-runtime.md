# ABCDeploy 配置与三环境运行模型

## 1. 三个环境

| 环境 | 用户入口 | 运行位置 | 主要用途 |
| --- | --- | --- | --- |
| `development` | 在本机运行 | 用户电脑 | 启停开发服务和检核本机配置 |
| `staging` | 发布中心的测试环境 | 测试服务器 | 自动接收 `main` 新版本并完成业务验证 |
| `production` | 发布中心的正式环境 | 正式服务器 | 人工选择测试通过版本发布或恢复 |

本机运行不是远程部署门禁。三个环境的配置、数据、容器命名空间、服务器绑定和部署记录相互独立。

## 2. 配置中心

配置中心是应用级复用资源，每条配置只有：

| 字段 | 含义 |
| --- | --- |
| 配置说明 | 用户用来区分用途或账号的名称 |
| 配置名称 | 项目配置 Key |
| 配置值 | 实际 Value；敏感值默认遮盖并安全保存 |

项目设置为每个环境维护 Profile 绑定。一个环境可以绑定多个 Profile，同一 Profile 也可以被多个项目或环境复用。`ConfigProfile` 只保存说明、Key、秘密引用和 revision，不保存反向项目/环境 bindings；`EnvironmentBinding.configProfileIds` 是使用关系的唯一事实源，配置中心的“被哪里使用”从所有项目工作区动态派生。

绑定是持续关系：再次打开项目时可以识别来源；配置中心条目更新后，界面应说明受影响的项目与环境，并在同步前确认。每个 Profile 保存 revision，环境绑定记录最近一次成功 Attempt 实际应用的 revision。“仅保存”只更新配置中心资源，不改变任何环境正在使用的值。只有用户显式选择唯一目标项目 + 环境后才创建配置 Task；Attempt 冻结该环境完整的 Profile ID/revision 映射。production 先确认影响，只有 Attempt 成功且完成时绑定集合仍与冻结输入一致，才用整份冻结映射替换 applied revisions；执行中改绑必须重新确认。Profile revision 与目标环境 applied revision 相同则不创建任务。删除 Profile 不得静默删除已经保存到运行目标的值。

## 3. 字段定义与展示

`.env.example` 是字段名、注释、顺序和默认值的权威来源。ABCDeploy 必须：

1. 保留注释、空行、未知字段和用户已有内容。
2. 有注释时以注释为主标题、Key 为副标题；没有注释时直接显示 Key。
3. 优先展示缺值必填项，再展示缺值可选项；已有值放到“查看全部”。
4. 只推荐 Key 匹配且作用域允许的配置中心条目，不翻译或猜测业务含义。
5. 允许直接填写，也允许通过绑定复用。
6. `DEPLOYDESK_*` 等内部字段保存但不出现在普通表单、缺失计数和配置中心候选中。
7. 没有用户字段时显示“无需填写，使用当前配置即可”，不制造空保存。
8. 完整性严格按项目声明的精确必填 Key 判断；相似名称、任意其他条目或数量不能替代缺失 Key。
9. Profile 一旦绑定，Key 身份只读；更名时新建正确 Key，并只解除指定项目 + 环境的旧绑定，不影响其他引用。

## 4. 合并规则

单个环境生成运行配置时按以下顺序合并：

1. 项目 `.env.example` 的结构和非敏感默认值；
2. 当前环境已经保存的项目值；
3. 当前环境绑定的 ConfigProfile；
4. 由 ABCDeploy 生成的内部运行值；
5. 用户本次明确修改的值。

同一 Key 出现多个候选且值不同时，不静默选择；页面让用户确认来源。`development` 的结果绝不自动写入 `staging` 或 `production`。

## 5. 项目设置中的配置状态

项目设置分别展示三个环境：

- 已绑定几项配置；
- 还缺几个项目要求的值；
- 最近一次保存或同步到哪里；
- 修改绑定、直接填写或解除绑定的入口。

发布中心只汇总阻塞状态，例如“测试配置还缺 2 项”。当前任务直接打开相应环境配置，允许原地引用或新增配置中心条目；项目设置只负责长期维护，不复制第二套配置表单。

## 6. Registry 与版本

Registry 是不可变版本保存位置，不是 production 独有设置。

- 第一次远程构建前完成并验证 Registry 连接。
- 构建端 push 地址与服务器 pull 地址可以不同，但必须解析到同一 Registry 内容。
- Version 以完整 commit 和每个已识别服务的不可变 Registry 引用/digest 标识；ArtifactRef 的 service ID 集合必须与 Project 精确相等，缺少、多出或重复任一服务时都不形成完整版本。
- staging 和 production 使用同一 Version 的完整服务产物集合；环境差异只来自运行配置、服务器和地址。
- Registry 凭据属于全局 Connection，项目只保存引用并能更换。

## 7. 服务器绑定

服务器 Connection 是应用级、环境无关的资源，不内置“测试服务器”或“正式服务器”身份。`staging` 和 `production` 各自通过 EnvironmentBinding 保存 Connection ID；项目可以选择同一台服务器，但系统仍分别验证环境绑定、容量、端口和隔离命名。

任何服务器第一次被任一环境绑定时必须顺序完成：验证凭据；发现或生成 ABCDeploy 专用 Ed25519 身份；确认并固定主机指纹；检查运行前提；分类共享访问服务；展示并确认影响；把 Connection 与 ServerCapability 标记为已验证；最后才建立环境绑定。一次性密码使用后立即清空，不落盘。连通后只读识别服务器能力，再按以下结果处理：

| 服务器现状 | 系统处理 |
| --- | --- |
| 干净且受支持 | Docker Engine/Compose 已可用且 80/443 无占用；展示将创建的运行目录和服务器级共享 Docker Caddy 容器，用户单独确认后幂等初始化 |
| 缺少运行前提 | Docker Engine、Compose 插件或当前用户权限缺失；停止初始化并给出一个明确补齐入口，不误判为干净服务器；裸机自动安装尚未闭环 |
| 已有兼容 Docker Caddy | 确认容器、配置入口、可写 `sites` 挂载和 reload 方式后复用；只追加当前项目 + 环境的路由片段 |
| systemd Caddy、其他代理、多个代理或无法确认归属 | 停止自动修改，保留只读诊断；更换运行位置或先由维护人员整理成可验证契约 |

一台服务器只维护一个占用 80/443 的服务器级共享 Docker Caddy 容器，不按项目重复安装。项目和环境仍使用独立 Compose 目录、网络、卷、配置和 `sites` 路由片段。普通路由更新不重写主 Caddyfile；只有能证明属于当前项目 + 环境的片段才可幂等更新或清理。每次变更先备份，在目标容器执行 `caddy validate`，再 reload，任一步失败都 rollback 原片段和配置。

当前实现只会自动复用已经挂载可写 `/etc/caddy/sites` 且主配置已导入该目录的兼容 Docker Caddy。对产品状态而言只有三种结果：`clean`（确认后初始化共享 Caddy）、`compatible`（直接复用）和 `blocked`；缺少 Docker/Compose 是 blocked 的前置原因，不是可继续的第四种能力。旧的 `adaptable` 状态一律迁移为 blocked。systemd Caddy、其他代理、端口冲突或尚无片段入口的实例都按阻断处理，不能把人工修改误报为自动完成。正式环境选择全新服务器也必须复用上述完整初始化检查；这仍是待实现要求，只保存连接或直接绑定不能标记为已验证。

真实桌面运行时目前还有两处与上述产品门禁不一致，必须按未完成项处理：服务器初始化脚本只从 Docker 容器识别 80/443 占用，尚不能在初始化前可靠识别 systemd 或其他宿主机进程；正式路由若已写在现有主 Caddyfile，运行时仍会把它标为 `takeoverAvailable`，允许用户确认后重写目标地址块。该检查无法证明原路由属于另一 ABCDeploy 项目，因此不满足当前“只有已知 ABCDeploy 项目才可转移”的规则。在运行时按归属证据收紧前，不能把静态原型的 blocked 状态当成桌面能力已经完成。

已验证连接在项目打开时先恢复本地状态，再后台复核。真正部署前始终重新验证，缓存不能替代发布门禁。

## 8. 运行和容器化

构建方式按以下顺序选择：

1. 复用项目已有并验证可用的 Dockerfile；
2. 使用支持技术栈的受控模板；
3. 根据标准启动命令生成通用容器定义；
4. 仍不能确定时，只问一个可理解的问题或给出交给开发工具的说明。

自动生成的静态网页容器可以使用 Caddy 提供容器内部静态文件，但不占用服务器的 80/443，也不替代统一边缘 Caddy。项目自有 Dockerfile 不被擅自改写。测试与正式服务器使用独立 Compose 项目、目录、网络、卷和配置文件。

## 9. 数据和地址边界

- 本机可自动创建开发用 PostgreSQL/Redis；远程数据库不会静默创建、迁移或删除。
- 测试与正式使用不同数据库或至少不同库、账号和缓存前缀。
- 自动生产迁移只有在备份、恢复和失败处理完备后才能开放。
- staging 不因缺少域名而阻塞，可使用安全预览。
- production 服务启动与 DNS/HTTPS 是分离事实；地址未就绪不回滚已健康的容器。
- 地址编辑先进入独立 draft；取消不得污染 EnvironmentBinding、发布复核或 Attempt。保存后才写入 `desired_address`。环境分别保存 `active_address` 与 `desired_address`；新路由、DNS 和 HTTPS 全部通过前旧地址继续在线，通过后才原子提升新地址并清空期望值。
- DNS 记录类型由实际运行位置决定：IPv4 为 A、IPv6 为 AAAA、主机名为 CNAME；页面说明与复制内容必须来自同一记录对象。
- 已完成发布后的地址维护只改变访问状态和地址绑定，不创建虚假的部署 Task/Attempt，也不重新部署工作负载。

## 10. 敏感信息

Token、密码、私钥、敏感连接串和 ConfigProfile Value 保存到操作系统密钥库。SQLite 只保存资源 ID、作用域、绑定与状态。完整环境文件不进入 Git、普通日志、错误卡或通知。

外部安全配置只在用户明确操作时于内存生成。确认保存、离开当前任务或短时超时后，客户端仅在剪贴板仍是该内容时清空，并释放内存中的配置包；不得覆盖用户后来复制的新内容。

客户端用于 SSH 登录服务器的专用私钥保存在用户的 `~/.ssh` 目录并限制为 `0600`，不存入 SQLite。项目流水线使用的私钥、平台 Token、Registry 密码和敏感配置值仍保存在操作系统密钥库；界面和只读 Connection API 不返回这些值或私钥路径。

## 11. 验收场景

- 只使用本机运行时，不要求 CNB、Registry 或服务器。
- 配置中心的一项配置可同时绑定多个项目，但环境范围清晰可见。
- staging 修改不会覆盖 production；production 发布也不会反写 development。
- 测试与正式绑定不同服务器时，各自部署和查询正确目标。
- 更换 Registry 或服务器凭据后，从原任务继续且版本资格不丢失。
- 干净服务器首次初始化完成后，第二个项目只复核并复用服务器能力，不重复安装 Caddy。
- 兼容 Docker Caddy 只追加项目 + 环境独立路由片段；只有另一已知 ABCDeploy 项目占用时允许走明确“转移地址”。systemd Caddy、其他代理、端口冲突、旧服务或未知路由归属全部阻断，不提供一次确认式接管。
- 解除某个项目环境的配置绑定时，其他项目或环境对同一 Profile 的引用保持不变；“仅保存”不会改变任何环境 revision。
- 显式配置应用只创建目标项目 + 环境的配置 Task；production 先确认，成功后才推进 applied revision，无变化不创建任务。
- 修改正式地址时旧地址持续在线，直到待切换地址的路由、DNS 与 HTTPS 全部验证成功。
