# ABCDeploy 配置中心与三环境运行模型

> 状态：当前专项规范，已与 2026-07-16 桌面客户端实现对齐。

## 1. 面向用户的四个场景

| 入口       | 用户目标                             | 默认行为                                               |
| ---------- | ------------------------------------ | ------------------------------------------------------ |
| 在本机运行 | 在自己的电脑启停和查看项目           | 快捷控制项目服务和运行依赖，检核必要配置               |
| 管理版本   | 首次完成上线设置，之后查看可部署版本 | 连接代码平台、保存项目版本、准备测试环境、开启自动部署 |
| 部署测试版 | 获得不影响正式用户的测试服务         | 主分支更新后自动部署，用户确认业务结果                 |
| 发布正式版 | 把某个测试通过版本交给真实用户       | 选择不可变候选，复用同一镜像，不重新构建               |

Dockerfile、Compose、镜像仓库、Caddy、SSH 和镜像摘要属于实现细节。默认界面只展示当前结果、影响和下一步；需要排障时才在“技术信息”中展开。

## 2. 配置中心

配置中心是应用级的可选工具，不隶属于某个项目，也不是本机运行或上线的前置步骤。每条配置只有：

| 页面字段 | 含义                                 |
| -------- | ------------------------------------ |
| 配置说明 | 帮助用户区分用途或账号               |
| 配置名称 | 项目配置文件中的 Key                 |
| 配置值   | 实际 Value，敏感值安全保存并默认遮盖 |

项目字段可以选择名称匹配的配置中心值并填入当前编辑环境，但这是一份一次性填入，不是持续绑定。项目保存后拥有自己的环境配置；修改或删除配置中心条目不反向修改已经保存到项目的值。

客户端不能仅凭同名字段自动选中，以免不同项目的相同 Key 被错误复用。敏感识别同时检查名称和值：数据库、缓存连接串，以及 URL 中的账号、密码、令牌和敏感查询参数都按敏感值处理。

## 3. 项目运行配置

项目 `.env.example` 是字段、注释和顺序的权威来源。ABCDeploy 必须：

1. 原样保留注释、空行、未知字段和用户已有内容。
2. 默认先显示缺少值的必填项，再显示缺少值的可选项；“查看全部”才展开已有值。
3. 有注释时用注释作为主标题、配置名称作为副标题；没有注释时直接用配置名称。
4. 不自行翻译名称、猜测业务含义或固化示例项目字段。
5. 允许直接填写，也允许从配置中心一次性填入名称匹配的已有值。
6. 本机、测试和正式配置分别保存和确认，本机值不自动进入远程环境。
7. `DEPLOYDESK_*` 等客户端内部字段继续随原配置保存，但不进入普通表单、缺失统计或配置中心候选。
8. 没有用户需要填写的字段时，直接显示“无需填写，使用当前配置即可”，不制造一次空保存。
9. 测试和正式敏感值不进入 Git、SQLite、普通日志、通知和错误信息。

本机配置交互的完整规则见 [本机运行产品定义](local-running.md)。

## 4. 首次上线设置

一次性设置只在“管理版本”完成：

1. **连接代码平台**：首版默认 CNB；没有 Git 时安全初始化 `main` 和首次提交。
2. **保存项目版本**：验证并使用 OCI Registry，国内试点默认腾讯云 TCR，也支持 CNB 制品库等适配器。
3. **准备测试环境**：连接 Linux 服务器，保存独立测试配置，准备测试域名或本机安全预览。
4. **开启自动部署**：在 CNB 网页完成一次无法由公开 API 代替的安全保存。

已验证且只有一个候选的全局资源自动复用；多个候选时才要求选择。未来步骤可以提前展开了解用途，但前置项未完成时不显示可执行表单、密钥或网页动作。完成最后一项后自动进入测试场景。

## 5. 容器化决策链

系统按固定顺序选择构建方式：

1. 项目已有可用 Dockerfile：复用并执行真实构建验证。
2. 命中已支持的高置信技术栈：使用 ABCDeploy 受控模板生成。
3. 根项目或工作区业务子包声明标准 Node.js `start` 或 `start:prod`：按实际 npm、pnpm、Yarn 或 Bun、服务目录和脚本生成通用多阶段镜像。
4. 外部构建引擎适配器可用：未来可接入 Railpack 或 Cloud Native Buildpacks。
5. 仍无法确定：一次只询问一个用户可理解的问题，或阻断并给出可复制给开发工具的处理说明。

声明 `build` 时在镜像构建阶段真实执行；没有时不虚构构建命令。包管理器和生产依赖裁剪也在构建阶段完成，运行容器不会临时访问 npm、Corepack 或其他包仓库。

生成的 Dockerfile 是项目部署定义的一部分，本机稳定运行、测试和正式使用同一构建逻辑。自动生成的静态网页运行时统一使用 Caddy；项目自有 Dockerfile 不被改写。

## 6. 本机运行边界

本机页面只提供项目服务、运行依赖和必要配置三个通栏模块。ABCDeploy 执行项目已经定义或能够高置信生成的运行方式，不承担依赖修复、代码修复、迁移脚本补写或 AI 修复。

只有要求运行的项目服务最近一次完整启动成功、必要运行依赖可用且阻塞配置已有值时，才开放“打开项目”和业务结果确认。本机动作不需要远程连接，也不得写入 CNB、服务器、DNS 或 Registry。

## 7. 服务器首次连接

系统依次：

1. 从 `ssh-agent`、SSH 配置和常见目录发现已有身份。
2. 没有可用身份时生成 ABCDeploy 专用 Ed25519 Key。
3. 读取并要求用户确认目标服务器主机指纹。
4. 服务器尚未接受公钥时，允许一次性输入当前登录账号的密码并幂等安装公钥。
5. 密码使用后立即清空，不写入磁盘、SQLite、系统密钥库或日志。
6. 已下载登录文件的用户可以在同一卡片改用该文件。

预览版使用用户提供的现有服务器账号，不自动创建新的操作系统用户。创建受限服务账号属于后续安全加固能力。

## 8. 必须保持的环境不变量

- development、staging、production 使用独立配置、数据库/缓存命名、容器命名空间和发布记录。
- 测试与正式可以在同一服务器，也可以绑定不同服务器；共享物理服务器时仍保持逻辑隔离。
- 主分支更新只构建一次不可变版本并自动更新测试环境。
- 正式版只能选择测试通过的版本，发布和恢复均不得重新构建镜像。
- 同一项目任一正式发布未结束时，只能继续或取消原任务，不能替换候选或并行创建第二条正式发布。
- 服务器初始化、覆盖本机 `.env`、正式发布和破坏性动作必须显式确认。
- 本机配置不得自动覆盖测试或正式配置；远程配置也不得反向覆盖本机。

## 9. 生产数据服务边界

预览版可以自动创建本机 PostgreSQL 和 Redis，但不会静默创建、迁移或删除测试与生产数据服务。远程数据库或存储连接可以由用户直接填写，也可以从配置中心一次性填入；不要求配置中心存在专门类型。

自动供应生产数据库属于后续独立能力，只有备份、恢复、容量、高可用和迁移失败处理完整后才开放。

## 10. P0 验收场景

### 场景 A：只在本机运行

用户补齐阻塞配置并启停项目服务和运行依赖，不连接 CNB、TCR 或服务器，也能完成本机使用。

### 场景 B：首次准备自动部署

用户在管理版本按四项设置前进；唯一已验证资源自动复用，未轮到的步骤不提前暴露表单；最后一次网页安全保存完成后直接进入测试版。

### 场景 C：自动部署和验证测试版

主分支更新生成不可变版本并部署测试环境；用户可以离开客户端，回来后从任务中心打开准确版本并确认业务结果。失败不会改变正式环境。

### 场景 D：选择版本发布正式版

用户可从多个测试通过版本中选择一个发布或恢复。生产使用候选记录的同一组镜像摘要；正式服务已运行但地址未完成时保留部署成果，只继续 DNS/HTTPS 检查。

### 场景 E：第二个项目

系统建议已验证的代码平台账号、Registry 和服务器，检测端口、域名、容器名和命名空间冲突；用户不重复粘贴同一凭据或选择 SSH 私钥。

## 11. 错误呈现

每个失败必须包含当前步骤、已经完成的检查、最可能原因、一个主修复动作和默认折叠的脱敏技术详情。第三方原始响应、Shell 输出和完整秘密不能进入一级页面或持久化状态。
