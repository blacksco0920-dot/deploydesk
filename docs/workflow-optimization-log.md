# ABCDeploy 用户工作流优化记录

> 本记录保留每轮假设、取舍和评分，防止产品在实现阶段重新退回“按技术组件搭流水线”的旧模型。评分为 2026-07-18 的专家启发式评审，不等同于真实用户测试结果。

## 1. 目标用户与根本任务

### 核心用户

- 使用 Cursor、Claude Code、Codex 等工具完成项目；
- 能理解“项目、测试、正式、配置”，但不应被要求理解 CI/CD、OCI、SSH、Caddy 或镜像摘要；
- 可能拥有一台国内 Linux 云服务器，也可能复用团队已有服务器；
- 第一次上线愿意提供必要授权，之后希望主要通过合并代码和确认测试结果完成发布。

### 根本任务

> 当项目已经能开发时，我想先得到一个别人可以打开的测试版；确认没问题后，把某个已经验证的版本安全地提供给正式用户。以后代码更新时，系统自动完成重复工作，失败也不要让我从头再来。

### 用户必须参与的事项

系统不能代替用户完成的只有：

1. 提供或确认外部平台授权；
2. 选择或提供运行服务器；
3. 填写项目自身需要、系统无法推断的业务配置；
4. 确认测试业务结果；
5. 提供正式域名并确认生产影响。

仓库名、Registry 路径、容器名、网络名、临时测试地址、Caddy 路由文件和自动化规则默认由系统决定。

## 2. 评分方法

| 维度 | 权重 | 满分标准 |
| --- | ---: | --- |
| 目标清晰度 | 10 | 用户 10 秒内知道当前目标、环境和下一步 |
| 首次成功简洁度 | 15 | 只处理不可替代的信息，没有技术清单和重复确认 |
| 概念负担 | 15 | 不懂部署术语也能完成；术语只在授权来源或技术详情出现 |
| 首次测试速度 | 10 | 先得到临时可访问地址，正式域名和低频设置不阻塞 |
| 日常发布效率 | 10 | main 更新后只需业务验收与生产确认 |
| 配置中心融入 | 10 | 自动匹配、持续绑定、显示来源与影响，不是安全剪贴板 |
| 状态真实性 | 10 | 在线版本、最近尝试、地址状态和当前任务彼此独立 |
| 故障恢复 | 8 | 保留任务、已完成事实和在线服务，只给一个修复动作 |
| 安全与可逆性 | 7 | 生产、路由和配置变化可预览、确认、校验和恢复 |
| Provider 可替换性 | 5 | 用户任务不依赖 CNB、TCR 或某家云的私有名词 |

停止条件分成两层，不能再混用：

1. **静态原型收敛**：独立反例审计确认没有 P0/P1 静态断路，且下一轮在同一评分口径下预计提升小于 2 分；达到后可以停止继续给假数据原型加功能。
2. **真实发布就绪**：必须补齐目标用户无提示任务、桌面运行时、真实 Provider、SSH/Caddy 故障注入与多平台实机证据；静态原型收敛和自动测试全绿均不能替代这一层。

## 3. 第 0 轮：当前实现基线

### 模型

发布中心一次显示一个阻塞项，但首次接入的真实顺序仍是：

```text
代码平台 → Registry → 测试服务器 → 测试配置 → 自动部署
```

正式服务器、正式配置和正式域名在第一次正式发布时再补。配置中心可以在字段中复用值，但仍以单 Key/Value 为主。

### 优点

- 不可变版本、测试资格、正式环境和失败任务已经分离；
- 同镜像摘要晋级、在线旧版保护和任务续跑成立；
- 日常 main 自动部署测试的路径已经较顺。

### 主要问题

- 页面隐藏了部分术语，却仍要求用户按基础设施顺序搭建；
- 首次上线约有 8–12 组判断，并有 CNB 网页往返；
- 项目设置、任务 Sheet 和隐藏整页场景形成双重导航；
- 配置中心看不到使用关系和更新影响；
- “准备服务器”在每个项目任务中重复出现，而不是服务器的一次性能力。

**评分：60/100。** 日常发布约 75 分，首次上线约 40 分。

## 4. 第 1 轮：把技术步骤折叠成一个设置向导

### 改动

- 把代码平台、Registry、服务器和配置放进一个 Sheet；
- 每次只展开一项；
- 已验证资源自动完成；
- 主页面不展示完整清单。

### 得分变化

方向感和页面信息量改善，总分提高到 **68/100**。

### 淘汰原因

这只是把原来的基础设施清单做成折叠面板。用户仍在回答“先连接哪个工具”，而不是完成“先得到一个测试版”。步骤减少了视觉噪声，却没有改变心智模型。

## 5. 第 2 轮：以“生成可访问测试版”为唯一首次目标

### 改动

- 首次首页改为“先生成一个可以打开的测试版”；
- 系统先扫描项目并生成上线计划；
- 临时测试地址由系统生成，不要求域名；
- 仓库名、镜像路径、网络和路由自动命名；
- 只把仍缺少的授权、服务器和业务配置交给用户；
- 执行进度改写为“生成版本 / 放到测试服务器 / 检查能否打开”。

### 得分变化

目标清晰度、概念负担和首次速度显著提升，总分提高到 **79/100**。

### 剩余问题

首次成功、自动化设置、日常发布和故障恢复仍共享同一种任务表现；用户完成首个测试版后，还不容易理解以后什么会自动发生。

## 6. 第 3 轮：分离首次成功、日常交付和故障恢复

### 改动

产品不再提供一条贯穿所有场景的流程，而是按用户意图区分：

1. **第一次上线**：得到可访问测试版并建立自动更新；
2. **日常验收**：查看 main 自动生成的新测试版，确认结果；
3. **发布正式版**：选择任一测试通过版本，确认生产差异；
4. **恢复与修复**：从症状进入，继续原任务。

项目默认页只显示环境事实和当前唯一任务。版本页负责版本资格与历史；项目设置只维护低频资源，不启动发布。

### 得分变化

日常效率、状态真实性和失败恢复提高，总分提高到 **87/100**。

### 剩余问题

如果每个项目仍重新“设置 CNB/TCR/服务器”，第二个项目体验不会明显变快；配置中心也仍然只是字段级复用。

## 7. 第 4 轮：引入可复用交付能力与环境期望状态

### 内部模型

系统增加两个只在内部使用的对象：

- **交付能力**：代码平台、版本存储、服务器访问与统一 Caddy 的可复用连接集合；
- **环境期望状态**：目标版本、运行位置、配置快照和访问地址。

用户不需要看到这两个术语。添加第二个项目时，系统自动匹配已验证能力，只在多个候选或权限失效时询问。

### 配置中心改进

- 仍保持“说明、Key、Value”的简单编辑；
- 项目环境保存持续绑定，而不是复制一次；
- 填表时自动匹配同 Key 候选并显示来源；
- 修改配置时展示受影响的项目和环境；
- 用户选择“仅保存”或“保存并应用到测试/正式环境”；
- 部署记录保存本次使用的配置版本，不回传明文。

### 得分变化

跨项目复用和配置融入提高，总分提高到 **90/100**。

### 剩余问题

服务器已有服务、授权失效、DNS 未生效等场景仍有很多技术分支；如果全部塞回首次页面，复杂度会再次外泄。

## 8. 第 5 轮：以差异和症状驱动的自适应任务

### 最终优化

- 系统每次只比较“用户想要的结果”和“当前事实”，生成一个当前任务；
- 已经满足的能力不重新出现，也不要求确认；
- 全新账号按缺口逐项准备，已有账号自动复用已验证的授权、版本存储、服务器能力和配置引用，不要求用户先选择接入模式；
- 服务器准备成为连接的一次性检查，部署时只做快速复核；
- 兼容的现有 Caddy 自动复用，需要改造时先展示影响并单独确认；
- 授权失败原地更换授权，配置缺失直接打开缺失字段，地址问题只检查地址；
- 测试或生产部署成功后，即使 DNS/HTTPS 未完成，也保留“服务已运行”的事实；
- 第一次正式发布只补正式运行位置、正式配置、正式地址和生产影响；代码授权与版本存储不重复设置；
- 任务详情始终说明“已经完成什么、什么仍在线、现在只需做什么”。

### 得分变化

当时自评为 **91/100**，并误判为达到停止条件。后续独立反例审计证明，这个分数把方案意图当成了可运行证据：全复用仍会固定提问、输入可以空值通过、配置中心只有 Toast、历史版本凭空出现、正式任务可并发、地址检查一次必成功，严格复核只约 **65/100**。该 91 分不再作为当前结论，但保留在记录中说明错误如何发生。

### 无法继续靠静态设计消除的扣分

1. 第一个项目仍需用户取得 CNB/TCR 等外部授权；
2. CNB 安全配置缺少完整写入 API，可能需要一次网页交接；
3. 裸 Linux 的 Docker 安装、云安全组和已有代理冲突存在真实风险，不能静默修改；
4. 不接 AI 时，非标准项目的服务和配置识别不可能百分之百可靠；
5. 没有目标用户测试，文案理解和任务完成率尚无实证。

这些外部不确定性仍然存在，但不能用来掩盖原型内部可继续消除的状态错误。因此第 5 轮“停止模型层优化”的判断被撤销，进入第 6 轮反例驱动优化。

## 9. 第 6 轮：状态真实性、逐能力复用与资源闭环

### 独立审计反证

第 5 轮自动化只验证演示脚本能点通，没有验证输入、持久化和领域不变量。独立审计复现了七类反例：空值被接受、首次历史被虚构、复用仍固定提问、自定义域名被忽略、关闭后正式进度归零、失败版本冒充当前测试版、刷新后任务消失。

### 本轮优化

- 用逐能力对账替代固定模式：全部复用零问、部分复用只问缺项、多个候选只问一次选择；
- 所有表单进入统一状态并逐项校验，秘密不写入原型 Web 存储；
- 配置中心在当前任务中完成引用、新建、绑定和更换，不再用 Toast 冒充结果；
- 增加应用级“连接与资源”，失效授权原地替换并展示影响项目；
- 显式分离 Task 与 Attempt，同一目标重试只增加执行次数，运行中和已完成任务均可重载恢复；
- `VersionValidation` 绑定版本、测试环境、配置修订和服务定义修订，失败尝试不进入可发布版本；
- 正式准备关闭后从原题继续，同一环境禁止并发第二个正式任务；
- 地址检查覆盖待确认、断网、部分生效和完成，任何状态都不重新部署已健康服务；
- 增加已有部署“继续管理 / 重新设置”闭环，选择前不创建任务，重新设置不停止线上服务；
- 本机运行恢复“项目服务 → 运行依赖 → 必要配置”三模块，并继续与远程发布解耦；
- 修复 Sheet 换题与校验焦点，窄屏触控目标提升到 44px，并补强制颜色样式。

### 证据

第 6 轮不再以文案匹配自证，使用 19 组反例测试覆盖输入、零问复用、持久化、互斥、地址故障、版本真实性、连接替换、已有部署、Task/Attempt 和可访问性。最终分数只在浏览器视觉验收和新一轮独立审计之后记录。

## 10. 第 7 轮：把页面状态还原为可审计事实

### 本轮发现

第 6 轮虽然修复了显眼断路，但仍有一类更隐蔽的问题：页面能显示“已准备”，底层却可能只有一个布尔状态，缺少能回答“具体用了什么”的权威对象。典型反例包括：多服务版本只保存一个摘要、服务器连接被名称暗示为测试或正式专属、任意配置项数量冒充必填 Key 已齐、等待用户输入却已经出现执行次数、接管已有部署时制造部署历史，以及新地址尚未验证就覆盖旧地址。

### 本轮事实修正

- **Version 完整性**：版本身份改为完整源提交 + 每个已识别服务的不可变 Registry 引用/digest 集合；测试资格和正式发布必须逐项匹配完整集合，不能用一个代表性镜像代替。
- **Connection 与环境解耦**：Server Connection 是应用级、环境无关资源；项目的 EnvironmentBinding 才表达测试或正式用途。
- **首次服务器绑定门禁**：任何服务器首次绑定到任一环境，都经过凭据、主机指纹、运行前提、共享访问服务分类、影响确认、验证和绑定；测试与正式新增服务器不得走两套标准。
- **配置精确性**：完整性按精确必填 Key 判断；绑定后 Key 身份只读，Profile 与环境应用保存 revision；“仅保存”不改变运行环境，解除绑定只影响指定项目 + 环境。
- **Task/Attempt 边界**：生成任务固定完整源提交，production 任务固定 Version 与完整服务产物集合。等待用户输入时 Task 可以存在但 Attempt 为零；真正执行时才创建 Attempt，并冻结连接、服务器、配置、服务定义和地址等完整输入快照。
- **已有部署接管**：接管按代码连接、Registry 连接、服务器完整身份/能力和最终确认补齐证据，只导入可证明事实；不创建部署 Task/Attempt，不执行部署，也不把凭据存在状态伪装成线上健康。
- **地址切换**：环境分离 `active_address` 与 `desired_address`；新路由、DNS 和 HTTPS 全部通过前旧地址继续在线，完成后才提升新地址。单独地址维护不制造部署记录。

### 阶段证据与未决项

第 7 轮结束时，假数据原型从 P01 起的持续反例集扩展到 **31 项自动测试**，覆盖上述状态对象、冷启动、服务器新增、配置 Key、Task/Attempt 和地址提升。即使 31 项全部通过，也只说明当时已编码的反例成立，是阶段证据，不是最终评分，也不代表平台已经达到停止优化或推广条件。

本轮在完成桌面、窄屏真实浏览器操作和独立反例复核前不记录新分数。真实桌面实现仍需核对：production 新服务器是否执行同一套能力准备、systemd 代理识别、可安全适配 Caddy 的备份/恢复，以及真实 Provider 最终一致性。

## 11. 第 8 轮：消除草稿、接管与产物来源的快照分裂

### 独立反例

第 7 轮 31 项通过后再次进行只读反例审计，仍复现了三类“页面说的是 A，执行保存的是 B”：取消修改正式域名后，复核页读取了未保存草稿而 Attempt 冻结旧地址；已有部署只验证连接就能硬编码生成配置和测试通过证据；更换默认版本仓库后，历史版本只有 digest 而没有可解析的原 Registry 来源。审计同时发现代码平台授权没有 Connection revision、单独地址维护会冒充正式发布完成，以及 DNS 固定生成 A 记录。

### 本轮事实修正

- 地址编辑使用独立 draft；取消只丢弃 draft，保存后才创建 `desired_address`，正式复核和 Attempt 只读取同一权威 EnvironmentBinding。
- 已有部署在匹配服务器完成身份与能力验证后生成不可变 `DiscoverySnapshot`，包含 fingerprint、server、observedAt，以及每个环境的版本、ArtifactRef、配置引用、地址和健康事实；最终接入只能映射快照已经证明的内容。
- Code Connection 保存 provider、repository scope、capabilities、status 和 revision；授权轮换递增 revision，每个 Attempt 冻结当时的 `connection_id@revision`。
- `ArtifactRef` 保存 service、repository、digest 和 Registry Connection revision；更换默认仓库会创建新的活动连接，旧连接以只读形式保留，历史版本来源不被改写。
- 单独地址维护完成记录为 `completion.kind=address` 并返回日常发布中心，不展示“正式发布完成”，也不创建 Task/Attempt。
- DNS 提示与复制共用同一个记录生成结果：IPv4 使用 A，IPv6 使用 AAAA，服务器主机名使用 CNAME。
- 正式准备最后一步发现资源失效时，自动回到具体缺口；“所有项目”补齐真实入口，不再保留死按钮。

### 证据

- 自动反例集扩展到 **P01–P37，37/37 通过**；新增 P32–P37 专门覆盖地址取消一致性、DiscoverySnapshot、Code Connection revision、历史 ArtifactRef、地址完成语义和 A/AAAA/CNAME。
- 真实浏览器桌面视口为 1280×720；首次测试四项、正式新增服务器、已有部署接管和地址取消均完成点击走查。
- 通过真实 iframe viewport 验证 820px 平板与 640px 窄屏：四个应用级入口均可见；640px 下四个工作区入口和四个项目入口触控高度均为 44px；全屏 Sheet 为 640×700，footer 保持可见，body 可独立滚动。

自动测试与浏览器证据仍不能代替真实 CNB/TCR/SSH/Caddy 故障注入，也不能代替目标小白用户首次使用测试。第 8 轮最终评分只在新的独立反例终审后记录。

## 12. 第 9 轮：补齐多项目、规则同步、配置应用与失败续跑

### 独立预审发现

P01–P37 守住了单项目中的快照真实性，但尚未证明真实添加第二个项目后状态不会串线，也没有覆盖当前 HEAD 漂移、自动化规则同步、配置保存与应用分离，以及 production 启动、配置、服务器和路由冲突是否都能沿原 Task 继续。服务器文档还把“空服务器”、兼容 Caddy 和尚未闭环的生产新增服务器初始化混写成了完成能力。

### 本轮事实修正

## 13. 第 10 轮：收口项目身份与单一事实源

### 独立终审输入

Round 9 的问题不是页面文案，而是同一事实存在多份来源：账号连接同时承担项目仓库身份，Version/Validation/Task 重复保存产物，production Task 与 Attempt 重复固定 ArtifactRef，ConfigProfile 反向缓存项目引用，服务器能力还保留无法证明的 `adaptable` 状态。只要这些双源存在，多项目切换、旧快照恢复或执行中修改就可能得到“页面看起来正确、实际执行指向另一事实”的结果。

### 本轮事实修正

- `CodeConnection` 只表达账号级代码平台授权；每个项目使用独立 `SourceBinding(project, connection, repository scope, verified, revision)`。账号可以共享，但仓库、Version、ArtifactRef 和 AutomationRule 均按项目隔离，错误项目仓库不能进入 Attempt。
- `versionsById` 成为 Version 唯一权威。Version 只能由 completed test/import Task 与 succeeded Attempt 共同产生，并保存完整提交、来源 Task/Attempt 与完整 ArtifactRef；`VersionValidation` 只保存环境验证证据。
- production Task 的 goal 只固定 Version ID、环境、意图和期望结果。Attempt 开始时从 Version 复制 ArtifactRef，完成时仍以 Task 固定 Version 为准，页面后来改选不影响任务。
- ConfigProfile 只保存配置自身和 revision，不保存反向 bindings。项目/环境引用只存在于 EnvironmentBinding；跨项目使用数量动态派生。配置 Attempt 冻结完整 ID/revision 集合，只有成功且完成时绑定仍一致才推进 applied revision。
- Caddy 能力只允许 `clean` 初始化、`compatible` 复用和 blocked。旧 `adaptable` 快照按冲突处理；只有另一已知 ABCDeploy 项目路由可以走明确转移，旧服务、第三方或未知归属全部阻断。
- 旧快照恢复清除 Validation 的产物/来源字段与 ConfigProfile 的 bindings；无法安全还原的旧版本只记录为 unresolved，不伪造资格。
- 跨项目浏览器走查再清除两个展示残留：测试地址按当前项目派生；本机服务列表和批量动作按当前项目 service IDs 生成，第二项目显示 `2/2` 而不是继承 OCR。

### 证据与评分状态

- 自动反例集扩展到 **P01–P52，52/52 通过**。P46–P52 分别覆盖 Version/Task/Attempt/Validation 一致性、production 固定目标、旧 adaptable、错误 SourceBinding、未知路由、执行中改绑和旧快照双源清理。

- 对照真实桌面运行时发现两项尚未实现的新门禁：初始化脚本只检查 Docker 容器的 80/443 占用，尚不能提前识别宿主机 systemd/其他进程；主 Caddyfile 的同地址路由仍可经一次确认接管，但检查结果没有项目归属证据。这两项已记录为运行时缺口，不能用静态原型的 blocked 状态冒充完成。

## 14. 第 11 轮：让发布资格和仓库验证都可证明

### 为什么继续优化

Round 10 已经消除大部分双重事实源，但仍把“对象存在”误当成“对象已被证明”：服务产物看起来完整却可能少服务或多服务；Validation 看起来有值却可能不是这次成功 Attempt 的真实输出；账号授权可用却被当成当前项目仓库已经验证。这三类问题都会让页面给出错误的可发布结论，并再次把用户赶回无意义的 Token 输入。

### 本轮事实修正

- Version 的 ArtifactRef service ID 集合必须与当前 Project 的 service ID 集合精确相等；缺少、多出或重复任一服务都不能进入权威 Version。无法安全迁移的旧版本转为 unresolved，不伪造资格。
- 成功测试 Attempt 保存不可变环境输出快照。Validation 的配置、服务、部署、服务器、地址和健康六项事实必须逐项匹配该输出，并继续与 Attempt 冻结输入交叉核对；同时篡改输出和 Validation 也不能绕过冻结输入。
- production Task 每次开始或恢复 Attempt 前重新判断目标 Version 资格，不依赖旧页面状态。导入已有部署的无副作用 Attempt 不再伪造 Provider 外部运行编号。
- CodeConnection 只证明账号可用；SourceBinding 必须来自明确的本机 Git remote 与 Provider 仓库读取结果，并保存验证时间、账号 revision 与证据 ID。恢复、添加、切换项目和账号 revision 变化只能使旧证据保持或降级，不能自动创建、纠正或升级。
- 用户发起首次生成、已有部署接入或自动化暂停/恢复/修复时，若账号可用但项目仓库待核对，系统自动执行一次只读核对。成功就继续原动作；失败不创建 Task/Attempt、不改变 AutomationRule，也不立即显示 Token。只有账号失效或用户主动选择更换账号才进入授权表单；新授权成功后仍恢复原始动作，不能只返回页面让用户重新点击。

### 当前证据与评分状态

- 自动反例集扩展到 **P01–P61，61/61 通过**。P53–P56 覆盖服务集合、环境输出、冻结输入和 production 重验；P57–P61 覆盖无权限失败、旧证据降级、账号修订、多项目切换、自动化写操作和已有部署接入。

## 15. 第 12 轮：任务与证据边界收口（按用户要求终止）

### 本轮最后完成的事情

- 明确 Task 创建时机：代码账号连接可以先保存；只有本机 Git remote 与 Provider 仓库只读核对成功后，才建立项目 SourceBinding 并创建 staging Task。Task 创建早于 Registry、服务器、配置等远程副作用，仓库读取失败则保持零 Task、零 Attempt。
- 开放任务槽不再只相信 `activeTaskId`，而是扫描项目全部 Task；重载时从事实恢复当前任务，发现多个远程变更任务则进入冲突阻断。Attempt 的开始和完成同时核对 Task、环境、序号、状态、冻结 Version、服务集合及目标输入。
- SourceBinding 分开保存本机仓库 scope 与 Provider 仓库 scope。Registry 和服务器分别维护不可变 evidence ledger；Attempt 固定 connection/server revision，历史 ArtifactRef 不会被后来修改的默认连接重写。

- 配置应用增加真实执行门禁：必须存在目标环境在线 Version、精确服务器证据、有效地址、完整配置和正确项目/环境归属；Attempt 冻结在线 Version、服务器修订、配置 ID/revision、先前 applied revision 和地址，完成前任一输入漂移都失败关闭。

| Provider 可替换性 | 4/5 | 用户任务使用通用结果词，平台名下沉到连接详情 | 首版实现仍以 CNB、TCR、Linux 和 Caddy 为试点 |
| **总分** | **91/100** | 与第 5 轮总分一致，但由逐维证据重新相加得到 | 不因缺少真实数据而补齐剩余 9 分 |

复评分与原评分相同不是互相印证的用户证据，只说明在当前评分规则下，文档模型和原型模型已经一致。任何硬门槛失败都直接判定原型不通过，不能用其他维度的高分抵消。

### 剩余不确定性只能由真实用户消除

纸面方案、假数据原型和自动化测试还能发现矛盾、断路和状态错误，但不能回答以下问题：

1. 从未部署过项目的小白能否在 10 秒内理解首屏，并在无人解释时完成第一个测试版；
2. 全新账号与已有资源复用是否真的无需用户判断“我是哪一种模式”；
3. 用户遇到授权不足、服务器已有服务、DNS 延迟或 HTTPS 失败后，是否理解“部署已成功、地址仍待处理”；
4. 配置中心的来源、引用关系和生产影响是否足够清楚，且不会诱导误操作；
5. 用户隔天回来后，是否能根据保留的事实继续原任务，而不是重新设置；
6. 完成首个测试版、首次正式发布和一次故障恢复分别需要多少时间、求助和返工。

这段“91 分必须冻结”的历史结论已经撤销。当前只在独立审计确认无 P0/P1 静态断路，且预计下一轮在同一口径下提升小于 2 分时，判定“静态原型收敛”。这不会自动升级为“真实发布就绪”；后者仍要求真实用户、桌面运行时、真实项目与外部平台证据。

## 16. 静态原型验收硬门槛

无论视觉如何变化，静态原型都必须满足：

1. **G01**：新项目首屏主标题是用户结果，不是资源或技术阶段；
2. **G02**：默认只出现一个主动作；
3. **G03**：首次测试不要求正式域名；
4. **G04**：已有连接、服务器能力和配置自动复用，不重复询问；
5. **G05**：用户可在同一任务中更换失效授权，不删除项目；
6. **G06**：代码更新自动产生测试版，客户端关闭不影响任务；
7. **G07**：任一测试通过版本都能独立创建正式发布任务；
8. **G08**：在线版本与最近失败同时可见；
9. **G09**：服务部署成功后，DNS、HTTPS 或 Caddy 问题不触发重新构建或重新部署；
10. **G10**：用户正常路径不出现 YAML、镜像摘要、SSH、Docker 或 Caddy；
11. **G11**：配置缺失在当前任务内引用或新增配置中心值；
12. **G12**：所有外部副作用之前已经保存可恢复任务；
13. **G13**：Version 是完整 commit、来源 Task/Attempt 与每个服务不可变 ArtifactRef 的唯一权威；Validation 不复制这些来源，正式环境逐项复用同一集合；
14. **G14**：Server Connection 不带环境语义，任何首次环境绑定都在凭据、指纹、前提、访问能力分类和影响确认之后；
15. **G15**：配置按精确 Key 与 revision 判断，引用只来自 EnvironmentBinding；“仅保存”不改变环境，精确解绑不影响其他引用；
16. **G16**：等待输入的 Task 没有 Attempt，Attempt 只对应真实执行且保存完整输入快照；
17. **G17**：已有部署接管不创建 deployment-kind Task/Attempt、不执行部署、不伪造线上状态；确认后只允许 completed import 审计记录证明 Version 来源；
18. **G18**：新地址全部验证前旧地址继续在线，地址提升和部署执行记录彼此独立。
19. **G19**：添加、打开和切换项目使用独立项目工作区；普通重载回发布中心，只有合法显式链接才恢复指定项目与页面。
20. **G20**：测试 Task 创建时固定完整 commit，之后的 HEAD 变化不影响 Task 或 Attempt 输入。
21. **G21**：AutomationRule 的 desired、observed、Provider 与 sync 独立；暂停、恢复和修复不创建部署 Task。
22. **G22**：配置仅保存不影响环境；显式应用才创建精确项目 + 环境的配置 Task，成功后推进 applied revision，无变化不创建任务。
23. **G23**：授权、启动、缺配置、服务器断连和路由冲突均复用同一 Task，只追加完整输入 Attempt，旧正式版本与地址持续在线。
24. **G24**：CodeConnection 只代表账号授权，项目 SourceBinding 必须精确匹配 Project、repository scope 与连接修订，并持有真实验证时间和证据 ID。
25. **G25**：production Task 只固定 Version ID、环境、意图和期望结果；Attempt 从 Version 复制产物，完成不读取后来页面选择。
26. **G26**：配置 Attempt 冻结完整 ID/revision 集合；执行中改绑不得用旧 Attempt 推进 applied revision。
27. **G27**：Caddy 只允许初始化、兼容复用或阻断；旧 adaptable 与未知路由不提供自动接管。
28. **G28**：Version 服务集合与 Project 精确相等；Validation 六项环境事实与来源 Attempt 输出及冻结输入逐项一致，production 每次执行前重验资格。
29. **G29**：SourceBinding 只能由明确的仓库读取结果建立；恢复、切换与账号 revision 变化不能制造证据，失败时不创建任务、不改自动化规则、不立即重问 Token。

各门槛对应的稳定测试编号见 [结果导向发布原型](product-prototype/README.md#硬门槛覆盖矩阵)。这些门槛只用于判断静态原型是否收敛；真实发布就绪仍必须满足目标用户、桌面运行时、真实 Provider 和多平台实机证据，不能由 61/61 或启发式评分代替。
