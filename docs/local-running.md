# ABCDeploy 本机运行产品定义

## 1. 定位

“在本机运行”是项目服务的快捷启停与 `development` 配置检核工具，不是上线向导，也不是测试部署的强制前置条件。

ABCDeploy 执行项目已经定义或能够高置信生成的运行方式。依赖声明错误、代码缺陷、业务数据库迁移和初始化数据缺失应交给开发工具处理；客户端只给出脱敏原因和可复制说明。

## 2. 页面结构

页面按用户关心程度固定排序：

1. 项目服务；
2. 运行依赖；
3. 必要配置。

三个模块通栏纵向展示，不增加“运行准备”总模块，也不因服务与依赖数量不同强行做成对齐网格。

## 3. 项目服务

每个服务展示用户可识别名称、状态、最近一次结果和适用动作：启动、停止、重新启动、打开。支持一键启动全部和全部停止。

状态规则：

- 未启动：没有正在运行的实例；
- 启动中：显示耗时和取消入口；
- 运行中：本次进程/容器仍存在且健康检查通过；
- 启动失败：说明端口、构建、进程或健康检查的结果；
- 已停止：由用户或系统完成停止。

只有要求运行的项目服务“最近一次完整启动成功”且当前仍在运行，才显示“项目可以打开”。不能因基础依赖健康或上一次历史成功就点亮。

## 4. 运行方式

默认提供“稳定运行”，尽量接近远程容器行为。只有所有项目服务都能可靠识别开发命令、源码挂载和刷新行为时，才显示“开发调试”。

- 稳定运行：修改代码后通常需要重启；适合验证接近部署的结果。
- 开发调试：代码变化自动重启或刷新；任何服务不支持时不承诺全项目热加载。

切换运行方式只影响本机，不触发远程构建或部署。

## 5. 运行依赖

运行依赖只展示本机项目确实需要且客户端能够安全控制的服务，例如 PostgreSQL、Redis。每项显示“运行正常 / 未启动 / 启动失败”和启动、停止动作，支持批量启动。

“容器健康”不等于业务数据已初始化。项目要求迁移或种子数据而尚未完成时，客户端说明该项目还需要开发侧初始化，不伪装成数据库故障，也不自动执行未知脚本。

## 6. 必要配置

配置字段来自 `.env.example`：有注释时注释为主标题、Key 为副标题；无注释时 Key 为主标题。优先展示缺值必填项，再展示缺值可选项；已有值在“查看全部”。

用户可以：

- 直接填写当前项目的 `development` 值；
- 选择 Key 匹配的配置中心条目并绑定到 `development`；
- 在项目设置中查看、更换或解除绑定。

本机值不会自动写入任何远程部署线路。配置中心条目只展示说明和 Key，敏感值默认遮盖。

没有用户字段时，显示“本机默认设置已经准备，不需要额外操作”，不制造一次空保存。

## 7. 启动顺序和结果

一键启动时：

1. 检查阻塞配置；
2. 启动运行依赖；
3. 启动项目服务；
4. 逐项健康检查；
5. 满足服务条件后开放访问入口。

用户也可以逐项控制。某个服务失败时，其他已成功服务保持运行，并明确显示失败对象；不会用整页“启动失败”抹掉部分成功事实。

## 8. 错误边界

客户端可以帮助：检查 Docker 是否可用、端口占用、服务日志、依赖连通性、缺失配置和已知运行命令。

客户端不负责：修改业务代码、猜测第三方密钥、补写依赖、自动执行未知迁移、创建生产级数据库或调用 AI 自动修复。

错误卡只给一个主动作，例如“补齐 2 项配置”“释放 3000 端口”“复制给开发工具”“重新启动后台服务”。技术日志默认折叠并脱敏。

## 9. 交互验收

- 用户不连接 CNB、Registry 或服务器也能独立使用本机页面。
- 服务和依赖支持逐个与批量控制，数量不同时布局仍清晰。
- 运行状态刷新不改变用户滚动位置或焦点。
- 开发调试只在真实支持时出现，不能对部分服务热加载做全局承诺。
- 项目可打开状态只由当前服务和最近一次完整启动结果决定。
- 必要配置可以直接填写，也能持续绑定配置中心的 `development` 条目。
