# ABCDeploy 本机运行产品定义

> 状态：当前专项产品规范，已与桌面客户端实现对齐；最近对齐：2026-07-16。

## 1. 产品定位

“本机运行”是可独立使用的项目场景，也是上线前可选的验证方式，但不是部署测试版的第一道强制门槛。ABCDeploy 不是本地开发环境修复工具，它只为已经具备运行条件的项目提供：

1. 项目服务的快捷启动、停止和重启。
2. 数据库、缓存等运行依赖的快捷启动、停止和重启。
3. 项目配置文件的缺失检查、填写和复用，为后续测试部署和正式发布维护配置。
4. 运行失败时整理问题信息，帮助用户返回 Codex、Cursor 等开发工具修复项目。

依赖缺失、编译失败、代码错误、迁移脚本缺失、启动命令错误和运行环境不兼容属于项目开发问题。ABCDeploy 负责准确呈现失败服务和关键信息，但不在客户端内继续构建依赖安装、代码修复或 AI 修复能力。客户端默认不连接 AI 服务。

## 2. 所在流程

添加和切换项目由应用左侧项目栏统一承担，不作为上线步骤。项目主区域使用四个稳定、可反复进入的场景页签：

```text
在本机运行｜管理版本｜部署测试版｜发布正式版
```

本机运行可以独立使用，不是部署测试版的强制前置条件。一次性准备属于“管理版本”，部署和业务验证属于“部署测试版”；正式发布必须使用测试通过的版本。整体关系见 [产品工作流](product-workflow.md)。

## 3. 页面信息顺序

页面按照用户关注程度排列，而不是按照底层技术执行顺序排列：

1. **项目服务**：用户最关心自己的项目是否运行。
2. **运行依赖**：说明数据库、缓存等项目依赖的服务是否可用。
3. **必要配置**：检查和维护本机运行及后续部署使用的配置。

三个模块均使用通栏纵向列表，不使用左右等宽卡片。项目服务和运行依赖数量不固定，并排布局会产生无意义的高度和视觉对齐问题。

页面不设置“运行准备”模块。本机环境、项目依赖、代码生成和初始化脚本等问题仅在某个服务启动失败时作为失败原因出现，并引导用户回到开发工具处理。

## 4. 项目服务

系统识别项目已经定义的可运行服务，并展示：

- 面向用户的服务名称。
- 当前状态。
- 最后一次启动结果。
- 单独启动、停止、重新启动或失败后重试的操作。

支持“一键启动全部”和“全部停止”。一键启动时系统按照项目已经声明的依赖关系和顺序执行；用户也可以逐个控制服务。尚未写出运行文件、但当前计划能够高置信生成的服务，从第一次进入页面起就视为可运行，不能先误报“需要开发处理”、等用户启动后又改变结论。若确实只有部分服务具备可靠运行配置，批量按钮必须明确写“启动可运行的 N 个服务”，底层也只构建和启动这些服务；其余服务继续显示“需要开发处理”，不能用“一键启动全部”暗示已经覆盖，也不能因为一个开发问题阻止其他独立服务运行。

服务状态保持简单：

```text
未启动
正在启动
运行中 / 上次启动成功
启动失败
已停止
```

每行右侧只表达当前状态，副标题补充不同维度的信息。项目当前已停止但最近一次完整启动通过时，使用“上次随全部服务启动成功 / 未启动”，不能同时写“等待启动 / 未启动”重复同一事实；没有成功记录时才使用“等待启动”。“随全部服务”明确这是最近一次完整启动留下的历史结论，不能在单个服务行使用容易被误解为对象范围的“上次全部启动成功”。

示意：

```text
项目服务

✓ 用户网站          运行中        [ 重新启动 ] [ 停止 ]
✕ 后端服务          启动失败      [ 再试一次 ]
○ 定时任务          未启动        [ 启动 ]

[ 一键启动全部 ]                  [ 全部停止 ]
```

### 4.1 本机运行方式

默认使用“稳定运行”，以接近测试和正式环境的容器方式启动。只有系统确认所有项目服务都存在可靠的开发命令、可用的开发镜像阶段和明确的源码目录时，才展示“开发调试”。

```text
运行方式

● 稳定运行
  接近正式环境，修改代码后需要重新启动

○ 开发调试
  修改代码后自动重启或刷新页面
```

开发调试只影响本机运行，不写入 `deploy.yaml`，不改变测试版和正式版的 Dockerfile、Compose、镜像或发布流程。系统自动识别受支持框架的开发命令，例如 NestJS 的 `start:dev` 或 `dev`、Vite/Taro H5 的 `dev` 或 `dev:h5`、FastAPI 的 `uvicorn --reload`；没有特定框架但根目录或工作区业务子包提供标准 `start`/`start:prod` 的 Node.js 项目同样是可部署服务，有 `dev` 或 `start:dev` 时继续提供开发调试。用户不填写也不编辑命令。

通用 Node.js 项目在稳定运行、测试版和正式版中共用同一份生成镜像：有 `build` 脚本时先在镜像内完成构建，再按 `start:prod` 或 `start` 启动；没有 `build` 时直接安装依赖并启动。这样 TypeScript 项目不依赖用户提前在本机生成 `dist`，也不会为纯 JavaScript 项目增加不存在的步骤。

项目已有 Dockerfile 时使用其中的 `EXPOSE`；否则，通用 Node.js 服务会读取 `.env.example` 已明确写出的数字端口，再回退到 3000。用户仍只维护正常配置，不需要另外理解或填写容器端口。

通用 Node.js 即使在页面显示为“网页服务”，数据库地址、端口、管理员密码等仍按服务端配置写入本机、测试版和正式版容器；不会套用纯前端过滤规则，把用户已经填写的必要配置静默丢掉。

稳定运行和远程环境的容器启动不会现场下载 pnpm、Yarn 或其他包管理器。依赖安装、项目构建和生产依赖裁剪都在制作镜像时完成，运行容器只进入正确项目目录并执行已声明的启动脚本，因此服务器临时无法访问 npm 也不影响已有版本启动。

开发调试会按每个服务实际所在目录运行 `dev` 或 `start:dev`，目录中的空格和引号由客户端安全处理；工作区包名不需要用户确认，也不会直接拼接成 Shell 命令。

开发模式使用单独的本机 Compose 文件，并只挂载已确认的源码目录，避免覆盖容器中的依赖。项目仅部分服务可识别时不展示入口，不允许一部分热加载、一部分仍按生产命令运行的模糊状态。切换运行方式前必须先停止项目服务。

容器构建的网络策略属于客户端内部能力，不作为用户设置。Docker 代理失效导致依赖下载失败时，系统自动切换为直连并重试；直连失效时也允许切回代理。某种方式被真实构建验证后保存在本机，后续项目优先复用，避免每次先等待同一种已知失败。只有连接拒绝、域名解析失败、网络超时等真实网络错误才能触发切换；构建脚本中正常出现镜像源或包源域名，不能把普通代码编译失败误判为网络问题并重复构建。逐个启动与一键启动使用相同策略，逐个启动同时构建所选服务声明的依赖。

ABCDeploy 自动生成的静态网页镜像统一使用 Caddy 提供静态文件、SPA 路由回退和 `/api` 反向代理，不再生成 Nginx 配置；项目自己已有的 Dockerfile 仍保持原样。自动生成文件需要拉取公开基础镜像时，客户端使用不含登录信息的隔离 Docker 配置，同时保留本机 Compose 与 Buildx 插件发现，避免 Docker Desktop 凭据助手异常让公开镜像也一直等待。该策略只用于 `.deploydesk/generated` 下由 ABCDeploy 生成的 Dockerfile；用户自有 Dockerfile 继续使用正常 Docker 配置和私有镜像登录信息。

ABCDeploy 自己的生成文件、运行状态和备份不得进入 Docker 构建上下文，否则每次记录本机状态都会让 Docker 误判为项目代码变化并重新构建。客户端只升级带 ABCDeploy 标记的 `.dockerignore`，加入 `.deploydesk/generated`、`.deploydesk/state`、`.deploydesk/runtime` 与备份目录；项目自己维护的 `.dockerignore` 不自动覆盖。项目代码或依赖真实变化时仍由 Docker 内容缓存正常失效并重新构建。

ABCDeploy 生成的 Caddy 静态网页容器启动很快，使用 1 秒准备期、2 秒检查间隔和 2 秒单次超时，最长约 30 秒确认失败；不能沿用后端服务的 20 秒准备期和 10 秒检查间隔。NestJS、FastAPI、数据库等可能需要初始化的服务继续保留原来的宽松健康检查。生成的 Vite/Taro H5 即使在项目清单中归类为网页服务，也必须使用 Caddy 镜像内可用的 `wget` 检查，不能调用镜像中不存在的 Node.js。

启动前先检查项目要使用的本机端口。端口已经被其他程序占用时，页面立即指出端口并停止构建，避免等待镜像完成后才失败。若占用者是 ABCDeploy 启动的另一个本机开发项目，页面直接显示项目名称并提供“停止该项目并继续”；底层只允许停止带 ABCDeploy 项目与 `development` 标签的容器，普通程序仍由用户自行处理。构建和启动期间，服务行持续显示“正在构建并启动”和已经等待的时间，并提供“停止本次启动”；停止时必须结束完整 Docker 子进程组，不能遗留后台构建。首次运行可能需要几分钟，但连续三分钟没有任何构建输出时应自动停止并解释为 Docker 下载或网络问题，不能只留下禁用按钮，也不能误导用户修改项目代码。代码构建失败时，首层至少指出失败服务和可安全归纳的原因，例如 TypeScript 问题数量或缺失项目模块，并提供“复制给开发工具”；安全摘要在重新进入项目后仍然保留，完整构建输出和真实配置不得落入页面状态。必要配置已有值但尚未保存时，配置模块明确提示“保存后即可使用”，保存成功后清除旧提示。

历史失败只在尚未有更新结果时用于恢复。用户后来成功启动任意一个服务后，旧失败提示立即清除；只有全部项目服务都运行成功时才把顶部场景点亮为“上次运行成功”，部分服务成功不能冒充整项验证通过。

## 5. 运行依赖

运行依赖独立于项目服务，展示项目已经声明需要的本地数据库、缓存、消息队列等基础服务。ABCDeploy 提供启停和状态检查，但不自行猜测或补写项目的数据初始化、迁移和业务逻辑。

示意：

```text
运行依赖

✓ 本地数据库        运行中        [ 重新启动 ] [ 停止 ]
✓ 缓存服务          运行中        [ 重新启动 ] [ 停止 ]
○ 消息服务          未启动        [ 启动 ]
```

运行依赖不能只检查进程是否存在。在项目已经提供相应检查方式时，应展示项目是否能够实际连接和使用。项目没有提供初始化方式或检查方式时，ABCDeploy 不自行发挥。

## 6. 必要配置

必要配置模块与项目的配置文件和应用级配置中心衔接。客户端不猜测配置项的业务含义，也不把某个示例项目的字段固化进产品。

### 6.1 展示规则

- 默认优先展示当前没有值的配置项。
- “查看全部配置”展开项目中的所有配置项。
- 排序依次为：没有值的必填项、没有值的可选项、已有值的必填项、已有值的可选项。
- 项目注释作为配置项主标题，配置名称（例如 `DATABASE_URL`）作为副标题。
- 没有注释时，配置名称直接作为主标题。
- 保留项目原有配置名称、注释、顺序和未知字段，不自行翻译或编造说明。
- 敏感值默认遮盖，不在普通界面、日志或错误信息中显示原文。

示意：

```text
必要配置                                      2 项没有值

第三方服务的访问凭证                         ← 项目原有注释
配置名称：SERVICE_ACCESS_KEY
[ 输入配置值                              ]
[ 使用配置中心已有值                      ▾ ]

CUSTOM_SETTING                              ← 没有注释时使用英文名
[ 输入配置值                              ]

[ 保存配置 ]                         [ 查看全部配置 ]
```

### 6.2 配置填入方式

每个配置项允许两种来源：

1. **仅此项目**：用户直接填写，只供当前项目和当前环境使用。
2. **配置中心**：用户选择已经保存的配置，把值一次性填入当前项目和环境。

客户端可以优先展示配置名称相同的已有配置作为候选，但不能仅凭名称自动选中，避免不同项目的同名字段被错误复用。本机配置不得自动带入测试或正式环境；每个环境分别确认使用的值。

## 7. 配置中心

配置中心是应用级入口，只保存可复用的通用配置，不再按 AI、数据库或第三方服务等类型设计复杂表单。每条配置仅包含：

| 页面字段 | 含义                                                 |
| -------- | ---------------------------------------------------- |
| 配置说明 | 帮助用户识别这条配置的用途或账号，例如“测试环境账号” |
| 配置名称 | 项目配置文件中的名称，例如 `KEY`                     |
| 配置值   | 实际使用的 `VALUE`；敏感值安全保存并默认遮盖         |

示意：

```text
配置中心

[ ＋ 添加配置 ]

配置说明                 配置名称                 配置值
测试环境账号             SERVICE_ACCESS_KEY       ••••••••   [ 编辑 ]
团队共用访问凭证         SHARED_ACCESS_TOKEN      ••••••••   [ 编辑 ]
默认文件保存位置         STORAGE_PATH             /data/files [ 编辑 ]
```

项目选择配置中心中的值后，可以在当前编辑会话中说明“来自配置中心 · 测试环境账号”；保存时把值写入当前项目和环境，不建立持续引用。之后修改或删除配置中心条目，不反向改写项目已经保存的配置。

## 8. 启动失败的处理边界

当服务因为开发问题启动失败时，页面不提供“自动修复”，而是展示：

1. 哪个服务启动失败。
2. 可理解的错误摘要。
3. “复制问题信息”主操作。
4. 默认折叠的脱敏技术详情。
5. 用户在开发工具修复后的“再试一次”。

复制内容应包含项目目录、失败服务、项目原有启动命令、退出状态、错误摘要和经过脱敏的关键日志，方便用户交给其开发工具继续处理。

## 9. 完成门槛

只有满足以下条件，页面才开放“打开项目”和运行结果确认：

1. 要求运行的项目服务最后一次启动结果均为成功。
2. 对应的必要运行依赖处于可用状态。
3. 阻塞本机运行的必填配置均已有值。

完成后由用户做最后的业务判断：

```text
✓ 项目服务已经启动
✓ 运行依赖正常
✓ 必要配置已经补齐

[ 打开项目 ]

运行结果符合你的预期吗？

[ 有问题 ]                      [ 准备上线 ]
```

系统只能确认服务启动和可访问，不能替用户判断实际业务功能是否符合预期。

## 10. 原型验收标准

- 用户首先看到项目服务，其次是运行依赖，最后才是配置内容。
- 项目服务和运行依赖均可逐个控制，也可一键启动或停止。
- 只有全部服务都能可靠支持时才展示“开发调试”，默认仍为“稳定运行”。
- 开发调试设置仅保存在本机，不影响测试版或正式版发布文件。
- 用户能区分“服务启动失败”和“配置缺少值”。
- 构建中持续显示等待时间，用户可以停止；无进展会自动超时且不会残留后台进程。
- 配置项只使用项目原有注释和配置名称，不由客户端猜测含义。
- 用户可以直接填写配置，也可以选择配置中心中的已有值。
- 页面没有“运行准备”、依赖安装或 AI 修复入口。
- 启动失败时，用户知道应把什么信息交给开发工具。
- 所有要求运行的服务成功前，不能误确认本机运行已经完成。
- 顶部“在本机运行”只在所有项目服务最近一次完整启动并通过检查后点绿；主动停止服务不清除这个已验证状态，后续启动失败则显示警告。
