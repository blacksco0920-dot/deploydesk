# ABCDeploy 产品工作流

> 当前模型：先让项目拥有一个可访问的测试版，再建立日常自动交付；基础设施是后台能力，不是用户流程。

## 1. 用户只需要理解的模型

项目内保留四个稳定入口：

```text
发布中心｜在本机运行｜版本｜项目设置
```

| 入口 | 只回答一个问题 |
| --- | --- |
| 发布中心 | 测试版和正式版现在是什么状态，我现在只需要做什么 |
| 在本机运行 | 如何启停本机服务、打开本机结果、补本机配置 |
| 版本 | 哪些版本已经产生，哪些已通过测试，可以发布或恢复 |
| 项目设置 | 低频维护自动更新、运行位置、环境配置和访问地址 |

用户心智模型只有一句话：

> 代码更新后，ABCDeploy 自动生成一个测试版；我确认能用后，可以把这个或任何以前测试通过的版本发布给用户。

CNB、TCR、OCI、SSH、Docker、Compose、Caddy、流水线、镜像摘要和构建编号不进入普通工作流。无法隐藏的 Provider 名称只在“授权来自哪里”和技术详情中出现。

应用可以同时管理多个真实项目。每个项目拥有独立的任务、环境、版本和页面工作区；普通打开项目或重载应用都回到该项目发布中心，只有项目 ID 与页面都合法的显式 deep-link 才直接进入指定项目页。

## 2. 系统怎样决定下一步

系统不执行固定 Stepper，而是持续比较：

```text
用户想要的结果
  - 一个可访问的测试版
  - 某个版本成为正式版

当前事实
  - 已识别的项目服务
  - 已验证的账号与服务器
  - 已绑定的环境配置
  - 已存在的版本、部署和地址

差异
  → 生成一个当前任务
  → 只突出一个主动作
```

页面可以用一句话提前说明完整剩余计划，例如“还需要确认 2 项连接、运行位置和 2 项应用信息”，避免用户产生“完成一项又冒出一项”的移动终点感；同一时刻仍只展开当前一个问题。

内部使用 `Connection`、`SourceBinding`、`ServerCapability`、`EnvironmentBinding`、`Version`、`VersionValidation`、`DeploymentTask` 和 `DeploymentAttempt` 等领域对象，但它们不是用户必须学习的概念。每类事实只有一个权威来源：账号授权属于 CodeConnection，项目仓库属于 SourceBinding，配置引用属于 EnvironmentBinding，提交与产物属于 Version。

## 3. 添加项目后的第一次成功

### 3.1 只读识别

添加项目仍只在侧边栏完成，不属于上线步骤。系统先只读识别：

- 项目服务和可访问入口；
- 已有 Dockerfile、Compose 或标准启动方式；
- `.env.example` 中的配置项、注释、必填性和顺序；
- 已有 ABCDeploy 部署定义；
- 可以复用的账号、服务器和配置绑定。

高置信结果直接采用，放进“查看识别结果”。只有结果会改变要启动的服务或对外入口且系统无法确定时，才问一个面向结果的问题，不让用户选择容器技术。

代码平台授权按账号复用，但不能由账号连接自动推断项目仓库。每个 Project 都要建立并验证精确的 SourceBinding。添加、切换或恢复项目时只读取并校验已有验证证据，不能凭账号可用自动创建、纠正或升级绑定；缺少证据、仓库不匹配或授权 revision 变化时先标为待核对，用户发起首次生成、已有部署接入或自动化写操作时再执行一次只读仓库核对。账号仍有效时不重复索要 Token；核对失败不创建 Task/Attempt，也不改变自动化规则，只有用户主动选择更换账号时才进入授权表单。

### 3.2 首屏目标

新项目发布中心的首屏固定表达：

```text
先生成一个可以打开的测试版

已识别 3 个项目服务。
先拿到测试地址，其他信息只在真正用到时再补。

[生成测试版]
```

首次页面不得显示空的正式环境卡，也不得先展示 CNB、TCR、服务器或自动部署清单。

### 3.3 完整但简短的准备计划

用户点击“生成测试版”后，任务面板先说明系统将做什么，以及还需要用户配合的事项：

```text
生成测试版前，还需要你完成 4 件事

○ 允许 ABCDeploy 获取代码更新
○ 选择生成版本的保存位置
○ 选择测试版运行的位置
○ 补齐 2 项项目运行信息
```

四项是全新账号的能力上限，不是每个项目的固定向导。系统先逐能力对账：全部可复用时直接执行、零个问题；部分可复用时只列真实缺项；多个有效候选时只询问一次选择。已经满足的事项不重新询问，唯一且已验证的候选默认复用，同时保留“更换”。

### 3.4 首次测试可能需要参与的六类事项

1. **确认系统识别结果**：仅在低置信且会影响结果时出现；
2. **允许获取代码更新**：只说明授权目的，Provider 名称进入来源详情；
3. **连接版本保存位置**：仅在没有已验证 Registry 连接时出现；
4. **选择测试版运行位置**：优先复用已验证服务器；服务器准备由系统后台完成；
5. **补齐业务配置**：只展示当前环境缺值项，并直接引用或新增配置中心值；
6. **确认业务结果**：测试地址可访问后，由用户判断功能是否符合预期。

测试环境是隔离且可恢复的，用户补齐最后一项信息后，系统直接开始任务，不再出现“资源已准备好 → 再点一次生成”的重复确认。

仓库名、版本存储路径、容器名、网络名、临时测试地址和自动更新规则均使用安全默认值。正式服务器、正式配置和正式域名此时不出现。

## 4. 生成第一个测试版

执行进度只使用结果语言：

```text
正在生成测试版

✓ 读取项目
✓ 生成可恢复版本
● 启动项目服务
○ 检查测试地址

可以关闭客户端，任务会继续，完成结果会保留。
```

系统内部完成：同步代码、构建一次、保存不可变 OCI 产物、部署测试环境、健康检查、设置临时地址和建立 `main` 自动更新。技术阶段只进入折叠详情。

完成后先显示：

```text
测试版可以打开了
[打开测试版]
```

用户打开后才出现：

```text
[还有问题，我先修改]  [可以使用，标记测试通过]
```

“还有问题”不是部署失败，只表示该版本暂不获得正式发布资格。用户继续修改并合并代码，测试地址随后自动更新。

## 5. 日常代码更新

默认自动化规则不是用户需要决定的功能开关，而是首次成功的结果：

```text
main 更新
  → 自动生成不可变版本
  → 自动部署测试环境
  → 健康检查
  → 等待业务确认
```

客户端可以关闭。再次打开时，发布中心按真实状态显示：

```text
有一个新测试版等你确认
7 月 18 日 19:13 · 修复登录问题
[打开并确认]
```

下方环境只陈述事实：

```text
测试版：v18 正在运行，等待确认
正式版：v16 正常提供服务
```

如果 v18 启动失败，必须同时显示“测试环境仍运行 v17”和“v18 更新失败”。没有待办且环境健康时可以只显示“一切正常，等待下一次代码更新”，不为保持按钮数量而制造动作。

AutomationRule 自身不是部署任务。它分别保存 desired、observed、Provider 引用与 sync 状态；暂停、恢复、同步失败和修复只收敛同一条规则，不创建部署 Task、Attempt 或版本，也不改变测试/正式环境。Provider 后续因代码事件执行构建和测试更新时，执行事实由对应领域记录独立表达，不能借规则状态冒充成功部署。

## 6. 版本资格

版本由完整提交和完整服务产物集合唯一标识；每个已识别服务都必须拥有不可变 Registry 引用和 digest。用户页面使用时间和修改说明作为主名称，完整提交、服务清单和 digest 折叠展示。

`Version` 是版本身份的唯一记录，保存项目、完整提交、来源 Task/Attempt 和完整 ArtifactRef 集合；其中服务 ID 必须与当前 Project 识别出的完整服务集合精确相等，缺少或多出任一服务都不能形成有效 Version。成功 Attempt 另外保存本次真实输出的不可变环境快照。`VersionValidation` 不复制版本来源或产物，只记录以下环境验证证据，并且每一项都必须与来源 Attempt 的输出快照及冻结输入逐项相等：

- 测试环境配置修订；
- 服务集合与部署定义修订；
- 测试服务器和地址；
- 健康检查结果与用户确认。

完整服务产物集合仍只存在于 Version 与 Attempt 输出中，资格判断时交叉核对，不复制进 Validation。同一产物集合使用不同配置重新部署时产生新的部署验证记录，不能用旧环境结果证明新配置一定可用；正式 Task 每次开始或恢复 Attempt 前都重新校验目标 Version 资格，缓存的“可发布”不能代替当前证据。

版本可以处于：生成中、测试部署中、等待确认、测试通过、测试有问题、已在正式环境、已废弃。失败尝试是任务事实，不伪装成新版本，也不抹掉历史资格。

## 7. 发布或恢复正式版

测试通过后，发布中心推荐当前候选；版本页允许选择任一测试通过版本。选择历史版本时，动作名称为“恢复此版本”。

首次正式发布只补正式环境独有内容：

1. 正式运行位置；
2. 正式配置；
3. 正式访问地址；
4. 生产影响确认。

默认推荐已验证资源，但测试服务器和测试配置绝不静默变成正式绑定。最终确认只展示：目标版本、正式地址、运行位置、当前在线版本、恢复保障和配置变化摘要。

```text
将把 7 月 18 日 19:13 版本发布给正式用户

地址：www.example.com
运行位置：正式服务器 A
当前正式版：7 月 15 日版本
恢复保障：新版本未健康前不会切换

不会重新生成，将使用测试通过的同一版本。
[确认发布]
```

正式发布不重新构建，也不按单一“主镜像”漏掉辅助服务。production Task 只固定目标 Version ID、环境、意图和期望结果；真正开始执行时，Attempt 从该 Version 解析并冻结完全相同的服务产物集合。完成时再次核对 Attempt 与 Task 固定 Version，页面后来改选版本不能改变任务。新版本健康后才更新在线事实；失败时保留原正式版本。

DNS 或 HTTPS 未完成时拆成两个事实：

```text
✓ 正式版已经在服务器运行
○ 正式地址尚未生效
```

环境把“当前生效地址”和“待切换地址”分开保存。此时只继续地址检查，不重新部署容器，也不伪造新的部署 Task/Attempt；新路由、DNS 和 HTTPS 全部通过前旧地址继续在线，通过后才把待切换地址提升为当前地址。数据库和外部数据不会随镜像自动回退，涉及迁移时必须单独说明。

## 8. 配置中心怎样进入主线

配置中心继续只编辑三项：**配置说明、配置名称、配置值**。复杂度由使用关系承担，不增加技术字段。

任务中的配置表单遵循：

- `.env.example` 注释为主标题，Key 为副标题；没有注释时 Key 为主标题；
- 缺值必填项优先，其次缺值可选项，已有值折叠；
- 按 Key 自动匹配配置中心候选，显示说明、来源和当前绑定环境，不回显敏感值；
- 配置是否完整只按项目声明的精确必填 Key 判断，不能用相似名称、任意条目或“数量已够”代替；
- 没有候选时原地新增，保存后自动绑定当前环境，不跳转页面；
- 环境保存持续绑定关系，不做一次性无来源复制；
- Profile 绑定后 Key 只读；更换名称通过新建正确 Key 并精确解除当前项目 + 环境绑定完成，不影响其他引用；
- ConfigProfile 只保存自身信息与 revision，不保存反向 bindings；项目和环境的使用关系只从 EnvironmentBinding 派生；
- 每个 Profile 和环境应用保存 revision；执行 Attempt 冻结目标环境完整 Profile ID/revision 集合，不保存业务明文。

修改配置时必须区分：

```text
[仅保存]
[保存并更新测试版]
```

“仅保存”只更新配置中心 Profile revision，不静默改变任何正在运行的环境。用户显式选择应用到某个项目环境后才创建配置 Task；正式环境先确认生产影响。只有 Attempt 成功且完成时 EnvironmentBinding 仍与冻结 ID 集合一致，才用整份冻结 revision 推进该环境的 applied revision；执行中改绑必须重新确认。无 revision 变化时不创建任务；应用目标只有一个推荐环境，解除引用也只作用于明确的项目 + 环境。配置中心详情从所有项目的 EnvironmentBinding 派生真实引用，删除或更换前展示影响范围。

## 9. 服务器和统一访问服务

服务器 Connection 是应用级、环境无关的可复用资源；只有项目的环境绑定才表达它服务于测试或正式环境。任何服务器第一次被任一环境使用时，都必须完整经过：验证凭据 → 确认并固定主机指纹 → 检查 Docker/Compose、权限、端口和容量等运行前提 → 分类共享访问服务 → 展示并确认将发生的影响 → 标记连接已验证 → 最后建立环境绑定。不得根据主机名或使用场景把未验证连接直接绑定为“测试服务器”或“正式服务器”。后续项目只做轻量复核，不把“准备服务器”重复放进每次部署流程。

用户只看到“运行位置”和“访问服务”：

- 干净且受支持的服务器：Docker Engine 与 Compose 已可用、80/443 未被占用；展示将创建的运行目录和服务器级共享 Docker Caddy 容器，确认一次后幂等准备；
- 裸机缺少 Docker/Compose：作为“缺少运行前提”单独处理；裸机自动安装尚未闭环，不能误报为空服务器或已可初始化；
- 已有兼容 Docker Caddy：确认配置入口、`sites` 挂载和 reload 方式后复用，项目 + 环境只维护自己的路由片段；
- systemd Caddy、其他代理、多个代理或无法确认归属：停止，不自动覆盖；更换运行位置或先由维护人员整理成可验证契约。

一台服务器只接入一个占用 80/443 的服务器级共享 Caddy 容器；普通路由追加不重写主 Caddyfile，也不重复安装项目专用代理。每次片段变更先备份，执行 `caddy validate` 后 reload，失败立即 rollback。正式版选择全新服务器也必须走上述同一初始化检查；这是尚待真实桌面运行时闭环的要求，不是已经完成的能力。目标地址冲突必须先判断归属：

- 当前项目自己的路由缺失或漂移：幂等修复；
- 其他 ABCDeploy 项目正在使用：进入明确的“转移地址”流程，同时展示来源项目和影响，不用通用确认直接覆盖；
- 其他来源、旧服务或无法确认归属：阻断并给出诊断，不提供一次确认式自动接管；用户只能更换地址，或先由维护人员把归属整理成可验证契约。

任何变更先备份并校验，热加载失败恢复原路由。移除项目只清理能证明由该项目拥有的片段，不卸载共享 Caddy，不触碰其他项目和未知服务。

## 10. 失败与任务续跑

所有外部副作用发生前，先保存 `DeploymentTask`。首次生成时版本尚未产生，Task 固定完整源提交、目标环境和“待解析版本”目标，构建成功后再原子绑定不可变 Version ID；已有版本发布时 Task 只固定 Version ID、环境、意图和期望结果。用户仍在补信息时 Task 的 Attempt 数为零；只有真正开始一次远程执行才新增 `DeploymentAttempt`，并从 Task 固定的 Version 复制完整 ArtifactRef。

每个 Attempt 冻结当次完整输入：代码/版本目标、每个服务的 ArtifactRef（repository + digest + Registry Connection revision）、代码 Connection revision、服务器绑定与能力 revision、ConfigProfile ID 与 revision、服务/部署定义 revision、当前地址和待切换地址。失败重试保持 Task ID 不变，只新增带新快照的 Attempt；恢复不能用当前页面的可变设置覆盖历史输入。代码授权轮换只推进 Connection revision；更换默认 Registry 连接不能改写历史 Version 的 ArtifactRef，旧连接至少保留读取该历史产物所需的能力。

失败一级页面固定回答：

1. 这次什么没有完成；
2. 哪些步骤已完成、哪个旧版本仍在线；
3. 用户现在只要做什么。

系统自动处理网络波动和可重试等待；只有权限、业务配置、服务器冲突、地址归属和生产影响需要用户操作。

- 授权失败：原地更新授权，保存后继续原任务；
- 配置缺失：直接打开缺失字段并引用配置中心；
- 服务器连接失败：更换或重新验证运行位置；
- 地址由其他 ABCDeploy 项目占用：打开“转移地址”流程，不直接覆盖；
- 地址不是由当前或另一已知 ABCDeploy 项目拥有、归属未知或代理冲突：停止自动修改并给出诊断入口，不提供自动接管；
- DNS/HTTPS 未完成：只检查地址，保留服务已部署事实；
- 项目代码、依赖或数据库初始化问题：说明返回开发工具修复，不伪装成客户端能够解决。

授权、production 启动、配置缺失、服务器断连和路由冲突均沿原 Task 恢复。配置缺失复用配置中心，服务器断连复用连接维护；每次真正继续只新增一条冻结完整输入的 Attempt，不创建第二条顶级任务。旧正式版本和当前地址在新 Attempt 完成健康检查与安全切换前持续在线。

外部网页操作前保存停点；用户返回后自动复核，同时保留“检查并继续”和最近检查时间。普通进入项目仍回到发布中心，不自动弹出旧任务。

## 11. 已有部署项目

检测到已有部署定义时，先展示系统发现的事实和推荐选择：

- **继续管理现有上线**：只读导入已有版本和环境事实，再补当前设备缺失的连接；
- **作为新的上线重新设置**：保留项目文件和线上服务，只清除本机该项目的 ABCDeploy 关联，从“生成测试版”开始。

选择“继续管理”后按“代码连接 → 版本保存连接 → 完整服务器身份与能力 → 远程事实快照 → 汇总确认”逐项补齐证据。只有服务器身份与本机线索匹配时，才生成不可变 `DiscoverySnapshot`，记录主机指纹、服务器、观察时间，以及每个环境实际发现的版本、ArtifactRef、配置引用、地址和健康状态；最终确认只能映射快照已证明的事实，缺失配置继续进入补齐流程。整个接管流程只建立管理关系，不创建 deployment-kind Task/Attempt、不执行部署，也不因为某个凭据存在就把线上状态标为正常。最终确认后允许为每个导入 Version 生成已经完成的 `kind=import` 审计 Task/Attempt，用于证明发现快照与 Version 来源；这类记录没有远程副作用，不能冒充一次部署。

选择前不读取远程历史、不恢复旧账号、不制造待办。页面必须具体展示“将导入什么 / 将清除什么 / 什么绝不会删除”，不能只让用户理解抽象的“已有部署状态”。

## 12. 在本机运行

本机运行是独立工具，不是上线门禁。它只控制项目服务、运行依赖和 `development` 配置；不会写入代码平台、Registry、远程服务器或 DNS。用户可以跳过本机页面，直接生成远程测试版。

## 13. 禁止重新出现的旧模型

- 不再使用“CNB → TCR → SSH → 配置 → Caddy”的用户向导；
- 不把本机运行作为远程上线前置条件；
- 不使用顶层部署 Stepper；
- 不等用户完成一项后才第一次透露还有哪些事项；
- 不要求跳到配置中心再返回当前任务；
- 不用最新失败覆盖当前在线版本；
- 不在 production 重新构建；
- 不把 DNS 未生效显示为整体发布失败；
- 不通过删除项目解决凭据失效；
- 不在普通进入项目时自动打开任务面板；
- 不在首次测试时要求正式服务器、正式配置或正式域名；
- 不把“是否开启 main 自动部署”作为小白必答选择。
