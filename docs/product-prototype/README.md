# ABCDeploy 结果导向发布原型

本目录使用假数据验证收敛中的用户工作流，不代表最终视觉稿，也不连接真实云资源。打开 [index.html](index.html) 即可体验。

## 本轮验证重点

旧模型让用户按代码平台、Registry、服务器和代理的技术顺序搭建部署系统。当前模型以用户结果为主语：先得到可打开的测试版，系统按已经掌握的事实计算差异，只在必须授权、选择、补业务值或确认生产影响时提问。

Round 12 按用户要求完成最后收口后停止：账号级 CodeConnection 可以先保存，但只有明确核对本机 Git remote 与 Provider 仓库读取结果后才建立项目 SourceBinding；远程任务只在该核对通过后、任何 Registry、服务器或配置写操作前创建。Task/Attempt、Registry、服务器和配置应用继续使用冻结修订与不可变证据，避免重载、换连接或跨项目时把新状态冒充成旧任务事实。

应用级入口为“所有项目、配置中心、连接与资源、任务”；项目内稳定入口为：

```text
发布中心｜在本机运行｜版本｜项目设置
```

- **发布中心**：环境事实、当前任务和唯一下一步；
- **在本机运行**：项目服务、运行依赖和必要配置，不作为远程发布门禁；
- **版本**：真实产生的版本、验证证据、正式发布与恢复资格；
- **项目设置**：当前项目绑定；可复用连接在应用级“连接与资源”维护。

## 十个可切换场景

1. **新项目**：首屏只承诺“生成一个可以打开的测试版”，没有空正式环境卡和技术资源清单。
2. **只补缺项**：代码连接、版本存储和服务器已复用，只询问仍缺少的必要配置。
3. **全部复用**：四项能力均已验证，零个问题，直接创建可恢复任务。
4. **多个候选**：只有运行位置存在多个有效候选，因此只做一次选择。
5. **已有部署**：先核对在线事实，再选择继续管理或重新设置；选择前不创建远程任务。
6. **正在生成**：只显示读取项目、生成版本、启动服务和检查地址等结果进度。
7. **测试版可用**：先打开业务地址，之后才能记录“还有问题 / 可以使用”。
8. **日常更新**：`main` 更新自动产生测试版，正式版保持不变，测试通过版本可独立发布。
9. **故障恢复**：旧在线版本不丢失，只替换失效连接，并在同一 Task 下增加 Attempt。
10. **服务器识别**：验证凭据和主机身份后，按运行前提与共享访问服务现状分类；确认影响前不绑定环境。

## 首次测试采用逐能力对账

全新账号最多缺少四项能力：

1. 允许获取代码更新；
2. 连接版本保存位置；
3. 选择测试版运行位置；
4. 补齐测试环境必要配置。

这不是固定四步向导。系统先计算事实：全部可复用为零问，部分复用只问实际缺项，多个有效候选只问一次选择。所有输入都进入状态并逐项校验，空值不能靠默认假数据通过；秘密不写入原型的 Web 存储。

配置缺失时在当前任务中打开配置选择器，可以引用已有条目，也可以原地新建并持续绑定。完成最后一个真实缺口后直接生成，不再多一次“准备完成”确认。

## 版本、环境与任务事实

- 版本只由真实完成的生成任务产生，冷启动不虚构历史记录；每个版本固定完整提交与每个服务的不可变产物引用和 digest；
- Version 保存项目、完整提交、来源 Task/Attempt 和完整 ArtifactRef 集合，服务 ID 必须与当前 Project 精确相等；成功 Attempt 保存不可变环境输出快照，`VersionValidation` 只保存测试环境、配置/服务/部署修订、服务器、地址、健康和业务结论，并逐项匹配来源输出和冻结输入，不重复产物与来源；
- 每个 `ArtifactRef` 同时固定服务、repository、digest 与产生它的 Registry Connection revision；更换默认仓库不会改写历史版本来源；
- 在线版本、最近尝试、业务结论和访问地址分别表达；
- 正式环境可选择任一测试通过版本，历史版本使用“恢复”而不是再次“发布最新”；
- 一个正式环境同一时刻只有一个进行中或待收尾任务；
- Task 保存目标；production Task 只固定 Version ID、环境、意图和期望结果，真正执行时 Attempt 才从 Version 复制完整 ArtifactRef 并冻结其他输入。代码账号可以在 Task 之前保存；项目仓库只读核对通过后、任何远程副作用之前才创建 Task。等待用户输入时可以有 Task 但还没有 Attempt；失败重试只增加 Attempt，重载后仍从原停点继续。
- 测试 Task 创建时立即冻结完整 40 位源提交；Attempt 继承该提交，不因之后的当前 HEAD 变化而漂移。
- 授权、启动、缺配置、服务器断连和路由冲突的恢复都复用原 Task；每次真实继续只追加一条完整输入 Attempt，旧正式版本与旧地址在成功切换前持续在线。

## 自动化规则与配置应用

- `AutomationRule` 分别保存 desired、observed、Provider 引用和 sync 状态；暂停、恢复、同步失败与修复均维护同一规则，不创建部署 Task，也不改变现有版本或正式环境。
- Config Profile 编辑默认只保存配置库 revision，且 Profile 不保存反向 bindings；引用从各项目 EnvironmentBinding 派生。只有目标环境存在真实在线 Version、服务器证据和地址均仍有效、配置完整且项目/环境归属一致时，显式“应用”才创建配置 Task；production 还要先确认影响。Attempt 冻结在线 Version、服务器修订、配置 ID/revision、旧 applied revision 和地址，完成时任一事实漂移都拒绝推进 applied revision。
- 解除引用精确到项目 + 环境；同一 Profile 的其他项目或环境引用保持不变。目标环境只有一个推荐候选；Profile revision 与 applied revision 相同则不创建无变化任务。

## 第一次正式发布

正式信息不阻塞第一次测试。用户第一次选择测试通过版本后，依次确认：

1. 正式运行位置；
2. 正式配置绑定；
3. 正式访问地址；
4. 目标版本和生产影响。

关闭面板后已完成题目和非敏感输入保留；再次进入从原题继续。正式任务使用测试通过的同一版本，不重新生成。服务启动成功后，如果 DNS 或 HTTPS 尚未完成，页面进入地址收尾阶段，并保留服务已部署事实，不触发第二次部署。当前状态模型仍把该阶段挂在部署 Task 上，尚未完整拆成独立 AddressOperation；这是已知后续项，不能把页面表现冒充为模型已经解耦。

## 连接、已有部署与本机工具

- “连接与资源”按账号级维护代码平台、版本存储和服务器能力，显示最近验证、Connection revision 与影响项目数；每个项目另有精确 SourceBinding。添加、切换和恢复项目只校验已有证据，不能自动生成验证；首次生成、已有部署接入或自动化写操作才使用已有账号只读核对仓库。读取失败不创建 Task/Attempt、不修改规则，也不立即索要 Token；更换授权不要求移除项目。Registry endpoint 必须是无用户名、密码、查询参数和 fragment 的规范地址，秘密只进入专用 Token 字段；每次修订写入不可变读写证据，历史 ArtifactRef 始终按冻结 connection revision 解析。
- “继续管理已有部署”依次核对代码连接、版本保存连接和服务器身份/能力，随后从匹配服务器生成不可变 `DiscoverySnapshot`；最终确认只映射快照已经证明的版本、产物、配置引用、地址和健康事实。不创建部署 Task，不重新部署，也不把本机是否有凭据伪装成线上状态。“重新设置”只重置本机绑定，不停止或删除线上服务。
- “在本机运行”保持独立，按项目服务、运行依赖、必要配置排列，不创建远程任务。
- 当前受支持的“空服务器”实际要求 Docker Engine/Compose 已可用且 80/443 空闲；裸机自动安装没有闭环。边缘访问能力是一台服务器一个共享 Docker Caddy 容器、项目 + 环境独立 `sites` 片段，变更采用 backup → `caddy validate` → reload，失败 rollback。已有兼容 Docker Caddy 可复用；systemd/其他代理、端口冲突、旧服务或归属不明全部阻断。只有另一已知 ABCDeploy 项目占用时提供明确“转移地址”。正式版全新服务器也应执行相同初始化检查，但这是待实现门禁，不是原型已经证明的运行时能力。

## Round 10 真实浏览器走查
